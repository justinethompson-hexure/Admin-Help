Step 5. Complete Reviewer Documents [Reviewer Instructions]

The organization determines whether to include reviewer documents and whether reviewer documents are optional or required.

If the organization requires reviewer documents, FireLight® disables the Approve button until after the reviewer completes the required reviewer documents. When reviewer documents are optional, the Approve button remains enabled whether or not the reviewer completes reviewer documents.

  1. If the order package includes reviewer documents, the reviewer clicks Reviewer Documents and fills in the fields on the applicable reviewer documents. FireLight pre-populates some fields from inputs on the application. Those fields are display-only fields and have a white background.

    NOTE: An organization can add Review Queue settings and fields for Reviewer and Supervisor comments to the reviewer documents in FireLight Admin. When the reviewer and/or supervisor add comments to those fields in the reviewer documents, FireLight adds those comments to the review Queue History and Audit Report.

  2. The reviewer clicks buttons on the Reviewer Documents page toolbar to do the following:

    • Back to Review Queue – returns to the Review Queue on the Review Queue page.

    • History – opens the Reviewer Documents History dialog. This is where the reviewer reviews the history of only the reviewer documents.

    • Save – retains and saves the updates the reviewer made to the reviewer document.

    • Complete – sets the reviewer document to the Complete status. The reviewer clicks Lock and Complete on the Finish Reviewer Documents dialog to lock the document to prevent further data entry and return to the application order in the Queue.

NOTE: When using Sequential Review Queues, FireLight retains reviewer documents and their comments in the original queue and across sequential review queues. Even though these comments will still be editable across each queue, the next reviewer is able to view them within the queue that they are assigned. Additionally, if an activity is rejected, the reviewer documents are retained, especially if the user would want to recall that rejection. This impacts all users who use reviewer documents within all available review queues (Post-Submit Queues, Pre-submit Queues, and Automated Queues).
