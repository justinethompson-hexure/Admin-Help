Activities

On the navigation bar, click Product Config/Activities to view the Activities page.

NOTE: Configurable Pre-Sale activity requires licensing. Contact your Strategic Account Manager for details. See also the FireLight Configurable Pre-Sale GuideFireLight Configurable Pre-Sale Guide.
Spotlight Videos: See the FireLight Configurable Pre-Sale and FireLight Scoring and Pre-Sale Spotlight Feature Videos for instructions.

Use the Activities page to create and configure a pre-sale activity using the activity layer of FireLight. An Activity represents a type of Activity a user can create (e.g., Account Opening, Pre-Sale, Needs Determination, etc.). The Activity can be configured independently and some of its workflow can be customized by the Admin on the Activities page instead of requiring code changes by FireLight development.

An organization can create new activities that have different names and icons on the Home page, different pre-fill codes, potentially different next steps, different locking rules, different deleting rules, and different reporting.

Configurable Activity Guidelines

IMPORTANT! In parent/child activities, you may need to write a rule to disable the use of custom action buttons that could change data in the child activity when coming from a pre-sale activity.

  • The top section of the Activities page contains the existing activities that are supported in FireLight (Account Opening, Application, E-Delivery, Fee Inquiry, Illustration, Needs Determination, Post Issue, Print Forms, Quote, SubPay, etc.). Selecting any of these activities displays the activity details below. Any new activity configured on the Activities page is included in the top section.
  • An organization can create more than one Pre-Sale activity and more than one Pre-Sale activity per Activity Type.
  • Activities are linked to Products and Packages via a Sales Agreement.
  • Pre-Sale activities can be sent automatically to the post-submission review queue.

  • New Pre-Sale activity deployments:

    • Carriers and Distributors: (1) Carriers create a product and package with the activity type that are deployed to the distributor via the sales agreement. Only the product, packages with the activity type, and sales agreements deploy. (2) The Distributor links the product to an Activity defined in their distribution environment. (3) The Distributor defines the Activity settings on the Activities page and deploys the Activity to their own environment.
    • Carriers Only: Carriers can manage the entire process for their own organizations. Carriers can deploy the Activity and its associated product, package, and sales agreement to Staging and Production environments, but not to the Distributor. Activity deployments must stay within the same organization.

  • When accessing FireLight through SSO: If the product is active in Groups and the new activity has the TypeCode defined for pre-fill, then when the SSO has that transaction type defined, FireLight launches the user into the activity in the SSO.

  • E-Signature and PreSubmit review are triggered based on the settings, if there are signature tags, and the script for review.

    NOTE: When the pre-signature review process is before the signature process, the Request Client to Fill & Sign option is not available on the Other Actions menu and we do not allow the Console to check out an app. See also Request Client to Fill & Sign [User Instructions].

  • Upon continuing to next steps, pre-fill data and lock down into the next activity are based on the Activities page settings.

  • The next steps are driven by first the activity type of the current activity and its activity type’s admin configuration, then the user permissions in the role code-based user group, then finally by the data items in the current activity. When the Known DataItems FLI_NEXT_ACTIVITY_JURISDICTION, FLI_NEXT_ACTIVITY_PRODUCT_CUSIP and FLI_NEXT_ACTIVITY_ACORDTYPE are set and valid, then it is possible for no options to be presented, but have the user redirected immediately to the new application. If only FLI_NEXT_ACTIVITY_JURISDICTION and FLI_NEXT_ACTIVITY_PRODUCT_CUSIP are filled and valid and there are multiple Activity Types for that product, then the user is asked to select which Activity type to proceed to on that product. If only FLI_NEXT_ACTIVITY_ACORDTYPE is filled and valid, then the user is redirected to the new activity page and asked to select a product. If none of those data items are filled or valid, then the user is presented an intersection of the Activity Types the user has access to and the next Activity Types selected for the current activity type.

  • Linked activities follow the activity settings for locking the initial Pre-Sale activity, which controls deletes of the initial activity if there are linked activities.
  • Upon continuing, the PDF from the initial activity is included in the documents of the next activity. The Pre-Sale activity name is the document type name. This can be used for the templates.
  • Audits from the Pre-Sale activity are included in the next activity.
  • The audits on both the Pre-Sale and the next activity include an audit record of the linkage.
  • In FireLight on the All Activities page, the Activity type includes the new activity.
  • Reports are run based on the billing code for the new activity.
  • Reports include the new activity.

NOTE: Existing Needs Determination and Account Opening activities were converted to a Pre-Sale activity type with customized names of Needs Determination and Account Opening and the activities remain linked.

LastUpdate (Purge Policy)

LastUpdate is updated when the following occurs:

  • The app data is updated. This includes when:

    • The user clicks Save or answers Yes to the save prompt when leaving an activity.

    • The user is prompted for recoverable changes when opening an activity and chooses to Recover.

    • At various times during data entry (print, custom action, CRM, etc.).

  • The activity name is changed.

  • The activity status or display status changes (for instance, from data entry to signatures, signatures to review, app submission, etc.).

  • A policy number is assigned to the activity.

  • The activity is checked out to the console or checked in from the console.

  • Pending app data is updated.

When the activity is submitted, LastUpdate is updated to the current time and will reset the purge countdown, so by definition it will not be near the purge time at submission. For example, if the purge policy is 60 days and it was not transmitted to DTCC within 60 days of submission, it would get purged, but that is not likely, because an activity in pending transmission status is seen and dealt with long before then.

The provider also has access to the activity information (including LastUpdate) via the app service GetApplicationInfo.

Clients can request to receive Detailed Activity Reports for pending purge and recently purged activities.

Add a New Activity

Add a New Activity

Do the following to create a new Activity.

  1. On the navigation bar, click Product Config/Activities to view the Activities page.

    Existing activities appear in a grid at the top of the page.

    To remove an existing activity, click the red X on the right of a grid row.

  2. On the upper left, click the Add New Activity link.

  3. Below the grid on the Activity Detail tab, complete the following fields.

  4. Click the Next Activities tab. Use this tab to define what Activities can be created next from the current Pre-Sale Activity. For example, a Pre-Sale Activity might include another Pre-Sale Activity or the Illustration Activity and/or the Application Activity as next activities.

    In the All Activities list, select an Activity and click the right arrow button to move it to the Next Activities list.

  5. Click the Save Activity button at the end of the Activities page.
  6. On the navigation bar, click Product Config > Products Create a product for your new activity.
  7. On the navigation bar, click Product Config > Packages. Create a package for the new activity product.
  8. On the navigation bar, click Product Config > Sales Agreements. Create a Sales Agreement for the new Package.

  9. On the navigation bar, click Organization Config > Groups. Select each user group that will use the new activity and do the following for each group:

    • On the Activity Access tab, select the checkbox for the new Pre-Sale activity you created.
    • On the Product Access tab, select the checkbox for the product you created for the new Pre-Sale activity.

  10. On the navigation bar, click Utilities > Deployments. In Synchronization Scope, select All Configuration. Enter a Reason for the deployment. In the grid, select the Deploy checkbox for the product to deploy. Choose whether to Deploy Now or Schedule Deployment. See "Guidelines" above for carrier/distributor and carrier only deployment information. See Deploy Data for deployment instructions.

  11. The FireLight Home page displays the new pre-sale activity and icon button in the Start New section. Click the button to start the pre-sale activity process. Upon completion of the pre-sale activity, click Continue to start the next step defined in the activity settings. That could be signing if there are active signature tags, pre-submission review if it was selected in the Activity settings and set in script, or start the submit process to send the output file to the endpoints and initiate the next activity workflow.

View Activity History

View Activity History

  1. On the navigation bar, click Product Config > Activities to go to the Activities page.

    Existing activities appear in a grid at the top of the page.

  2. Click an Activity in the grid.

  3. Click the Activity History tab.

    The tab displays the Action, Old Value, New Value, Action Date, and Login of the user that performed the action.

  4. Click the Next and Prev buttons to navigate among the Activity History pages.
  5. In Utilities, Reports page, Admin Reports, in Admin History Reports, run the Activity History Audit Report to view an audit history report for selected activities during a selected time frame.
