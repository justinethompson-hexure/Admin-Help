Work Group Processing

  • CMR xxxxxx Abi In Development (Carriers and Distributors)

    Currently, activities initiated in an organization's FireLight instance are only visible to individuals provisioned into that instance of FireLight. Implementation and creation of inter organization work groups will allow users to have shared visibility into an activity, receive notifications for the activity life cycle and allow members to have the same visibility. The current activity/application ownership model in FireLight is very rigid and accessibility options are cumbersome to manage. The ideal solution will allow multiple individuals to access an activity without being required to take specific action such as Transfer, Share, etc. Status and notification functionality is enhanced to add a list of individuals to receive notifications/have visibility of a single activity from start to finish.

    Impacts:

    • New Organization setting to allow work group setup

    • Work Groups section in User Preferences

    • New Work Group management capabilities

    • Ability for defined work groups to access activities with edit/read-only rights

    • Ability for users within a work group to receive notifications for the life cycle of an activity

Work Orders

Implement Client Protection on Forms

  • CMR 258253 Keerti, Abi In Design (Carriers and Distributors)

    This enhancement will make forms available to only clients when they login into FireLight using 3rd party links. Currently, client signers can log into FireLight using 3rd party links generated from Client Fill and Client Fill & Sign requests. Using this enhancement, FireLight admin is able to display or hide forms when the client signers have logged into FireLight . To implement this protection, "Client Protection" option under the "Form Properties" link on form designer screen in the admin tool is used.

    Impact:

    • FL App: 3rd Party login URL for Client Fill and Client Fill & Sign

    • FL Admin: Designer > Form Designer > Form Properties > Protection > Client (selection in dropdown)

    • FL Admin: New data item "FLI_IS_CLIENT_FILL_LOGIN" that can be used to write rules to identify if client has logged into FireLight using 3rd Party login URL.

Implement Timeout for Passcodes

  • CMR 269829 Keerti, Abi In Design (Carriers and Distributors)

    This enhancement will allow applying a timeout to the validity of passcode required for 3rd party login to FL by clients, agents and reviewers. Currently, there is no time limitation for the passcode to be valid. To implement a standard practice for passcode (OTP) expiry, this enhancement will allow clients to enforce a timeout using an org setting.

    Impact:

    • A new text box "Passcode Timeout" is added to page FL Admin > Organization Configuration > Organization Settings > Signature (tab).

    • A new "Resend Passcode" button is added to 3rd party login page to resend passcode in case of expiration.

Fixed Defects

 

Admin

  • DR 271019 Error when generating tag list in Download Tag List utility. Problem: When the button to download the Tags was clicked, a null error could happen, but would definitely not do anything else. The download was never properly fired off. The window.open that was supposed to start the download process, would be blocked by popup blocking in the web browser. Resolution: Fixed null value issue, window.open issue (was being blocked), added a spinner to denote that the user must wait for the process to finish.
    Areas Impacted: Utilities > Download Tag List screen in Admin

Integration

  • DR 262677 Sporadic Scheduler BackOfficeService Failures due to Collection Modification on Async Waiting. Problem: This issue has resulted in some back office processing to complete but remain as processing in the workflow. Resolution: In Azure the Scheduler process continues to run, which does not require a wait after kicking off the BackOfficeManager.ProcessWorkflows(). Because of this I removed the logic to track the ActiveTasks. This logic looked flawed because there were no locks on this list when it could be modified by different threads.
    Areas Impacted: Scheduler
