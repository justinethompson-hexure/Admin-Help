Add Multiple Copies of a Form

NOTE: Having a form not duplicated results in no change to the value, so this data is compatible with legacy data and no conversion needs to take place.

If you add multiple copies of a form to a package (for example, additional beneficiary form), the original form becomes a template for the duplicate copies.

Use the Form Designer page to add a template moniker [FORM_INDEX], [FORM_INDEX_PLUS_1], and/or [FORM_COUNT] identifier to mark the DataItemIDs to change on the duplicate copies. The identifier is replaced by the index number of the copy starting at 1. The form names and form numbers have the index appended at the end.

[FORM_INDEX] Sample Rule

FIELD_NAME[FORM_INDEX] <= data item for the nth form copy (the rules engine automatically generates indexed rules for each form copy)

Advantages:

  • Eliminates duplication of rules – for a form with 10 copies, one rule suffices rather than 10 nearly identical rules
  • Prevents problems loading rules – if a form has 3 copies in a package, but a rule contains code referencing copies 1 thru 4, none of the rule’s script code are loaded or executed
  • Facilitates editing of rules – Within the Rules Windows in the Form or Wizard Designer, "List Only Rules for Selected Control" shows rules for the selected field when using the FIELD_NAME[FORM_INDEX] syntax, but not when using the FIELD_NAME1 format.

Differences between the Combined Wizard+Forms UI and Legacy Forms UI:

Legacy forms operate on the current page regarding radio buttons, and updating a radio button affects only fields with the same Group on the current page.

Wizard (React UI) intrinsically operates on the entire wizard and forms collection, and updating a radio button affects all fields with the same Group for the application as a whole.

Technically, the Group field should always be indexed on a multiple-instance form, like the field name. Indexing the Group field works correctly for legacy forms and the combined wizard+forms UI and is the recommended approach.

Sample Rule:

[FORM_INDEX] Sample Rule

 <block> <setvisibility dataitemid="Owner_Signature[FORM_INDEX]"> <or> <compare op="=="> <const value="2" type="String" /> <diget dataitemid="FLI_ALLOWESIGN[FORM_INDEX]" /> </compare> <compare op="=="> <const value="3" type="String" /> <diget dataitemid="FLI_ALLOWESIGN[FORM_INDEX]" /> </compare> </or> </setvisibility> </block>

[FORM_INDEX] Identifier Examples

Following are examples of how each identifier would change within the system for each copy of a Beneficiary form.

DataItemId

Insured[FORM_INDEX]_FirstName

1 = Insured1_FirstName

2 = Insured2_FirstName

Form Name

Beneficiary Form

1 = Beneficiary Form 1

2 = Beneficiary Form 2

Form Number

ITBFCO

1 = ITBFCO1

2 = ITBFCO2
