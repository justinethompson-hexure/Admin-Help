Button

IMPORTANT! In parent/child activities, you may need to write a rule to disable the use of custom action buttons that could change data in the child activity when coming from a pre-sale activity.

Use the Button control to add a custom action to a form. The Action Type field property determines the action the Button control initiates on the form. For example, when Action Type is set to Custom List, clicking the button opens a selection list.

FireLight® supports Custom Action Print and Upload buttons on template forms. FireLight validates whether the template is uploaded before the user data entry can reach 100%. The template appears in the Open list to provide instructions. The template form can include the Custom Action Upload and Custom Action Print in Button controls to provide clear instructions with easy links to complete the template requirement.

FireLight supports triggering a custom action call from any page within the same package, regardless of whether the button or fields to trigger that call are on that same page. The call can occur only when a custom action button is enabled. If the button is disabled, the call cannot occur. Additionally, a call can occur only on a single activity, meaning that the call cannot occur from a presale when the custom action button is in an application. The clickcustomaction rule can reside on any form or wizard in the package and trigger a custom action button.

A Button control and associated input controls often belong to a unique Data Group. The Button control initiates an action, such as opening a custom list, uploading a custom file, opening a CRM, or initiating a carrier-specific custom action. Associated input controls display the results of the action such as custom list column data.

NOTE: The value of the Button click count is incremented with each click. See "Add a Rule to Disable a Custom Action Button After Three Attempts" below for a sample rule to control the number of times the Button can be clicked.

    Custom Action

    Set the button to initiate a custom action on the form. Some Custom Actions are coded by the provider specifically for your organization. You must select a Custom Action Type and then enter one or more parameters in Action Parameter when applicable. NOTE: The clickcustomaction rule can reside on any form or wizard in the package and trigger a custom action button.

    Custom Action Type – select a custom action type in the list:

    • Regular Provider Action initiates a custom provider action that allows the Provider to send the binary PDF data as well as the document tag associated with the PDF to add a document or replace an existing uploaded PDF during the data entry process. You must enter a value in Custom Action Parameter for the action (Upload) and a document tag value (e.g., Sample Doc) that is provider specific for your organization. Audit history displays the initial addition and when changes occurred to the document. NOTE: To add this ability to your organization's Provider, contact your Strategic Account Manager.
    • Print initiates a print action. See Action Parameter for Print options.
    • GIACT initiates a web service call to GIACT Systems for banking account verification and authentication. You must enter GIACT In Action Parameter. See GIACT Services Integration in Features for setup instructions.
    • LexisNexis initiates a web service call to LexisNexis for client identity validation or for risk classifier information. For client identity validation, enter LexisNexis in Action Parameter. For risk classifier information, enter LexisNexisRisk in Action Parameter. See LexisNexis Client Identification Verification During E-Signature and LexisNexis Risk Classifier Integration in Features for setup instructions and feature information.
    • Multiple Action – set the button to initiate a sequence of actions. You must enter parameters in Action Parameter. See Create User, Transfer, Email Custom Action (Concierge Paper Workflow) for detailed information.
    • APPS initiates a web service call to the APPS Scheduled4Real paramed online ordering system. Enter the applicable Custom Action Parameter for the action taken by the APPS custom action button: APPSCheckAvailability, APPSMakeReservation, or SubmitParamedOrder. Ensure Data Group is set to Insured or Other Insured as applicable.
    • ExamOne initiates a web service call to the ExamOne ScheduleNow paramed online ordering system. Enter the applicable Action Parameter for the action taken by the ExamOne button: ExamOneGetMobileTimes, ExamOneConfirmMobileTimes, ExamOneGetInCenterLocations, ExamOneGetInCenterTimes, ExamOneConfirmInCenterTime, SubmitParamedOrder or SubmitHealthPiQtureOrder. Ensure Data Group is set to Insured or Other Insured as applicable.
    • Launch Morningstar initiates a call to Morningstar Intelligence Tool.
    • Data Entry Review initiates the data entry pre-signature review process. This custom action type requires an organization to have an API set up to perform the call and is designed to be used only in instances where the organization requires a data entry review prior to the client fill and sign process.

    Action Parameter – Add an action parameter for the selected Custom Action Type.

    • Regular Provider Action – When Custom Action Type is set to Regular Provider Action, enter the action and the document tag that is in the provider for your organization in Action Parameter. For example, enter Upload Sample Doc.

    • Print – When Custom Action Type is set to Print, leave Action Parameter blank to print all forms. Type Print in Action Parameter to print all forms with the same Attachment TC type as the current form (this is the Attachment TC setting on the form in the Form Library). To print only forms with a specific Attachment TC type, type Print|[Attachment TC type]. For example, to print forms in the Form Library with an Attachment TC type of 73 (in ACORD 1035 Exchange Forms, Attachment TC type = 73 in the Form Library), type Print|73. For an example, see the Cooperative Technologies 1035Yellow Pages topic. The custom action print button follows the same print restrictions set at the organization level as the print TC and the Display/Print PDF.

    • GIACT – When Custom Action Type is set to GIACT, type GIACT in Action Parameter.

    • LexisNexis – When Custom Action Type is set to LexisNexis, type LexisNexis in Action Parameter for the LexisNexis client identity verification.

    • LexisNexisRisk – When Custom Action Type is set to LexisNexis, type LexisNexisRisk in Action Parameter for the LexisNexis risk classifier information.

    • Multiple Action - When Custom Action Type is set to Multiple Action, select or enter the applicable action parameters. The format for entering parameters is "parameter 1," "parameter 2," etc. As the sample naming implies, the parameters identify the action and the entry order identifies the action sequence. It is not necessary to number parameters (e.g., CreateUser, TransferActivity, EmailUser). See the Create User, Transfer, Email Custom Action (Concierge Paper Workflow) topic in Features for detailed information.

    • APPSCheckAvailability – When Custom Action Type is set to APPS, enter APPSCheckAvailability in Action Parameter for the button that checks for paramed exam availability.

    • APPSMakeReservation – When Custom Action Type is set to APPS, enter APPSMakeReservation in Action Parameter for the button that makes the paramed exam reservation.

    • ExamOneGetMobileTimes – When Custom Action Type is set to ExamOne, ensure ExamOneGetMobileTimes is in the Action Parameter for the button that gets the times available for mobile paramed exams.

    • ExamOneConfirmMobileTimes – When Custom Action Type is set to ExamOne, enter ExamOneConfirmMobileTimes in the Action Parameter for the button that confirms the time available for the ExamOne mobile paramed exam.

    • ExamOneGetInCenterLocations – When the Custom Action Type is set to Exam One, enter ExamOneGetInCenterLocations in the Action Parameter for the available ExamOne In Center locations.

    • ExamOneGetInCenterTimes – When the Custom Action Type is set to Exam One, enter ExamOneGetInCenterTimes in the Action Parameter for the available ExamOne In Center times.

    • ExamOneConfirmInCenterTime – When the Custom Action Type is set to Exam One, enter ExamOneConfirmInCenterTimes in the Action Parameter for confirmation of the ExamOne In Center paramed exam time.

    • SubmitParamedOrder – When the Custom Action Type is set to APPS or ExamOne, enter SubmitParamedOrder in the Custom Action Parameter for the custom action button that submits the paramed exam order.

    • SubmitHealthPiQtureOrder – When the Custom Action Type is set to ExamOne, enter SubmithealthPiQtureOrder in the Custom Action Parameter for the custom action button that submits the HealthPiQture order.

    • For Morningstar custom action button, set Action Parameter to Morningstar. See Morningstar Integration in Features for setup instructions.

    Custom File Upload

    Set the button to launch a Documents dialog in . In Documents, if the document type is not found in the Document Type list, it is automatically enabled regardless of the Admin setting. Also use the Custom File Upload setting to upload image files to the Documents dialog if image files are allowed. See the Documents [User Instructions] dialog in Other Actions for user instructions.

    Enter the Type of Doc to Upload and determine whether to select the following options:

    • Allow Non-PDF File – If non-pdf image files and .xps document files are allowed, select this checkbox. The following image file types are allowed: .jpg, .tiff, .png, .gif, and .bmp. After upload, the image files are converted into .pdf files and stored the same as other files.

    • Disable Change Doc Type – Select the checkbox to not allow the agent to change the Document Type. Clear the checkbox to allow the agent to change the Document Type.

    Custom List

    Set the button to access a custom list that was uploaded for your organization. You must select a custom list number in Data Group and a custom list name in Custom List.

    • Data Group – Select the Custom List {#} number (Custom List 1, Custom List 2, etc.).

    • Custom List – Select a custom list name for the button to open. The list options are the names of the custom lists that exist on the Custom Lists page in Organization Configuration.

    You can add up to nine Control fields for displaying custom list column data. See Custom Lists in Organization Configuration for guidelines and instructions to add, update and delete custom lists in .

    NOTE: The associated fields (e.g., text boxes) that receive the data from the custom list columns, must select a column name from the custom list in the Data Group Property in the field properties and use the same Custom List number in Data Group as in the button control.

The following sample rule is for a text box field (with the same data item ID in the Name field property as the Button) that displays the number of times that the button has been clicked. The rule requires the button to be clicked at least once.

CRM Button Guidelines

The following information is for using data groups in CRM integrations, such as Cooperative Technologies 1035 Yellow Pages, Redtail, Salesforce, and Whitepages.com.

This concept works well for the CRM data integrations because all you have to do is set the Data Group and the Data Group Property to the field being mapped. Because the field names do not matter, you can set this up one time for Whitepages.com and Redtail and do not have to do anything different other than set the configurations and set the Data Group in the custom action buttons.

  1. In the Button control, set the Action Type to CRM, set the CRM Type to the CRM call associated with the custom action, and set the Data Group for that specific set of data.

    For example, in the Get Client Info button field properties, set the Action Type to CRM, set the CRM Type to Redtail, and set the Data Group to Owner.

  2. In the form fields that receive data from the CRM, select the Data Group and Data Group Property.

    For example, in the Owner_FirstName field, set the Data Group to Owner, which tells the system to pull back the owner detail from the call to the CRM, and set the Data Group Property to FirstName, which tells the system to put the first name of the owner in this field. Repeat this process for each field that receives data from the CRM call.
