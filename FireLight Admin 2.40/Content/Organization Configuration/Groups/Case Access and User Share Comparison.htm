Case Access and User Share Comparison

Four different Case Access and User Share options in FireLight® allow a user or users to see another user’s activities. Typically only one option is selected as they may conflict with each other.

Case Access and User Share Options

  1. Case Access allows all users within a group or groups to view all activities within the specified group or groups.
  2. Case Access by Firm ID allows users within a firm to view activities only within the same Firm ID when Case Access By Firm ID is enabled for a group. Case Access by Firm ID can be hierarchical with different levels of access per group (e.g., agent, branch manager, region manager, top manager, etc.).
  3. User sharing allows a user to share all activities with another user within an organization when Enable All Activities User Share is enabled for a group or groups.

  4. User sharing allows a user to share a single activity with another user within an organization when Allow Activity Share is enabled for a group or groups.

Case Access

Case Access is enabled at the group level. You can elect which group or groups to share activities with. This includes all activities within the specified group or groups. You are able to selectively define this for each group. You can also limit case access within a group by Firm ID.

Case Access by Firm ID is used as a second level filter along with the Case Access by Group Role Code. The carrier or distributor sets up Firm IDs through the tx1228 when users SSO in to FireLight or through FireLight Access and the user accounts. Firm ID can be as simple as the organization ID or a number such as "1" or it can be a hierarchical Firm ID. FireLight filters Firm ID by comparing the Firm ID of the application using a "starts with" comparison. The Firm ID is pipe (|) delimited between hierarchical levels. Each level can be defined with any alphanumeric designations (for example, by Firm|Region|Branch|BranchAgent as in ABC|Western|ABC001|Agent0001, ABC|1|001|0001, etc.).

User Share

User share, all activities and single activity, allows you to share with a specific user. It is enabled at the group level but applies to all users within the organization.

There are two types of user share:

  1. Global share all activities (Enable All Activities User Share) – allows you to share all your activities with another user.
  2. Single share one activity (Allow Activity Share) – allows you to share one activity with another user.

The available users to share with are based on the ‘Organization’ and the ‘Groups’ that have the user share option enabled.

When setting up the user share with another user, the information that appears during a search for a user is:

  1. User Name, which is the ID passed in the SSO or entered in a direct login.
  2. User Full Name, which is entered in the Preferences. It may also be passed in the SSO.
  3. External Map Code, often mapped to the user email address.
  4. You can search on any of the above 3 items.

This again is based on any user from the organization and groups that have the user share option enabled and that matches the entered string for the search.

When a user share request is sent to a user, he or she must accept the share request. When that user logs in to FireLight, only the user who created the user share request appears in the User List dialog on the All Activities page. If multiple shares are created, more than one user name will appear in the User List dialog. A user clicks the search button next to the User List to open the User List dialog and uses the search text box and button to locate a new user to add them to the User List dialog.

RepCode

RepCode provides activity visibility and security for organizations and their representatives. Each rep can have their own RepCode and a RepCodeVisibility list that contains other reps’ RepCodes to allow them to access those reps’ activities.

The RepCode is provided and controlled by the organization through the SSO data provided during an SSO login from their rep. The organization handles all the security and provides us with a primary RepCode (for the user via “IsPrimary” attribute) and any other RepCodes.

  1. When the user is logged in they will have SessionState variables that are populated.

  2. The RepCodeCreation session variable is populated with the value in a <RepCode IsPrimary=true></RepCode> tag. This will be used to assign the OwnerRepCode for every activity created in that session by the user.

  3. The RepCodeVisibility SessionState variable is assigned from the list of RepCodes provided in the SSO login data.

NOTE: The Disable Editing Other Users' Activities checkbox on the Groups, App tab, needs to be cleared (unchecked) for the following options to work.
