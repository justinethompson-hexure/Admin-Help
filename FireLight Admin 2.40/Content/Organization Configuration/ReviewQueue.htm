Review Queue

Spotlight Video: See the Review Queue Script Workflow Spotlight Feature Video for more information.

On the navigation bar, click Organization Config and click Review Queue to go to the Review Queue page.

This is where you create a post-submission queue stop workflow prior to the orders being transmitted to the carrier’s back office. The review queue can be a reviewer post-submission review queue or an automated post-submission review queue through the provider that does not require reviewer interference.

See the FireLight Review Queue GuideFireLight Review Queue Guide for pre-submission and post-submission reviewer instructions. In the FireLight Platform User GuideFireLight Platform User Guide, New Activity section, see the "Pre-Submission Review Step"Pre-Submission Review Step [User Instructions] topic for agent pre-submission review instructions.

PreSubmit Review Queue Guidelines

NOTE: If the organization uses a Pre-Signature Review Queue, Client Fill and Sign is disabled. Client fill is still available.

  • If your organization uses the pre-submission review process, FireLight® automatically creates a "read only" PreSubmit review queue. The PreSubmit review queue cannot be edited or change its name.

  • To enable the PreSubmit review queue, select the Allow Pre-Signature Review or Allow Post-Signature Review checkbox on the Product Configuration, Activities page.

  • Firms can auto send applications to the PreSubmit queue by setting a data item. In FireLight on the Preferences page in the Reviewers section, users need to add the pre-submission reviewers' names and email addresses if the firm is not using auto send.

  • To force or require a pre-submission review using rules, see FLI_REQUIREMANUALREVIEW on the FireLightBase tab in the Common Tags spreadsheet for information. On the same tab in the spreadsheet, also see the FLI_LOCK_REVIEWERS (prevent users from adding or clearing reviewers), FLI_REVIEWER_NAMES (semicolon-separated list of reviewer names) and FLI_REVIEWER_EMAILS (semicolon-separated list of reviewer email addresses) data items for information on how to conditionally pre-populate and disable reviewers.

IMPORTANT! The PreSubmit review queue does not use a client organization's review queue script.

Add a Post-Submission Review Queue

Set Up a Review Queue

  1. On the navigation bar, click Organization Configuration and then click Review Queue.

  2. Click the Add New Queue link on the upper left and add review queue information below the grid.

    Queue Name is the name of the review queue.

    Queue Number is a unique number for the review queue.

    Email Group is the email address or distribution list of the review queue group that receives a notification email when an application is submitted for review. Leave blank if you do not want to send an email notification to a review queue group. The format of the email address is validated. If incorrect, an error message appears in red at the bottom of the page. The Email page contains the template, Review Queue Email, which can be modified except for the QUEUE_NAME, QUEUE_NUMBER, and CASE_NAME tags. See "Edit Email Subject Lines, Locale, and TemplateText" in Email.

    Remarks field contains any additional explanatory information for the review queue.

    Additional Info

    Adds optional sub-header text for each activity in the Review Queue in . For example, type Policy Number: [Policy Number] in the Additional Info text box to have the policy number display below the activity name in the Review Queue for each activity.

    • Values must be entered in template format.
    • Values must be included in the List Profile, which is specific to each organization and their database.

    • Multiple items can be used as Additional Info values with the use of a semi-colon. For example, type Product Name: [Product Name]; Issue State: [Issue State] to display the Product Name and Issue State in the sub-header for each activity in the Review Queue in FireLight.

      NOTE: To add a dataitem to the Additional Info text box, the dataitem needs to be included in the List Profile, along with the description of the dataitem (ex: Owner_FirstName is Owner First Name), in the organization's database. To add a dataitem and description to your organization’s current List Profile, contact the Strategic Account Manager. After the database update, add the description of that dataitem (e.g., Owner Name: [Owner Full Name]) to the Additional Info text.

  3. Select applicable checkboxes.

    Select the following checkboxes for Review Queues with applications that have more than one part, such as an agent-submitted application for part one and a medical history questionnaire from a tele-interviewer for part two. Application packages with more than one part will have an OpenApp button available for viewing the application after the Reviewer clicks the Review button and locks the application.

    Disable Approval Actions hides the Approve, Reject, and More Info buttons when the reviewer clicks the Review button.
    Disable Show Application On Lock does not automatically open the application when the reviewer clicks the Review button.
    Enable Children Links shows the OpenApp button. The reviewer clicks the OpenApp button to open the application for review.

  4. Go to the Review Queue Deadline section to add a review deadline (the number of business days orders can remain in the queue).

    1. In the At Deadline list, select an option:

      • Rejected to have the system automatically reject orders when they reach the review Deadline Length.
      • Approved to have the system automatically approve orders when they reach the review Deadline Length.

    2. In Deadline Length, enter the number of business days to count for the deadline expiration day count for the queue. If the activity in the queue is not reviewed by the number of days entered here, the activity is automatically approved or rejected from the review queue within an hour of the time and day when the deadline expires. The default is 0, which means no deadline is set. See "Enable the Review Queue Expiration Deadline Date" on this page for more details.

      Select the checkbox for any of the following options for types of days you would like to count as days to be included in the deadline expiration day count.

      • Deadline Date - Include Weekends includes weekend days in the deadline expiration day count.

      • Deadline Date - Include FINRA Holidays includes FINRA holidays in the deadline expiration day count. Click the FINRA Holidays link to open a calendar of FINRA holidays for the current year. FINRA holidays do not include a half day of Market Day.
      • Deadline Date - Include Market Days includes market days in the deadline expiration day count.
      • Deadline Date - Continue from Previous Queue is for sequential queues, where the deadline date counter is carried over to the next queue or for a PreSubmit queue where the deadline date is carried over to a post-submission review queue.
  5. Click the Edit Review Queue Script link to edit the workflow script for back office processing to include this review queue. See “Edit Review Queue Script” on this page for instructions.

  6. Click the Save button to save the review queue entries.

    The new review queue is added to the grid with a Queue Type of Post-Submit.

    NOTE: To edit a review queue, select the review queue in the grid and then edit the review queue information below the grid. Edit the review queue script if applicable.

  7. Deploy the Review Queue Script using the Review Queue Configuration in Deployment.
  8. Add designated reviewers access to the review queues by a reviewer role code in Groups so they can review, approve or reject orders electronically using the application. See the Review Queue tab in Groups.
  9. To ensure Review Queue emails are sent to an agent: on the Review Queue tab in Groups, ensure the Disable Action Email Notifications checkbox is cleared (unchecked) for the reviewer role code. In Email on the and Post-Submit Review Queue Approve and Post-Submit Review Queue Decline email templates, ensure the |MESSAGE| or <MESSAGE> replacement marker is somewhere within the email body. In FireLight on the agent's My Preferences page, ensure the agent's Send Message Center Emails checkbox is selected.

Review Queue Workflow

  1. An application is submitted by an agent after all the prerequisite steps are finished (such as PreSubmit review, Signatures).
  2. According to the organization’s Review Queue script, the application is then inserted into one or more review queue(s).
  3. A reviewer logs in to FireLight and opens the Review Queue page. The reviewer's role code that is set on the Groups page determines the review queues that the reviewer can access.

  4. Reviewer opens the applicable Queue, locks the order for review, reviews the documents and history for the order, and clicks a button to approve the order, reject the order, or request more information from the agent.

    • When the reviewer approves an order, the order is automatically submitted to the back office for processing.
    • When the reviewer rejects an order, the agent needs to copy the order and provide the requested changes in a new order.
    • When a reviewer sends a request to the agent for more information, the order is in "Request Sent, Pending Reply" status until the agent replies to the review queue message with the requested information. Upon receiving the requested information, the reviewer can proceed with reviewing the order.

Automated Review Queue Workflow

An automated review queue operates in an automated state, which approves or rejects a post-submission review without manual intervention by a reviewer.

  1. An app is submitted by an agent after all the prerequisite steps are finished (such as PreSubmit review, Signatures).
  2. According to the organization’s Review Queue script, the app is then inserted into one or more Automated queue(s).
  3. The Automated queue logic gets started and immediately tells the organization’s Provider to run the logic it has in place for the Automated Review process, whatever that may be (defined by client organization in conjunction with their Strategic Account Manager and the Integration team).
  4. The Provider may need to wait some time for the organization’s process to finish, which can take up to hours or days. During this time, the base system continues to trigger the Provider to check whether the process is finished. It will trigger the Provider every amount of minutes as defined in the queue’s Automation Retry Interval setting.
  5. Once the Provider understands that the process is finished (could take a while, or could be immediate), it will then tell the base system to either Approve or Reject the review item.
  6. The review script is rerun at this point, just like with regular queues, to determine whether the app should be inserted into any other queues, whether automated or not. If automated, the process starts over again, just for a different queue. Additionally, if the app was inserted into multiple Automated queues, the next queue’s item (related to the same app) will run through the same process. Only one Automated queue item can run at a time for a given app.
  7. The Provider will have access to the Queue names and numbers to help it understand contextual differences for review items: which queues they belong to. This allows for diversification of business requirements.
  8. Once an application is finished with all possible queues, it will move on to the final Submission logic.

Assign Reviewers to Specific Activities

FireLight Admin

  1. Click the Edit Review Queue Script link to open the Review Queue Script Window.

    Locate an activity, product, or type in the script. Add a new "ReviewerID" attribute to the review queue script to automatically assign the reviewer who will review the activity, product, or type.

    NOTE: In an automated review queue, the assigned person will always show as "Backoffice user" instead of a specific assigned person, even if a reviewer has been assigned within the script.

The script will use the logic where if the ReviewerID is present, FireLight will assign the reviewer regardless of this button being active on the UI.

 

FireLight App

  1. FireLight sends an email to the specified reviewer once the case is assigned to notify them it is ready for review. Another email is sent when the review has been completed. The emails are sent to the reviewer's User Full Name and User Email Address that are entered in the General section of the Preferences page for the reviewer in FireLight. An email address has to be entered in the Preferences Section for the reviewer to receive an email. Also, FireLight sends an email to any email address that is listed within the Email group text box within the Review Queue tab in the Admin tool. If an email address has been added to this text box, the same email that is being sent to reviewers is sent to this email address regardless whether an email is being sent to a reviewer email.

  2. The reviewer logs in to FireLight and clicks the Review Queue button on the Home page.

  3. The reviewer opens the order with a default order status of "My Locked orders."

    NOTE: The reviewer can access any other status from the Status dropdown at any time.

  4. If enabled in FireLight Admin, the reviewer can manually assign the order to another reviewer. See Step 6. Move, Assign, or Unlock an Order [Reviewer Instructions] in the Review Queue Guide.

Audit Report

The audit report will capture the following:

  • Assignment of a specific reviewer if manually assigned or "Back Office Assigned" if assigned automatically via the Review Queue script, Case Name, and Review Queue Name.

  • Status changes from "Pending Review" to "In Review" to "Locked for Review."

  • Email notifications including the Application Name, Review Queue Name, and email address the email was sent to.

Review Queue Report

  • Includes the Reviewer Name column to capture the assigned reviewer and unassigned reviewer and a Reviewer ID column to capture the reviewer's map code.

  • If a case has not been assigned to a reviewer, the Reviewer Name and Reviewer ID columns display "Unassigned." See Detailed Activity Reports for more information.

Enable the Review Queue Expiration Deadline Date

Enable the Review Queue Expiration Deadline Date

FireLight can track the number of days that an activity has not been reviewed in the Review Queue on a per-queue basis by setting a deadline expiration day count for each applicable queue. For example, it can track the timeframe for the FINRA 2330 rule that requires Variable Annuities to be completed and processed before a 7-day timeframe, which expires within an hour of the time of day on the 7th day set in the Deadline Length field.

The system checks the Review Queue for expiration deadlines every hour. Deadline dates expire within an hour of their designated time and day, which is the time the application was added to the queue plus the Deadline Length number of days.

Do the following to have FireLight track a deadline expiration in the Review Queue, set the types of days to include in the deadline day count, and notify the agent via email when the expiration deadline date is reached.

In FireLight Admin:

  1. In Organization Configuration/Review Queue page in Deadline Length, enter the number of business days to count for the deadline expiration day count. If the activity in the queue is not reviewed by the number of days entered here, the activity is automatically rejected from the review queue within an hour of the time and day when the deadline expires. The default is 0, which means no deadline is set.

  2. Select the checkbox for any of the following options for types of days you would like to count as business days to be included in the deadline expiration day count. The default setting is "cleared" for each of the checkboxes.

    • Deadline Date - Include Weekends includes weekend days in the deadline expiration day count.

    • Deadline Date - Include FINRA Holidays includes FINRA holidays in the deadline expiration day count. Click the FINRA Holidays link to open a calendar of FINRA holidays for the current year. FINRA holidays do not include a half day of Market Day.
    • Deadline Date - Include Market Days includes market days in the deadline expiration day count.
    • Deadline Date - Continue From Previous Queue If the Enable Manual Queue Move option is enabled in Organization Configuration/Groups on the Review Queue tab or your organization has a sequential review queue set up, select this checkbox to honor the Deadline Length reflected for the current queue in the next queue when the reviewer moves an order from the current review queue to another review queue. If the checkbox is cleared, this means the continue counter will start over at 0 in the new queue to count the days until the deadline date.
  3. If the Enable Manual Queue Move option is enabled in Organization Configuration/Groups on the Review Queue tab or you have a sequential review queue set up, you can select the Deadline Date - Continue From Previous Queue checkbox to honor the Deadline Length reflected for the current queue in the next queue when the reviewer moves an order from the current review queue to another review queue. If the checkbox is cleared, this means the continue counter will start over at 0 in the new queue to count the days until the deadline date.
  4. Click Save at the bottom of the Review Queue page.

  5. In Organization Configuration/Email page:

    1. In Email Category, select Review Queue.
    2. In Email Template Type, select Deadline Expiration email template.
    3. Make any applicable changes to the template text for your organization.
    4. Click Save Email Template at the bottom of the page.

In FireLight:

The Review Queue order in the FireLight Review Queue displays an Expires date when a deadline is set for a review queue. This text turns red when two or fewer business days remain for the review deadline.
