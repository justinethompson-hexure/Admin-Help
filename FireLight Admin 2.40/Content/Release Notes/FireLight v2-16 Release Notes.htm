Clear Provider when Copy with Changes Is Used

  • CMR 202565 Formerly, when a distributor used Copy with Changes they were allowed to copy the activity to another carrier’s product. When this happened, AppSub codes from the original case could remain in the new case’s AppSub file. Now when Copy with Changes occurs, the base system clears the case of any data related to the prior carrier to prevent data from being retained and potentially being transmitted in the AppSub file. When Copy with Changes occurs, the base system clears the case of any base, provider, rule and unknown scoped data related to the prior carrier, but retains the user-entered and 103 pre-filled data. The clear provider will apply to both Copy with Changes across carriers and Copy with Changes in the same carrier. Once the user data and 103 pre-filled data are applied, the new provider will run. This prevents product data from being retained and potentially being transmitted in the AppSub file. This update will remove any potential Personally Identifiable Information (PII) that could be passed to another carrier.

    Clear Provider Impacts:

    • Copy with Changes
    • AppSub files when Copy with Changes is used

Common Tag Conversion Tool Enhancements

  • CMR 213031 This enhancement includes two improvements to the Common Tag Conversion tools that were introduced in 2.14. The two improvements were identified during demos and training. Prior to this release, the spreadsheet only included the common tags in the downloaded report and it did not include the Export Values. Secondly, we did not have a way to only identify the data items that are not common tags. With 2.16, these two issues have been addressed. We added logic to include all export values from the forms and wizards as well as the common tags. This will help the user and eliminate the need to go back to the forms to determine the export values and enter the values manually into the spreadsheet. Additionally, we added a checkbox on the Download tag list page to determine if Non-Matching Common tags are in the download tag list tool. This checkbox is used to include only tag names that do not align with our Common tags in FireLight. Clients will still be able to pull all values simply by not checking that box.

FireLight Rules API

  • CMR 202519 Expanded the FireLight APIs to support hitting FireLight to run rules and return validation messages and data item IDs. With this project, firms can use the FireLight Rules API for any activity type (needs determination, presale, e-Application, etc.) and return dataitem ID values and validation messages. The input is a JSON file passed to FireLight via the API. The FireLight rules will run and return a response of updated dataitem ID values and validation messages. This will give carriers/distributors the ability to leverage all of the rules already built into FireLight for other channels. Licensing is required to use this API.

Obfuscation

  • CMR 199769 Added the ability to obfuscate data on screen so that it cannot be seen by individuals walking by in high traffic areas, such as bank lobbies. Users can selectively display the data. When focus is on a given field, the contents are visible. The user can display all of the data on a page to review for accuracy by clicking a toggle button in the page title bar to show or hide all data entry on the page for a limited period of time. Obfuscation encompasses all fields on the UI—fields in both carrier and distributor form packages and FireLight UI fields that are part of the activity workflow.

    Obfuscation Impacts:

    • Admin – Groups, App tab: Under Obfuscation/Screen Security settings, added "Enable Obfuscation" and "Timeout" options for enabling obfuscation and setting the number of seconds for the data to remain visible in periods of inactivity. Post Submission Status

The workflow includes the provider polling an outside source to return the current status. FireLight updates the Display status and External status in the Recent Activities and All Activities as well as notifies the agent of the status change. Notification to the agent occurs with the status change, an onscreen icon, message center update, and an email with the details of the status.

The Review Queue statuses will now reside in the Display Status reflecting if the activity has been approved, declined, or needing additional information. Print Only Forms Management (Mutual Fund Forms, 529, Life)

  • CMR 199470 Introduced a new database partition for Hexure to manage products, forms, packages and sales agreements that are not sponsored in FireLight (owned and managed by a carrier) and deploy those to distribution. We are calling these Print Only products. All forms and packages managed by Hexure that do not have a supporting carrier to receive submission will only be available for Print; hence, they are called Print Only Products.

    Hexure can load and manage any form a distributor needs available in FireLight for a Print Only activity including account opening paperwork, mutual fund forms, 529 forms, life forms, etc. Additionally, Hexure can manage rate tables and product details when a sponsoring carrier does not want to manage these tasks themselves.

    Access to products: Distributors is able to access these forms using the same carrier/product/jurisdiction selection available today for the existing activities, but non-sponsored products will use the Print Forms logic. The package type delineates the activity type (Application, Post Issue, etc.). Added a new "Include Print Only Products" setting in each existing activity (except E-Delivery) in the Admin/Product Configuration/Activities page. This setting allows a one-stop location for agents to access both full sponsored activities like e-Applications and Print Forms. Distributors will still need to grant product access, but if the product is enabled, these non-sponsored products is available under the normal application, subpay, print form, post issue, quote, and illustration activities. If accessed via application, quote, illustration and post issue, FireLight logic will identify that the product is non-sponsored and only allow Print. e-Signature will not be available for non-sponsored forms. Upon submit, no submit will occur. Distributors, Licensing is required to use these forms.

    Print Only Forms Management Impacts:

    • Admin/Product Configuration/Activities - added a new "Include Print Only Products" option
    • Admin/Organization Configuration/Groups/Product Access – added a new section for Print Only products
    • FireLight UI/Activity – new available products when "Include Print Only Products" option is selected for the Activity in Admin

  • CMR 205762 Automated Review Queue - Added the ability to support fully automated review queues with automated web service calls within the review queue that will provide automated approvals and rejects. The automated review queue performs web service calls through the use of the provider, which allows clients to build customized review queue workflows with the specific web service call needed to support the desired workflow. For example, clients can use this workflow and the provider to build automated queues for underwriting reviews, funds availability checks, NIGO checks, etc. As the web service is triggered, response messages is sent to the queue (i.e., reject or approve) to move the activity either to an approved-submitted or rejected status.
