Accessibility - FireLight App UI (Work in Progress)

  • CMR 243549 (Carriers and Distributors) Enhancements to support Web Content Accessibility Guidelines (WCAG) - Finish Level A, Begin Level AA Adherence to Web Content Accessibility Guidelines is critical to navigation for individuals who require assistance with web use. Assistive technology is also negatively impacted as a result of web content that does not conform to defined accessibility guidelines. There are three levels of Web Content Accessibility Guidelines (WCAG) compliance: A, AA and AAA. The levels of accessibility are broken down into four principles: Perceivable, Operable, Understandable and Robust (POUR). Enhancements are needed for FireLight to complete compliance for the Web Content Accessibility Guidelines for Level A and AA. Enhancements to FireLight will continue; pursuing the desired goal of accessibility Level AA.

    NOTE: In Admin Help, see the FireLight WCAG Guide for more information.

    Impacts:

    • Clients who use Automated testing

    • Embedded clients who interact with the object model of FireLight web pages

    • Multiple impacts across the system: Impacts on client branding will need reviewed (Hexure will make updates for clients, clients needs to review)

    • IXN Embedded Quote platform is not expected to be WCAG level A or AA compliant until a future release

E-Signature - DocuSign

  • CMR 237379 (Carriers and Distributors) Support the use of DocuSign within FireLight With this enhancement, FireLight is integrating with DocuSign to allow the option for our clients to use the DocuSign services for their e-signature captures. This is an available option for only DocuSign Remote Signature.

    Assumptions to consider with this project:

    • DocuSign must be honored anytime it is enabled regardless of the Org initiating the application. Meaning, Carriers enable DocuSign and deploy the app to the distribution. FireLight must honor DocuSign for the distribution as well as honor it in the carrier Org.

    • DocuSign will support Remote Sign Only.

    • FireLight is handing off the signing process for remote sign when DocuSign is enabled. Any Email notification, Signing and audit is completed within the DocuSign environment.

    • When DocuSign is enabled, the Signature Audit will not be captured within the FireLight Signing audit, however FireLight is receiving a signature audit back from DocuSign that is attached to the application and sent within the client's payload.

    • FireLight will need the signed documents with the digital signature applied and the DocuSign signature audit returned 100% complete to FireLight. The clients that need DocuSign implemented have the DocuSign digital signature, tamper evidence and audit approved.

    • FireLight must be able to retain the digital signature without breaking the hash for the DocuSign documents.

    • FireLight must be able to send documents completed by DocuSign to the different firms (Distributor receive X&Y forms and Carrier receives A, B, and C).

    • FL will need a bi-directional ability to cancel the eSignature (A user can cancel the signature from FireLight and from DocuSign).

    • The User MUST have authentication when sending PII information to DocuSign, so FireLight will still send the last 4 digits of the SSN and DOB of each party to DocuSign, as well as sending the signer's mobile phone number to initiate SMS Text Verification through DocuSign.

    • Clients wanting to use the DocuSign service from an organization level will need to contact Hexure in order to enable the service.

    Impacts:

    • FireLight E-Signatures - Specifically the "Send Email Request" or Remote Signing capability

    • Organization Settings - Enabling the DocuSign option

      Third-Party Configuration Table - service type for DocuSign, new column for sub firm "Control Group"

    • App Changes - New UI page for "DocuSign Signing Ceremony" information that can be entered and initiate the call to DocuSign

    • Reports - the Third-Party configuration Report is altered to show the DocuSign signed applications and add the Control Group

    • Audits - New audits is created to capture the calls to DocuSign, any cancellations, and the completion of the signature process

    • Display Statuses - New display statuses will show when the signature process is in process, cancelled, or completed through DocuSign

    • Transfer - Transfer is disabled when DocuSign is used, even if it is selected within the groups settings. Share will not be impacted

    • Client Fill and Sign - Client Fill and Sign is disabled when DocuSign is used, but will also notify the user of this through a tool tip

    • Rules

  • CMR 248383 Save on initiation and completion of async (provider) call. Create an attribute or element on the node that is configurable to call on the callasyncaction rule node. This would allow a save to happen on both initiation and completion of the async call to a provider. If this attribute is set, 1) a permanent save of the activity data (from the current data session) is performed before initiating the async action, and 2) when the async action completes, the pending data is updated in the saved app data, so that if the user never performs any more data entry in the session, the updated values from the async action is persisted with the activity. If the user does continue data entry, it will still check for pending changes and update the current session values (like it does today) so that if the user saves the activity later on, it will include any pending data and their latest data entry inputs.

    This enhancement supports allowing the provider to trigger a save prior to a call and after a call is completed (i.e., Milliman). The benefit of this enhancement is, if a user exits the session without saving the data needed for a web service call, we will have already saved the data and it can be retrieved. The same is true if the user exits after the call is complete but does not save, the data from the call is stored. Implementing this enhancement will eliminate a compliance risk.

    Impacts:

    • N/A - For the 2.21 release, this is a temporary fix for a client. Auto Save implementation in a future release will solve this for all clients.

  • CMR 239391 (Carriers and Distributors) Workflow Separator FireLight needs the ability to separate out the workflow so firms can control progression to the next page based on all of the validations being cleared. For example, present a final checkbox once the app is 100%. Today, to do this technically you cannot be 100% because once all data is entered, a new requirement is presented which means the app is no longer at 100%. The old consumer portal had this logic where the final acknowledgement and the next button would not show until the previous screen was 100%.

    This enhancement will add the ability for the admin to configure a workflow separator between wizard screens that must be clear of all errors before the user can proceed. This enhancement will eliminate the need to control the workflow via rules which is not smooth and error prone. From a UI perspective, when the separator is added, the wizard workflow will hide the Next button and grey out the screen navigation until all the errors on the previous screen are corrected. There are many use cases for this. One common use case is for agent attestation prior to continuing to signing.

    NOTE for Embedded Clients: With the new Section Break feature the BundlePage has been modified. If Embedded clients are using the Section Break feature they may need to modify JavaScript to handle pages that are visible but not selectable. This should not affect Embedded clients not using the Section Break feature.

    This is an optional enhancement. Clients that do not want to add sections to their wizard will not be impacted. Those that add section breaks will have the below impacts.

    Impacts:

    • Wizard Design - New object of Section Break is added to the Page Order screen to group pages within a wizard and create the sections.

    • Wizard Validations - Section Break will impact only Wizards and not forms.

    • Wizard Navigation - User is unable to navigate to another section of the wizard until the prior section is 100% complete with no errors.

    • Embedded API - the API for embedded that returns the wizard pages and navigation graph will now show the section breaks and pages within the section breaks.

    • Rules - Rules can be created using section break names.

    See "Add and Delete Pages and Section Breaks" in Admin Help.

  • DR 248513 Missing Page of Template Form in Display/Print PDF. Problem: client expects to pass an uploaded template document from a pre-sale activity to an application and have it function as an uploaded template document in the app. This is not currently supported by FireLight. Changeset 48194 for iConnect 224368 in the 2.21 branch sets the doctype to NeedsDetermination for documents fed over from a needs activity, which is part of the solution for this. However, it still doesn't operate as a template in the child because the document tag is also being set to NeedsDetermination (for all documents from the original activity). This is incorrect, because a) there must be only one template document with a given doc tag in an activity, and b) the client wants to use the same document tags in the pre-sale and application to identify these documents. Resolution: Modify the base system to still pass through the original document tag, allowing a) each document to be distinguishable and support operating as a template document upload, and b) organizations to configure their template document types for their needs. NOTE: This only addresses template documents coming from a pre-sale or illustration. Additional changes would be required to support traditional templates documents (doctype AdditionalDocWithSignature) being fed over other activity types and functioning as uploaded templates in the child activity. It is unknown whether this is desired/needed and what impacts it could have.
