Admin: Wizard and Form Designers

  • DR 256054, 256066 Invalid characters in the element name. Problem: In 2.22, the characters in the field name/Data Item ID were limited to only letters, numbers, underscores and brackets. Resolution: Allowed all characters except apostrophes in the field name/Data Item ID.

E-Signature Requests

  • DR 256110 Canceling e-signature request removes all review queue information for activity. Problem: When revoking an e-signature request (really, any request type), all review queue information for the activity was also being deleted. FireLight Embedded

  • DR 255588 Suitability Packages not populating at end of AOE in "Print Documents" last page. Problem: It appears the Packages and Forms are out of sync between the DB and Session State. However, sometimes, when the export value in a radio button is boolean, e.g. "true," and the value is set by the rules without specifying the type as String (e.g. <const value="true" />) the actual value stored in the data item is capitalized, e.g. True/False, which is interpreted as not a match. Replace Rule Set and Primary Rule Set in Multiple Packages - Package Level

  • CMR 250572 Complete (Carriers and Distributors) This enhancement will allow replacing Rule Sets assigned to multiple Packages at the same time. This new feature will also replace Primary Rule Sets in the Forms/Wizards if present in those Packages to make the replacement smooth for the clients.

    Primary Rule Sets are required in a Forms or Wizards when there are rules with Form Index. Today, replacing a Rule Set in the package breaks the linkage between Form Index fields on the form/wizard and the rules written using Form Index fields. Rule Sets are intended to be swapped in and out of packages for product changes but the current breakage when there are Form Index makes the swapping more difficult. With this enhancement, when swapping rule sets in a package, the swap window will present a list of packages that have the original primary rule set and provide a checkbox swap the Primary Rule Sets in the Forms and Wizards that have the original Rule Set. This swap will occur in the Package level to reduce the chances of missing a location where the Rule Set swap would break the Form/Wizard Form Index.

    Impacts:

    • Product Configuration -> Packages -> Replace Form/Wizard in Package (link)

    • The existing link "Replace Form/Wizard in Package" on below Package page is updated to accommodate this new feature.

    • A new tab is added to window "Replace Form/Wizard in Package" to replace Rule Sets in Packages. From this tab, with the event of replacing Rule Set in the selected Packages, Primary Rule Set in the Forms/Wizards present in those Packages will also be replaced.

    • Product Configuration -> Packages -> Replace Form/Wizard/Rule Set in Package (link) - Within "Packages to Replace" box, column "Primary Rule Set" is displayed to indicate a "Yes" or "No" value for packages in case the forms/wizards present in those packages have the selected Rule Set as a Primary Rule Set. When replacing the Rule Set for the selected Package(s), Primary Rule Set will also be replaced for the Forms/Wizards under the Package(s) that has a "Yes" value.

Enrollment Activity

  • CMR 236720 In Testing (Carriers and Distributors) The purpose of the following workflow is simplification of the Application build & Participant signature aspects of the enrollment process. This new functionality will improve the workflow experience for Agents and Case Managers (CM). The streamlining of this functionality will allow users to manage the workflow much more efficiently and allow the sponsoring carrier to enter a market involving larger group/enrollment scenarios.

    A new activity module is created in FireLight to support this new workflow. Major aspects of this activity will include the following:

    • Case Managers is the primary FireLight users in the workflow. The CM will access FireLight through their existing SSO.

    • Case Managers will start the new enrollment activity by selecting it from the FireLight home page. On selection, the activity will launch and will present 3 data collection screens/views.

    • “Plan” page – Plan information is collected on the first screen. Plan information may include, but is not limited to Employer ID (required), Employer Name, Ownership, Plan Type, etc. Full definition of information collected to be defined. This information is passed via common tag to linked/child applications that is created for each participant. Data collected/required will vary by organization. It may be necessary to assure fully completed plan information exists prior to allowing navigation to the Participant page. For Day 1, the Enrollment process will only support the Primary Owner and the Primary Insured.

    • “Participant” page – The participant page will allow addition, deletion, and update of participants in the plan. The primary method of adding participant information is through census upload. The Participant page is presented in a grid view with columns to be defined. The grid view must support validation to identify situations where required information is not present or is formatted incorrectly.

Generate Unique Addendum Doc for Each Form in the Activity

  • CMR 244655, 243975 Complete (Carriers and Distributors) This feature is a change from the base addendum function to a custom approach that provides a unique addendum for each form present in the Activity. This is a global change that will impact all clients. The feature will generate an individual addendum document for each of the forms present in the activity, at the time of Print PDF action. All soft fields present on a form is placed into a separate addendum document specific to that form. Each individual addendum will generate directly after the corresponding form and not at the end of the package.

    This change will impact Print PDF functionality for all the activity types, except for Print Forms activity. Addendum document will not generate for Print Forms activity after the Fillable PDF enhancement is implemented with 2.22 release.

    Impacts:

    • When soft limit is set and exceeded, impacts to FL App -> Activity -> Other Actions -> Display/Print PDF - Show Addendum doc for each form in the activity in the same sequence of the forms. This change will not impact Print Forms activity.

    • Provider updates - Prior to December, outbound feeds contained a single addendum. After the December release the outbound feed could add an addendum for each form. Integration providers are being evaluated individually to determine additional impact due to the way some providers manage addendums. Clients should also evaluate their systems that are receiving an outbound feed from FireLight to ensure there is no specific logic around how addendums are being processed.

      Example: Previously, the addendum would be shown as a form number of "FLI-10000" and now it is shown as "FLI-10002," "FLI-10003," etc.

      The new addendum name is also "Addendum - <formname>" whereas previously it was "Application Addendum."

    • Carrier forms may need to be set up/evaluated for the Addendum reference order if a specific order is desired. Addendum references is numerically applied from left to right, top to bottom and not defined by tab order of the form.
