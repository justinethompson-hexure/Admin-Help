2.10.0.xxx August 11, 2017 Minor Production Release

Enhancements

Display/Print PDF Page

  • CMR 163326 (Available for all clients) A SML work order added a 'Check/Uncheck All' checkbox above the left column of the grid on the Display/Print PDF page in FireLight. Use the new checkbox to select or clear all documents in the list. This feature is enabled when an application is unlocked or the Enable Print Individual Forms option is selected on the App tab of the Groups page in FireLight Admin.

Email Notifications

  • CMR 147639(Available for all clients) A SFG work order added the ability to send an auto-generated email from the organization email address to the agent and/or reviewer when an application has been submitted after being approved in the manual review queue. To use this feature, ensure Allow E-Approval is selected on the Organizations page to add the manual review queue step to the workflow. In FireLight Admin Profiles on the Groups page, App tab: select the Enable Submit Email for Agent checkbox to enable the Agent Completed Application email and select the Enable Submit Email for Reviewer checkbox to enable the Reviewer Completed Application email on the Email page. To make use of the new email template replacement tags, contact your Strategic Account Manager to add the <CLIENT_NAME> and <FLI_CONFIRMATION_NUMBER> values to the Application Search Items stored for each application. The <CLIENT_NAME> tag requires a dataitem of the same name to be added to the application form via a rule. If policy number is not being set in the Provider, you can also add a dataitem to the form to set the value for <FLI_CONFIRMATION_NUMBER>. See the Email page in Admin Help for a rule example and a list of available template tags for the new templates. Also added an Organization option to allow organizations to set the frequency of an auto-generated submit reminder email to agents. The Submit Reminder email is sent if one or more applications have completed the signing ceremony for all signers, but the applications have not been submitted. The frequency of the email is based on the option (Daily, Weekly, Never) selected in the Submit Reminder Frequency list on the Organizations page in FireLight Admin. For example, if Daily is selected, the agent will receive an email a day after one or more applications have completed the signature process. The email will list the application name, policy number, and client name (if client name is added to the application search items for the organization) of each of the signed applications that have not been submitted. (The list does not include purged applications.) To add client name to the application search items, contact your Strategic Account Manager. See the Submit Reminder email template on the Email page in Admin Help.

  • CMR 155605 (Available for all clients) A FST work order added the ability to send email notifications that provide access to the signed application documents after each client signs an application. (This is in addition to the existing final Download Completed Application email notification.) In FireLight Admin, added an editable Leave-Behind Email template to the Email page in Profiles. To enable the new email template, select the Enable Leave-Behind Email checkbox on the App tab in Groups for the role codes that will receive the email. Sending of the email message will also be recorded in the application audit history.

List of Required Signers Page

  • CMR 157030 (Available for all clients) A KOC work order added the signer's name next to the signer role on the List of Required Signers page in FireLight. This ensures the correct person is signing the document set during the signing ceremony. This is especially useful when the Enable Single User to Sign Multi Signer Types option is selected on the App tab of the Groups page in FireLight Admin.

Review Queue

  • CMR 160336 (Available for all clients) A USA work order added the ability to include the Reviewer's Internal ID on the Accept and Reject messages from the Review Queue. Clients can structure the 1228 to include the name and email in the correct location and additional information could be included in the 1228 for the Provider to use. Contact your Strategic Account Manager.

Rules

  • Additional Feature (Available for all clients) Added two new Rules nodes, 'deformat' and 'elseif', to the Rules window of the form Designer in FireLight Admin. Use the 'deformat' node to strip characters from a string that do not match the format. Use the 'elseif' node to write if-elseif-else statements. See the Admin Help for examples of the new nodes.

Signature Types

  • CMR 163897 (Available for all clients) An ALZ client work order added three new signer types. On the form Designer page in FireLight Admin, the following new options to the Signer Type field property were added to the Signature control: Investment Advisor Representative, Investment Advisor Representative 2, and Investment Advisor Representative 3.

Fixed Defects

  • DR 149953 Download of Apps. Users could only download 4 applications at a time. This has been fixed by making the download go to a new tab, increasing the download maximum file size to 150MB, and increasing the maximum time for a request to 8 minutes.

  • DR 156298 Product not available in UI as of Sales Agreement date. This has been fixed.

  • DR 160719 Wording of the review status it's being shown differently in UAT. This has been fixed. "Review Status" is displaying properly in "My Recent Applications" section of main page.
  • DR 161966 App data was lost when transferring from Console. This has been fixed.
  • DR 163181 Carrier-only documents were visible to the client. This has been fixed by modifying the client view of application documents to exclude carrier-only documents.
  • DR 161931 Signatures coming through as a line or incomplete. This has been fixed by increasing the size of the captured image size from 150 to 2048 pixels.
  • DR 162448 Agent’s name is missing in client fill and sign email. This has been fixed.
  • DR 163405 Reviewers with an apostrophe in their username are unable to accept/reject apps. This has been fixed.
  • DR 163473 Carrier Selection is showing in prod despite only one carrier having product to select. This has been fixed.
  • DR 163810 Signature Email link generates "Server Error." This has been fixed.

2.10.0.169 June 8, 2017 Production Release

Enhancements

Cooperative Technologies Yellow Pages Report

  • CMR 157721 Add third-party usage report to track the usage of yellow pages (LFG)  – Added a Third-Party Service Usage - Cooperative Technologies Report for LFG. See the ReportsReports page in Admin Help to view a sample of this report.

Encryption and Signing of PDF Files

  • CMR 155174 The package being SFTPd is encrypted but not digitally signed  – When encrypting and transferring files using PGP file encryption, implemented the ability to sign the PDF files with the FireLight private key and encrypt the files with the client organization's public key.

Forms Designer

  • CMR 157731 Date format on the final package after signature is attached  – In FireLight Admin, added the ability to alter the format of the date included in the signature control stamp as well as the output date format on forms in the PDF by adding a Date Format field property with multiple date format options in the Signature control and Calendar control field properties in the form Designer.

Integration

  • CMR 153555Consume and process a 103 through a REST web service – Added base enhancements that provide back office capability to OCR existing paper applications directly into FireLight, return any NIGO responses, and finish normal completion processing of the application. Client makes a SOAP request to FireLight with application data in ACORD 103 format (with KeyedValue items). FireLight maps (or attempts to map) the data of the 103 to a new application.

Jurisdiction Limits

  • CMR 157979 Remove jurisdiction limits  – Modified the FireLight jurisdiction approval design to allow more than 63 jurisdictions. Any number of additional jurisdictions can be added to the system for organizations that require them.

LexisNexis Integration

  • CMR 135461 Integrate with LexisNexis for full 3rd party Authentication – Added integration with LexisNexis, a third-party client authentication vendor. The FireLight client needs to contract directly with LexisNexis for the services. FireLight incorporates the services Instant ID and Instant Authentication into the client verification process during the signing ceremony for both the Sign Now and the Sign Later (external signing) processes for more deeply authenticating the signer. For implementation details, see Features. (The enhancement will look at the forms in the signing package to determine if any of the forms contain a LexisNexis authentication, and if they do, LexisNexis is called using the contracted firm’s credentials.)

Other Actions Menu

  • CMR 160021 Other Actions/Documents/Document Type list  – In FireLight, added "Custodian Paperwork" document type to the Other Actions menu/Documents/Document Type list.
  • CMR 158126 Print individual forms from a locked application  – Added the ability to print individual forms from a locked application. In FireLight Admin on the Profiles/Groups page, added a new Enable Print Individual Forms checkbox on the App tab. Select the checkbox to enable the individual checkboxes on the Select Documents to Print dialog that appears when the user clicks the Display/Print PDF button after an application has been locked in FireLight. NOTE: In Groups, you cannot select the Disable Display/Print PDF and the Enable Print Individual Forms checkboxes for the same role code.

Redtail Integration

  • CMR 155962Update Redtail to use credentials from the application data  – Modified the Redtail provider to get user credentials from a data item within the application. This allows the organization to create a form that captures the credentials and enable a Redtail integration from individual user accounts. DataItems FLI_REDTAIL_USERNAME and FLI_REDTAIL_PASSWORD can be used to store credentials for the Redtail, instead of organization config settings. You need both password and username in dataitems before it uses the dataitems, and these dataitems will override any config username/password.
  • CMR 159928 Additional Fields Needed For Redtail  – Made changes in FireLight to map the following fields provided by Redtail: Job Title, Date Of Birth, Marital Status, Party Type, and SSN.

Reports

  • CMR 158783 New Activity Reports Needed (AEA)  – Added the DTI Detailed Submission Report, FireLight Detailed Submission Report by Channel and the FireLight Carrier Submission Summary Report for AEA. See the ReportsReports page in Admin Help to view samples of these reports.
  • CMR 158790 Daily Activity Report changes  – Added an optional Role Code column to the DTI Detailed Submission Report in PDF and Excel formats.

Signature Types

  • CMR 159805 Added the following Signature Types for SML – In FireLight Admin, added the following to the Signer Type field property for Signature controls: Grantor, Grantor 2, Translator, Replaced Policy Owner, and Replaced Policy Owner 2.

Whitepages.com Integration

  • CMR 153534 Lincoln Reverse Client Lookup - White Pages – Added integration with Whitepages.com via an API to pull client data into FireLight. The FireLight client needs to contract directly with Whitepages.com for the service. For FireLight implementation details, see FeaturesFeatures.

2.10.0.35 April 7, 2017 Production Release

Enhancements

Application Statuses

  • CMR 137812 Extend the available list of statuses that are visible for an application – Added "Review Status" to the Recent Applications and All Applications views to indicate review status for Approved, Declined, and More Info Requested. Added custom declined text for pop-up notification when entering an application that has been declined within the internal review queue. In the manual review process, the application is unlocked when the reviewer declines, so the review status is not updated to Declined. Added additional information to the custom signature step text for pop-up notifications on whether the agent, client, or both signatures are missing from a locked application.

Client Email Address Mapping

  • CMR 153188 Map client email address to client fill, sign, fill and sign and sign in person screens filled out by producer  – In FireLight, mapped the InsuredEmailAddress and PolicyOwnerEmail fields to the client verification screens for client fill, client sign, client fill and sign, sign in person, and email request to sign.

Cooperative Technologies 1035 YellowPages Integration

  • CMR 157526 Use carrier settings for Cooperative Technologies 1035 integration  – Changed FireLight base to use the carrier (form) organization credentials if not defined in the distributor for Cooperative Technologies 1035 integration.

Deployments

  • CMR 109486 Admin Deploy: List and display which form fields are being deployed on the Deploy tab  – In FireLight Admin on the right of the Deploy page, added form history links in the totals boxes. Clicking the links opens a Forms History dialog that lists the form changes.
  • CMR 146347 Schedule Deployment: Ability to see all currently scheduled deployment is needed.  – In FireLight Admin on the upper right of the Deploy page, you can now view a list of all scheduled deployments by clicking the View All Scheduled Deployments link.

Emails

  • CMR 144135 Enhance emails to support branding in emails  – Added the ability to support branding with logo and colors in emails by using HTML. An Email Preview of the HTML template is included in the Email Profile page in FireLight Admin. Organizations need to contact their Strategic Account Manager(PM) and send the initial version of their HTML template to their FireLight PM to be loaded by Hexure—very similar to the initial branding setup. This ensures that the HTML is valid and set up correctly. See the Email Profile page for a sample HTML template and email preview with logo and color branding added.

Fees, Expenses and Commissions Repository

  • CMR 156543 Fee Repository Enhancements  – In FireLight Admin on the Fees page, added a Delete Files button. In the Download column of the grid, select the checkbox for each file to delete and click the Delete Files button.
  • CMR 157878 DOL - Limit fee fund range query to product codes within the PPfA  –  In the fees, expenses and commissions repository, enhanced the fund queryfee rule to consider only data items whose product code suffix is defined in the PPfA Lite.
  • CMR 158149 DOL - Repository changes needed for CST commission schedule ID & type  – In the fees, expenses and commissions repository, modified the CST query to use common tag data items, FLR_Commission_Schedule_Id and FLR_Commission_Schedule_Type, to locate the correct commission data, which must be initialized by the carrier as needed.

Generate Link Without Email for Requests

  • CMR 146337 Access Forms to Sign without Following Link in Email  –  Added support for a workflow that allows a client to access a fill request or the signing ceremony via a FireLight Client Portal. Clients that do not have access to email for remote signing or client fill requests, can go to this website, enter their application unique ID, authenticate, and then sign or fill. The advisor needs to provide the shortened website URL and Passcode to the client via phone or text. The workflow supports authentication prior to signing just like the normal FireLight workflow. This enhancement also includes shortened website URL links for external requests sent via email.

Integration

  • CMR 157354 Override dataitems provider call is needed for all transaction types  – Removed the check for application transaction type to allow a call into the provider for override data items regardless of the transaction type.

Mapping

  • CMR 134976 Admin: Mapping Section Enhancement – Added the ability for distributors to link two data items together—when one data item changes, the linked data item also changes. This is at the organization level and goes across packages and duty types. Modified the rules engine to allow certain rules to run recursively, where if one rule affects another rule that affects a field, it is run again. The Mapping Profile contains a new DataItem Sync tab for defining linked data items. The Deployment tab on the Deploy Profile page contains a new Data Item Sync Configuration synchronization scope for deploying the linked data items.

Rules

  • CMR 146992 Identify data items that are in a rule but not in any forms in the package  – In FireLight Admin, added a new optional "addifnotfound" parameter to the diget rule node, which adds the data item if it does not exist. This can be used when sharing a master rules form across different packages, and some of the defined rules may include control names that may not exist in each package where the master rules form is used. In Advanced Rules Category and Element Definitions Reference, see "Example 3" under the "|diget" node in "|GetSet" for an example using the new parameter.
  • CMR 155319 New Rule Node: To act as if button is clicked  – In FireLight Admin, added a new Rule Node, clickcustomaction, that can be used to write a rule to fire a click event of a custom action button. The custom action is called per the properties set for the custom action in the Button control. In Advanced Rules Category and Element Definitions Reference, see "|clickcustomaction" under "|Control" for an example of the rule node.

Fixed Defects

  • DR 156596 Reminder email contains html and will not send. This has been fixed.
  • DR 157224 Create an Application Button Fails with Post Issue Forms. This has been fixed. It will now create either an application or whatever single transaction type is set.
  • DR 157289 Extra spacing in HTML emails. When sending an HTML email, extra line spaces occurred in the HTML version of the email. This has been fixed.
  • DR 157362 Form field is missing from application. This has been fixed.
  • DR 157730 FTA - Call() Not getting hit for Post-Issue Service Forms. This has been fixed. Provider is now called in the submit logic to allow normal outbound transmission for all transaction types.
  • DR 157765 Implement a new SFTP component to support new ciphers. This has been completed. We implemented a new component to handle more ciphers for our outbound SFTP communications from our providers.
  • DR 157901 Handle CT Integration endpoint difference in Prod vs test. This has been fixed.
  • DR 158698 Update existing app fails for NewBusinessService call. Updating an existing application was not working. This has been fixed.
  • DR 158948 DOL - Fee repository not loading fees stored for all organizations. This has been fixed.
