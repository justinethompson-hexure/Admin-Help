FireLight v2.13 Release History

2.13.0.247 December 21, 2018 Minor Production Release

Enhancements

Admin

  • CMR 187306 Added a new Utilities tab in FireLight Admin for providing maintenance utilities. In Utilities, added a new Transfer Activity Utility page to use for permanently transferring all activities from one user to another user. To enable this feature, go to Profiles > Groups > Admin tab and select the 'Enable View Transfer All Activities' option. For details, see the Utilities and Transfer Activity Utility topics in Admin Help.
  • CMR 191806 On the Profiles > Products page, added a new "Accident and Health" option to the Product Type list.

Applications

  • CMR 188708 Have a hyperlink show in the PDF sent to the client. When an application is sent to the client and they open the PDF, the hyperlink within the PDF file is now available. Hyperlinks are now stamped on locked so they appear on downloaded forms.
  • CMR 190542 Added a new navigation bar for Wizard e-Applications that gives an updated look that heavily uses CSS in order to allow for greater client customization. This project does not affect the current navigation/status bar for the regular forms implementation of FireLight.

Benchmarking

  • CMR 155968 Added the ability to sync Benchmarks from UAT to Staging to assist in testing efforts. Enabled Benchmark import and copy functions. You can now export benchmark cases from one environment and then import the benchmark cases to another environment for testing. See the "Benchmarking" topic in the "Manage" section of the FireLight Platform User Guide for details.

Buttons

  • CMR 186616 Added a "Please Wait" spinner for buttons that take awhile to execute on forms and wizards so users will know to wait until the process completes.

Client Fill and Client Fill and Sign

  • CMR 190701 Added the Client Fill and Client Fill and Sign functionality to wizard activities. Now agents can send Client Fill and Client Fill and Sign requests from the Other Actions menu for both forms and wizards.

Cooperative Technologies 1035YellowPages

  • CMR 189194 There were some discrepancies with the handling of the return values from CT 1035 Yellow Pages. After meeting with Cooperative Technologies, we fixed the logic for FLI_ALLOWESIGN values: use location information instead of last requirement; use the CT GetCarrierInformationV3 method to retrieve all information in one call; avoid double search call when pressing Enter; fix duplicate adding of phone/email when selecting the same company multiple times; cleanup.

E-Delivery

  • CMR 190021 Added the ability for carriers to do E-Delivery reversals for reissue requests sent back to FireLight. Firms send in a reversal via API (121) using the Reversal TransMode Typecode of 8. This retains the product ID and policy number, but removes any previous package details and resets the status. The audit indicates a new case was sent, but the old audit remains. The TransRefGuid, product ID and policy number must match the original. Only cases in request reissue (display status of Change Requested by Agent)/request change (display status of Change Requested by Client in client directed cases) can be reversed. The status of the reversed case is reset as if it was a new case. If it is agent directed, the status goes to Pending Agent Review. If it is client directed, the status goes to Pending Client Review.
  • CMR 190048 Disable edit email and name needed for e-Delivery requests. The "to" name and the "to" email are now read-only if the values are pre-filled by the 121 and cannot be modified. This only affects the client fill and sign for e-Delivery.

Google Address Autocomplete

  • CMR 186639 Added the ability to use the Google Place Autocomplete Address database in FireLight for both forms and wizards. In FireLight Admin, added a new input field type with a Data Group property to the Design pages for Forms and Wizards. Add these fields to forms and wizards so users can begin entering an address in the Address Autocomplete field type and press Enter to open a Google Places dropdown list of addresses from which users can select an address to autocomplete the associated address fields on the form or wizard. See the Address Autocomplete (Forms) and Address Autocomplete (Wizards) topics in Admin Help for Admin instructions. See the "Address Autocomplete" topic in the New Activity > Data Entry Step of the FireLight Platform User Guide for user instructions. This feature is not available for Console. This feature is enabled for all organizations. Contact your PM to disable this feature. NOTE: This service is not Health Insurance Portability and Accountability Act (HIPAA) compliant. The integrations are configured so that we can disable them for an organization, which we will do if we enter into a HIPAA Business Associate Agreement (BAA) with the organization.

Illustrations

  • CMR 189803 Added the ability to send an email to an agent with a link to an illustration PDF output. The user clicks the 'Send Email' button on the Illustration activity navigation bar to send an email to the agent. The agent clicks the link in the email to open the saved illustration PDF output in a browser. No UI options are available. There is no limit to the number of emails/variations that can be sent with a single illustration (e.g., sales desk person keeps changing inputs in the same illustration to generate emails with link to output). See the email template in Profiles > Email for Admin instructions. See the "Illustration" topic in the "New Activity" section of the FireLight Platform User Guide for user instructions. This feature is not available with the Console.

Postal Address Validation

  • CMR 186362 Added the ability to integrate with the Lob.com address validation API (https://lob.com/docs/java#us_verifications) to accomplish address validation on a set of fields without the user needing to trigger the validation through a custom action button, etc. Validation is set through FireLight Admin via a rule node that returns various string values, allowing the rule to further determine logic based upon the success or failure of the validation check (such as error messages or warnings). See the Address Validation topic in Admin Help > Features for Admin instructions. In the FireLight Platform User Guide, see the "Address Validation" topic in the New Activity > Data Entry section for user instructions. NOTE: This service is not Health Insurance Portability and Accountability Act (HIPAA) compliant. The integrations are configured so that we can disable them for an organization, which we will do if we enter into a HIPAA Business Associate Agreement (BAA) with the organization.

Print Forms Activity

  • CMR 190206 Implemented a forms repository inside of FireLight for a carrier's new business applications and forms that currently exist on the carrier's website. The carrier sends users directly to the new Print Forms activity via SSO or by clicking the Print Forms button on the Home page in FireLight to access and print their forms from the repository without having to fill in forms or submit to the carrier. This gives agents an easy way to access and print the forms the carrier has loaded into FireLight. To add this feature, contact your Strategic Account Manager. See Print Forms Activity in Admin Help and the "Print Forms" topic in the "New Activity" section of the FireLight Platform User Guide for instructions.

Provider Lookup

  • CMR 190009 Added the ability in FireLight Admin to add a button or a custom action to a form or wizard that would search the National Provider Identifier Standard (NPIS) database for a specific healthcare provider or office inside FireLight and pre-fill associated provider information fields using the existing CRM functionality. See the Provider Lookup topic in Admin Help > Features for Admin instructions. In the FireLight Platform User Guide, see the "Example: Provider Lookup (Form View)" topic in the New Activity > Data Entry > Action Buttons section for user instructions.

Review Queue

  • CMR 173297 Added the ability to route cases from a distributor instance of FireLight to the Carrier instance of FireLight Pre-Submission (Manual) Review Queue and to conditionally pre-populate and disable reviewers through the following data items: FLI_LOCK_REVIEWERS - if set to "true" the user will not be able to add or clear the reviewers; FLI_REVIEWER_NAMES - a semicolon-separated list of reviewer names; and FLI_REVIEWER_EMAILS - a semicolon-separated list of reviewer email addresses. The number of names must match the number of emails. If not, the reviewers will not be pre-populated. The distributor needs to enable the 'Allow E-Approval' option on the Activity tab in Profiles > Organizations to enable the Manual Review Queue in their organization and ensure that there is a rule defined for all of their other carriers that turns FLI_REQUIREMANUALREVIEW data item to false.

Signatures

  • CMR 189648 Added identifiers to the text in the eSign Declined page to allow for customization of the text in the CSS file for each organization.

Training Videos

  • CMR 182779 Added FireLight training videos to the FireLight application. A user clicks the Training button on the navigation bar to open a new tab or window with hyperlink images to training videos based on their current location in FireLight. To enable this feature: (1) select the 'Enable Training' checkbox on the App tab in Profiles > Groups and (2) go to the Training tab in Profiles > Groups and select the checkboxes for the training videos to make available for the role code. See the "Training" topic in the FireLight Platform User Guide for user instructions.

Wizards

  • CMR 191404 Updated styles for wizard components to vertically center all items inside of each row. This will affect all wizards for all organizations that do not have custom styles, which overwrite the align-items style. Formerly, Wizard components were globally set to align at the top of their rows instead of the center. This affected text fields and radio buttons. Radio buttons also had a large bottom margin set. These were changed to vertically center components, and radio buttons no longer have an extra margin added.

  • DR 192136 When the client clicked the URL in the email that was sent post-signing, FireLight displayed an error message that the signature was completed instead of directing the client to the application documents. Integration

  • DR 189884 Duplicate sequence number generated by BO application service resulted in lost transmission when the integration provider transmitted an App Sub to their SFTP endpoint. This was fixed by using an atomic increment of the counter.
  • DR 191251 SFTP fails to connect "Requested service is not implemented." The SFTP client failed to connect to newer servers using the more secure key exchange protocols and ciphers. This was fixed by extending the IFtpService with a new method that creates an IFtpClient using a FtpContext object. This object details everything currently needed to make an FTP or SFTP call. It also includes a property to set the UseLegacyGroupExchange value in the Rebex client. The property is nullable so it will not be set unless the caller sets it in the FtpContext. This will prevent changes to existing clients. Patched UAT 12/14/2018.

Localization

Postal Address Validation

Review Queue

  • DR 190178 Exceptions were occurring when loading the Review Queue. This was fixed.
  • DR 191416 Exceptions were occurring when loading the Review Queue. This was fixed.
  • DR 191707 Users could still add Reviewers through Preferences when the manual review process was locked down via a rule using the FLI_LOCK_REVIEWERS, FLI_REVIEWER_NAMES, and FLI_REVIEWER_EMAILS data items.   • DR 191913 Additional exceptions were occurring when loading the Review Queue when the user's group had firm ID access enabled, but didn't have a firm ID. This was fixed. Patched UAT 12/14/2018.

Rules

  • DR 190804 In certain circumstances, some data that was not in use was not being removed when an application was created or when it was locked. This was fixed.
  • DR 191531 Posting an empty string validation message within a rule no longer displayed as an error. This was fixed.

Security

  • DR 190224 External authentication vulnerability. The external authentication page was susceptible to a brute force attack because it did not block after X number of failed attempts to log in. This was fixed by setting a maximum number of attempts a user can try to access FireLight in one session. If that limit is exceeded the request is revoked and a custom audit message is appended to the application. The limit of allowed failed attempts is currently defaulted to 4 attempts.
  • DR 190225 Cross-site scripting vulnerability. The Summary dialog that is available by clicking an activity name within an activity or from the Other Actions menu > Summary option, allows the user to change the name of an activity. JavaScript could be entered into the name, which is executed within the browser. The likelihood of an exploit is low because the user would have to be tricked into entering malicious code within the activity name. This was fixed so JavaScript cannot be entered into the activity name.

Signatures

  • DR 190168 Touch signature (finger sign) did not work in chrome. This was fixed.
  • DR 190513 E-Signing with stylus on touch screen did not work on Surface Pro. This was fixed by fixing a touch issue with Chrome and Edge that affected stylus usage.
  • DR 191472 Received the generic error when moving to the Signatures page and clicking the "Use E-Sign" button. This was caused by an issue with data items on optional forms that were not included in an application package. This was fixed.

Wizards

  • DR 190439 Wizard was not showing changes on browser refresh. Have to start a new test case to see changes in the wizard. This was fixed.
  • DR 190440 Drop-down fields were not responding to required rules (not showing as red or required) in wizards. The issue was validation messages were not displaying on focus after a call to update the field. This was fixed.
  • DR 191393 The Enable Auto Save feature was not working. This was fixed. Implemented the Enable Auto Save feature in wizards.
  • DR 191797 A conditional wizard was showing when the conditions were not met. This was fixed. The system now checks if the page is visible/enabled and if not, gets the next visible page on load and reload. Patched UAT 12/7/2018.

2.13.0.137 October 26, 2018 Production Release

Enhancements

Admin

  • CMR 147676 Added autocomplete to common tags. Autocomplete provides context-aware suggestions based on data as it is typed in the "Name" field property in a form or wizard. For example, if you start to type "Owner," the field will begin to display the Owner common tags. If you continue to type "Owner_F," the autocomplete options will continue to narrow down to Owner_FirstName and other fields that contain "Owner_F." Hexure manages the list of common tags available for autocomplete. See Common Tags for a link to the CommonTags.xlsx file that you can download.
  • CMR 177313 On the Profiles > Organizations page > Activity tab, added an "Expiration Reminder Frequency" option that allows an organization to set an email frequency (Never, Daily, Weekly) for all agent and client expiration warning emails. Also added the <EXPIRY_DAYS> variable in all of the expiration warning email templates to convey the number of days until expiration.
  • CMR 179465 Added an automatic notification if an unlocked Wizard is in use by another user and has been saved by that user since they began their work. The notification asks the second user if they want to overwrite the first user's saved changes, discard their own changes, or cancel their own changes.
  • CMR 180424 Added new deployment scope options to deploy only the Product Configuration scope. To avoid confusion, we renamed the existing Product Configuration scope option to the All Configuration scope option that deploys data at the top level (Products, Packages, Sales Agreements, Rule Sets, Forms and Wizards). Please refer to the design document for a chart of the original deployment scope names with the replacement names where it applies, as well as the new options that have been added.
  • CMR 188103 Implemented a transaction-level application purge policy because the purge process now handles E-Delivery cases differently than other Activities. Since e-Delivery cases have an expiration we do not purge E-Delivery cases that have not expired. The Profiles > Organization page now has settings separated onto five tabs (Organization, Security, Activity, Signature, and Admin) instead of a single page. The Activity tab has a new Purge Completed E-Delivery setting with a default of 10 days. The default is much lower because E-Delivery cases use more space than other Activities and we need to limit the space used. The number of days allowable before purge of E-Delivery cases must be between 7 and 30 days.
  • CMR 188325 In the Wizard Designer, added the ability to add a grid structure (Data Grid) that is bound to a custom list. For example, Data Grids can be used for Fund Allocations, Contingent Beneficiary / Other Insured Lists, etc. See Data Grid for instructions.
  • CMR 189904 Added a new email template in Profiles > Emails for notifying the agent when an E-Delivery case is waiting for agent review. Patched UAT 10/23/2018.

CRM

  • CMR 179395 Added the ability to retrieve the email address from all integrated CRM systems.
  • CMR 181092 Added the ability for FireLight to look at the user's CRM Preferences to determine what CRM to call for pre-filling data into FireLight. Firms can use a single custom action button, e.g., "Lookup Client," to initiate the CRM web service call. This eliminates the need to have a custom action button per supported CRM in a firm's forms and wizards.

E-Delivery

  • CMR 146995 Added support for E-Delivery of client contracts. An E-Delivery request initiates with FireLight receiving an ACORD DocSend (TX121) message that drives the E-Delivery workflow. FireLight supports Agent and Client review workflows with the options to approve and request changes, and collect e-Signature. During the process, FireLight provides status messaging back to the carrier to keep track of the contract delivery. See the e-Delivery Guide for detailed setup information.

Embedded UI

  • CMR 183132 Added an advanced solution that allows firms to embed wizards created in FireLight Admin into any external user interface, for example, a firm's UI, the Salesforce UI, ForeSight®, an agent portal, a third-party tool for wealth management plannings, etc. Contact your FireLight PM for assistance.

FireLight Illustration Phase 2

  • CMR 183133 Added support to use the FireLight Illustration content in an API for outside sources to consume the Illustration and some navigation enhancements to the FireLight Illustration activity. Additional work is completed to ensure the Azure hosted site and ForeSight® Illustration updates (rates, state approvals, product approvals, etc.) are kept in sync. For testing, FireLight is integrating with our TestBench application to provide regression testing for FireLight illustrations. This ensures regression failures can be found sooner than relying on manual testing. Contact your FireLight PM for assistance.

FireLight User Interface

  • CMR 186973 Because e-Application navigation does not always apply to other activities, added a unique navigation toolbar to FireLight Illustration and E-Delivery activities. FireLight Illustration toolbar includes Alerts, Illustrate (Quick View), Print and Apply. A link icon appears in All Activities when an application is linked to the illustration. E-Delivery toolbar includes Alerts, Requests, Cancel E-Delivery, Request Re-Issue and Approve. An icon appears in All Activities when an e-Delivery is linked with other E-Delivery cases.

Preferences

  • CMR 179950 Added an Agent section with the following fields to the Preferences page: First Name, Last Name, CRD Number, Phone Number, Business Address (separate fields for Street, City, State, Zip Code), Firm Name, Firm Address (separate fields for Street, City, State, Zip Code). All new data items are mapped to common tags so the data will automatically transfer from Preferences to the forms and wizards. Also added mapping of the Email in Preferences to a global data item. Leverage the current logic in the Group option "Enable Override SSO" to allow the user to override the entries in the Preferences fields.

Product Types

  • CMR 188007 Added a new Registered Indexed Annuity product type.
  • DR 188443 Added a new Policy Change product type.

Reports

    • Number of Registered Users by Organization ID

    • Number of Active Users by Organization ID

Review Queue

  • CMR 180225 In FireLight Admin, added a new email template to notify the reviewer when an agent responds to a "More Info" request from the reviewer. The system sends a notification to the reviewer that locked the case if the case has a reviewer email set in Preferences and the system also sends a notification to the review queue group if the group has an email group set up in FireLight Admin. If the case has no reviewer email and no review queue email group is set up in Admin, the system sends nothing. The new email template can be customized to fit your needs.

  • CMR 188647 Added performance changes to the Review Queue by restructuring how the data is read and manipulated. This will significantly improve request times on larger post-submission review queue loads for clients who have hundreds or perhaps thousands of items in their review history. This affects most of the review queue: all queue lists, history, reviewer lists, searching, and advanced filtering.

Rules

  • CMR 178907 Added back to FireLight the ability to auto update rule sets when changing a field data item ID or Global Scope field property. In the Form and Wizard Designer, if the user changes the Name (data item ID) or Global Scope properties in a form or wizard control, a dialog prompts the user when there are associated rules. The user can choose to update to the new field name in all associated rule sets, update just the primary rule set, or not update any rule set. This does not update rule sets that are not included within the same packages as the form or wizard being updated.
  • CMR 180745 FireLight now removes data that is not in use when an activity is created from a copy and when it is locked. When “Copy As Is” or "Copy With Changes" is initiated, FireLight removes the original application's unused data from the new activity at the time it is created. Additionally, when an activity is locked any unused conditional data that may have been added then subsequently made inactive is removed. This prevents the extra data from interfering with rules and from being sent when the activity is submitted.

Signatures

  • CMR 180964 Enhanced the SMS Text Authentication and validation using a SMS text vendor, Twilio. This enhancement sends the unique FireLight passcode to the provided mobile number when SMS signature tags are active. The functionality applies across all client signature workflows (Sign Now, Remote Sign, and Client Fill and Sign). This feature provides one additional layer of validation because both the email and the phone would have to be compromised to forge the signature. Mobile Number verification was added to the Validate Identity Verification option in Organizations. See SMS Text Authentication for details.
  • CMR 181438 Added the signer role and name to the header of the signing method screen (Sign Now, Send Email Request) to more clearly indicate that the signing ceremony is initiated for the selected signer.
  • CMR 181439 Added the signer role and name to the header of the Sign Now signer Verification screen to more clearly indicate that the signing ceremony is initiated for the selected signer.
  • CMR 181440 In Remote Sign, renamed the View Application button to Review Documents button to remove confusion when clients complete signing.
  • CMR 181441 In Remote Sign, reordered the buttons so that the Sign Application button is first, followed by the Review Documents button to avoid confusion when clients want to sign the application.
  • CMR 181442 To remove confusion in the signing ceremony, FireLight now automatically moves to the next form when the user selects the acknowledgement checkbox for the current form.
  • CMR 181443 To remove confusion during the signing process, the Unlock Application option is now included on the List of Required Signers screen in addition to the Other Actions menu.
  • CMR 185345 Added Reviewer and Principal Reviewer signer types to signature fields.

Templates

  • CMR 181596 Added support for custom action upload and print buttons on a template form. FireLight validates whether the template is uploaded before the user data entry can reach 100%. The template appears in the Open list to provide instructions. The template form can include the Custom Action Upload and Custom Action Print in Button controls to provide clear instructions with easy links to complete the template requirement.

Wizards

  • CMR 183237 Added template support to wizards.

Fixed Defects

Admin

  • DR 182381 Delete Rule tooltip did not disappear after deleting a rule. Removed "Delete Rule" tooltip. In the future, we may implement a tooltip component of our own that will work, but that is a later enhancement.
  • DR 182565 There was a performance issue with saving large rule sets. Improved efficiency of save call on the rule set dialog to mitigate problems with save operations taking a long time on rule sets with a large number of rules.
  • DR 182687 Client name was not being replaced in expiration email templates in the subject line. Added the data for the agent name and the client name to the subject line replacement values to allow the email tag to be used in the subject line in addition to the email body.
  • DR 182802 Exclude from back office(print only) forms were being submitted electronically. This was resolved by changes for v2.13.
  • DR 183269 Enable Submit Email for Agent group option was not working. This was fixed by accounting for agent's preference email when determining who to send final emails after final review queue approval.
  • DR 183997 If not already set, disabled the Will Be Replaced By option in forms within the Form Library because it added unnecessary complexity to the system. It should eventually be removed. It is better to set the form expiration date in This Form Expires On field. Then set the replacement form beginning date in This Form Begins On field and add the form to the applicable packages.
  • DR 185116 In the Wizard Designer, when you click Edit Item Source in the field properties for a Dropdown List field, added a scroll bar to the Item Source dialog when the data entry exceeds the capacity of the Item Source dialog.
  • DR 187214 An error message was triggering when the signed State did not match the agent State. In agent signature controls, the Enforce State Match option was automatically enabling after being disabled, which was causing issues when the agent State did not match the owner and insured State during the signing ceremony. This was fixed.
  • DR 187421 There were issues with wizard handling of add/remove packages. This was fixed by updating the wizard implementation to save current forms plus wizards.
  • DR 188095 Keep FLI_DECLINED_ESIGN data item and app DeclinedESignature flag in sync. When setting the FLI_DECLINED_ESIGN data item to blank, the application DeclineESignature flag was not getting turned off. This change just ensures that the data item and app property always remain in sync.
