  • DR 245029 FireLight Access Users Firm ID Issue. Client Impact: Medium - With 2.20 we introduced a feature for FireLight Access that allowed a user to be part of multiple Firms. The migration process introduced a bug that did not take into account that a user can be part of multiple organizations. This caused data to be corrupt and has been resolved. Problem: The conversion of user firm ID (from being stored in the 1228 to being stored as a column in the database, a change needed to support multiple firm IDs for a user -- iConnect 233321) was not correctly handling users that exist in multiple organizations. The user's firm ID was being updated (for all organizations) to the firm ID that user had in the last processed organization instance. Resolution: Fixed the conversion of the firm ID to be organization-specific. Also corrected the logic to remove multiple instances of a user's firm ID to be specific to the current organization (this was also global and could result in the user's records in other organizations being deleted when removing multiple instances of a firm ID for a user).

    Areas Impacted: Access conversion of user firm IDs and deleting of multiple instances of user firmIDs

2.20.192 April 9, 2021 Production Release

Known Issue

This affects the combined wizard + forms UI, when a wizard (actually any wizard within the activity) has multiple fields on different panels or pages with the same data item (i.e., MY_DATA_ITEM). If one instance is in a chain of descendant controls where a parent field is disabled via a rule, with the implementation of iConnect 240231, this will now disable all children of that parent, including the one with data item MY_DATA_ITEM, and this will also disable any other instances of field with data item MY_DATA_ITEM on the wizard (because it’s the data item itself that becomes disabled), but not its descendant controls / data items in the other instances. When this occurs, such descendant fields will still be enabled and could have validation messages, but those messages will not count toward % complete or be shown in the error list. The workaround is to not use the same data item for multiple fields in different sections of any wizard within the activity.

E-Signature - DocuSign (Work in Progress)

  • CMR 237379 TFS17306 Julie (Carriers and Distributors) Support the use of DocuSign within FireLight. We are still working on document retrieval from DocuSign for this enhancement. With this enhancement, FireLight is integrating with DocuSign to allow the option for our clients to use the DocuSign services for their e-signature captures. This is an available option for our Remote Signature only for this release.

    Assumptions to consider with this project:

    • DocuSign must be honored anytime it is enabled regardless of the Org initiating the application. Meaning, Carriers enable DocuSign and deploy the app to the distribution. FireLight must honor DocuSign for the distribution as well as honor it in the carrier Org.

    • DocuSign will support Remote Sign Only.

    • FireLight is handing off the signing process for remote sign when DocuSign is enabled. Any Email notification, Signing and audit is completed within the DocuSign environment.

    • When DocuSign is enabled, the Signature Audit will not be captured within the FireLight Signing audit, however FireLight is receiving a signature audit back from DocuSign that is attached to the application and sent within the client's payload.

    • FireLight will need the signed documents with the digital signature applied and the DocuSign signature audit returned 100% complete to FireLight. The clients that need DocuSign implemented have the DocuSign digital signature, tamper evidence and audit approved.

    • FireLight must be able to retain the digital signature without breaking the hash for the DocuSign documents.

    • FireLight must be able to send documents completed by DocuSign to the different firms (Distributor receive X&Y forms and Carrier receives A, B, and C).

    • FL will need a bi-directional ability to cancel the eSignature (A user can cancel the signature from FireLight and from DocuSign).

    • The User MUST have authentication when sending PII information to DocuSign, so FireLight will still send the last 4 digits of the SSN and DOB of each party to DocuSign.

    Impacts:

    • FireLight E-Signatures - Specifically the "Send Email Request" or Remote Signing capability

    • Organization Settings - Enabling the DocuSign option

    • Third-Party Configuration Table - service type for DocuSign, new column for sub firm "Control Group"

    • App Changes - New UI page for "DocuSign Signing Ceremony" information that can be entered and initiate the call to DocuSign

    • Reports - the Third-Party configuration Report is altered to show the DocuSign signed applications and add the Control Group

    • Audits - New audits is created to capture the calls to DocuSign, any cancellations, and the completion of the signature process

    • Display Statuses - New display statuses will show when the signature process is in process, cancelled, or completed through DocuSign

    • Transfer - Transfer is disabled when DocuSign is used, even if it is selected within the groups settings. Share will not be impacted

    • Client Fill and Sign - Client Fill and Sign is disabled when DocuSign is used, but will also notify the user of this through a tool tip

FireLight and Swiss Re Magnum Integration

  • CMR 224601 H 14780 Cynthia (Carriers and Distributors) Enhance FireLight to leverage the Magnum Pure Engine APIs within a FireLight application to gather underwriting data for life applications.

    The Magnum Pure Engine APIs is embedded in FireLight and shown in a dialog when opened by clicking on a Custom Action button added to a wizard page in the application.

    When Magnum is opened, data is passed in the bootstrap process from the FireLight application to set up the correct interview questions in Magnum.

    When Magnum is closed, the data is saved and passed to FireLight, which is mapped to the appropriate form fields so the data is submitted with the application.

    FireLight will query the Magnum API for a status. Once the data is complete in the Magnum interview the status is set to complete. FireLight rules is able to leverage the status to recognize when the application can be set to 100% complete when data entry is finished. After the Magnum interview is complete, it can be reopened and updated prior to application submittal.

    FireLight will also retrieve and store the case level underwriting decision when the interview is complete. Additionally, the full underwriting decision tree is stored in JSON format and made available to the provider for added customizations.

    Note that the Magnum setup is handled by the carrier, not the distributor. When a distributor runs a carrier product activity, the carrier custom action button, custom lists and provider are called after the relevant distributor components.

    Impacts:
