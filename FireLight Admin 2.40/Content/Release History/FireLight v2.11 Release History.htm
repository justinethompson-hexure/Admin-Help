2.11.0.476 April 13, 2018 Minor Production Release

Enhancements

Disability Income Product Types

  • CMR 176638 Added the Disability Income Conversion and Disability Income Increase product types to FireLight.

Email – Sub Firm Branding

  • CMR 180598 Added the ability to brand the email templates for sub firms on the Email page. To add branding for sub firm emails, contact your Strategic Account Manager to have the FireLight team create the list of Branding Firms for you. Once that list is created, you can manage the sub firm emails on your own by selecting an email template and then selecting a sub firm in the Branding Firm list on the Email page. NOTE: Organizations that currently have Branding Firms have had their Branding Firm list created in their environments.

Global Data Items

  • CMR 173296 When the organization setting for Allow E-Approval (manual PreSubmit review) was selected and the FLI_REQUIREMANUALREVIEW global data item was set to "false" and the user clicked the Continue button, the user was still seeing the sub menu with the "Request Review" and "Submit" options. This was updated. Now when the organization setting for Allow E-Approval is selected and the FLI_REQUIREMANUALREVIEW data item is set to "false," the system will not show the Review step or the sub menu with the "Request Review" and "Submit" options. When the organization setting for Allow E-Approval is selected and FLI_REQUIREMANUALREVIEW is set to "true," FireLight takes the user directly to the PreSubmit review process and does not show the sub menu with the "Request Review" and "Submit" options.

Localization

  • [CMR 148022 ] Began adding components in v2.11 to improve support for multiple locales. In FireLight Admin, added: Localization dialog on the Organizations page, Product Locale list on the Products page, Locale list on the Email page. Signer Types

Client Fill and Client Fill & Sign Requests

  • CMR 173078 For more clarity, changed the wording on the buttons in the Complete Application dialog when a client clicks 'Complete/Log Off' during a Client Fill or a Client Fill & Sign request. Custom Print

  • CMR 162883 An ALZ work order added a custom action print button override, overriding the organization level Allow NIGO Printing and Print Submitted Only print settings. This allows users to click a custom action print button to print forms bundled by tc value or forms selected in the Select Documents to Print dialog regardless of the organization level print settings. In FireLight Admin on the Groups page App tab, select the new 'Enable Custom Print' checkbox to enable the feature for each applicable group. A new FLI_ENABLE_CUSTOM_PRINT global dataitem holds the value of the Enable Custom Print group setting, which allows form rules to set the visibility of the custom action print button based on the group setting of the application owner. For a sample rule, see the "FLI_ENABLE_CUSTOM_PRINT" data item in the "Common Tags" topic in Admin HelpCommon Tags spreadsheet.

Designer

  • CMR 152962 Added Social Security Number masking options via a new field property that marks a field as a Social Security Number (SSN) and allows the input to have a mask applied during form entry that hides the original value. The mask is applied on the server, which secures the value by hiding the original value from the browser. In FireLight Admin in the Text Box control, select an option in the 'SSN Format' field property drop-down list to indicate the field is a SSN and set the level of masking: None (not a SSN field), Visible (123-45-6789), Partial (XXX-XX-6789), or Full (XXX-XX-XXXX). The default value is None, which indicates the field is not a SSN field. The Mask and Is Password field properties cannot be used when the SSN Format option is other than None.
  • CMR 159834 In FireLight Admin, added the ability to create your own reflexive entry pages, screens and questions that provide a fresh consumer facing workflow in addition to the existing FireLight form based solution. Clients interested in this solution should contact their Strategic Account Manager.
  • CMR 165962 An ALZ work order added the ability to allow radio buttons to appear as circles or squares. In FireLight Admin in the Radio Button control field properties, select Round or Square in the 'Radio Button Styling' list. When a user selects a radio button on the form, Round styling fills with a dot and Square styling fills with a check mark.

  • CMR 166429 An ALZ work order added the Roboto, Roboto Bold, Montserrat Light, and Montserrat Regular fonts to the Field Properties dialog in the Designer.

E-Signature

  • CMR 137217 Added the ability to control whether the agent signs first or last in the signing ceremony. In FireLight Admin on the Organizations page, added an 'Agent Signing Order' dropdown list with three options: Default (agent can choose whether to sign before or after client signers), First (agent must sign first, client signer buttons are disabled until after agent signs), and Last (agent must sign last, agent signer button is disabled until after all clients sign). In a distributor environment, the signing order will follow the signing order setting for the agent's organization.
  • CMR 147377 Added the ability to disable the Sign Now and Sign Later processes via rules. In FireLight Admin, added a check to see if the current application requires e-signature and added two new global dataitems FLI_DISABLE_SIGN_NOW and FLI_DISABLE_SIGN_LATER. Add radio buttons to a form with the FLI_DISABLE_SIGN_NOW and FLI_DISABLE_SIGN_LATER tag names. When the FLI_DISABLE_SIGN_NOW radio button Export Value is set to "true," the E-Signature process will disable the Sign Now process and send the signature request email. When FLI_DISABLE_SIGN_LATER is set to true and FLI_DISABLE_SIGN_NOW is set to false, the E-Signature process will go to the Sign Now process. The global dataitems can be hidden on the form and used in rules.
  • CMR 165897 An ALZ work order added the ability to disable the Send Email Request option in the e-signature process, which bypasses the Client Signature Choice page, and users go directly from the List of Required Signers page to the Client Identification Verification page. E-signature capture can only be done through the Sign Now process. In FireLight Admin on the Groups page App tab, select the 'Disable Send Email Request' checkbox to enable this feature by group role code.

Fees, Expenses and Commissions Repository - Fee Inquiry Transaction

  • CMR 161212, CMR 161212-2(Available to all clients that have licensed the repository) A Forethought work order added a 'Fee Inquiry' transaction type and related package type to FireLight to support querying fees from the Fees and Expenses Repository outside of an application. The Fee Inquiry transaction type allows users with access to the new transaction type to query the selected product fee values and display them in a dynamic fee worksheet. This transaction type is available via SSO and through manual selection of the Fee Inquiry transaction. The fee inquiry transaction is a one-time view into the fees for the selected product. The Fee Inquiry is not saved and will not be available in Recent History or All Applications. The only actions available for this inquiry is print/download. There are no other actions available including save, signatures, review, send to client, or submit. The fees can now be pulled based on the order of the data in the PPfA and do not require a static placeholder in the PDF. Only the fees that are in the PPfA for the selected product is displayed which creates a more user friendly fee display. To do this, the enhancement added a fee attribute called number (number="1" for the first fee, "2" for the second fee, etc.). The Fees and Expense License or DOL License is required to support the new transaction. If you are already licensed for the repository, there is no additional license and there are no transaction fees to use the inquiry. If you are not licensed, please contact your Strategic Account Manager for an addendum to your contract to support the repository. See "Fee Inquiry Transaction" in "Features" in Admin HelpFeatures for implementation, rules information, and sample rules.

Review Queue

  • CMR 147473 Merged the Manual Review process into the Application Submission (OSJ) Review Queue functionality in order to bring the features of the application submission review queue to the Manual Review. This process adds new features to the Manual Review process and facilitates maintenance of the combined sub-systems. Rule Sets

  • CMR 144137/159836 Enhanced UAT to support saving projects and deploying projects independently. Added the ability to group rules into rule sets and add the applicable rule sets to packages, which allows maximum reuse of the same rules across multiple packages. Rule sets provide beginning and expiration dates to assist with release planning. Rules can be grouped into categories (distributor-specific rule set, owner rule set, product-specific rule set, etc.) so multiple users can each work on a different rule set at the same time and lock their rule set for editing to prevent overwriting. Rule sets can be deployed independently of forms. With the implementation of rule sets, FireLight converts all of your previous form level rules to rule sets with the rule set names matching the form names. In FireLight Admin, added the Rule Sets page in Profiles where users can do most of the rules work. FireLight continues to support the Rules Window in form Designer to manage rules associated with a selected control. In Admin Help, see the Rule Sets page in Profiles for detailed information. NOTE: If you exported forms that had rules in FireLight v2.10 and need to import those forms back into FireLight, be sure to upload those forms with the Use exported definitions checkbox selected before the move to FireLight v2.11 production. After the production release on October 27, 2017, rules in exported forms will not be included in the upload forms process.
  • CMR 152609 In FireLight Admin, added the ability to search for rules in all packages and rule sets on the Rule Sets page. This allows users to search for rules in rule sets across all packages.

Signature Security

  • CMR 164629 Implemented a security enhancement for signing all generated PDFs using the more secure SHA2 encryption instead of SHA1 encryption for signatures. If your organization requires SHA1 encryption for signatures, select the Use SHA1 for Signatures checkbox on the Organizations page in Profiles to continue using SHA1 encryption.
