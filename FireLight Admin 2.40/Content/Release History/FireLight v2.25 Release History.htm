2.25.137 September 23 UAT and September 30, 2022 Production Patch Release

Fixed Defects

Console

  • DR 268543 Console Checkin fails to generate PDFs. Problem: Console check-in prevents apps from locking, but still continues check-in process on failure. Resolution: If there is an exception when locking and unlocking this should prevent check in. Also if checking in app, we should assume app is not checked out when locking.
    Areas Impacted: Console check-in. Locking/unlocking apps

  • DR 269691 Console cases - display Signing Ceremony Only forms during data entry. Problem: When App was Initialize in the console, it was not taking into account the Form Duty Type. Resolution: Fixed Console Initialize Process.
    Areas Impacted: Console Forms

DTCC

  • DR 269875 DTCC 99 items question/answer limit bug. Problem: There is a 99 items limit in the question/answers records, which needs to be fixed. Resolution: Loosened our validation requirements.
    Areas Impacted: Apps going to the DTCC

E-Delivery

  • DR 268446 eDelivery Cases Coming back without Signatures on PDFs (2 Cases). Problem: Signature not included on PDFs for e-delivery cases in some instances. Resolution: If the agent signing is started and then exited, the wizard view changes to forms only and the continue button has an option for submit. This will skip the agent signing process and begin the submission process. Both cases were missing the agent party created when agent signing begins and were missing the agent signature. As child case did not contain completed agent signatures, it could not be submitted along with the parent case and edelivery case would not contain the signatures. Fixed the return to form only view- e-delivery must always remain in wizard view to retain correct navigation options.
    Areas Impacted: E-delivery after lock view and continue button options

PDFs

  • DR 268824 GetActivityPDF API returns documents not in package at lockdown. Problem: getactivitypdf and Print PDF pull latest forms in order to get optional forms. Resolution: If Locked, Don't get any required forms that are not filled out before lockdown.
    Areas Impacted: getactivitypdf API and PrintPDF

Wizards and Forms

  • DR 262482 Illustration glitch when changing withdrawal or subsequent payments “Start Type” to age & other value not saving. Problem: Updating Schedule fields could cause the system to enter a constant update loop that caused erratic behavior and could prevent further schedule field updates. Resolution: Fixed. Schedules apply their update only once.
    Areas Impacted: Wizards

  • DR 266499 Update field response contains masked values in DataItem.OriginalValue. Another issue was found with the current solution. The issue was found in an e-Delivery case that uses a rule that checks Original Value. Problem: The DataItem.OriginalValue was not getting saved to the Database because of the change to not serialize it. Resolution: The Original Value is only serialized when the value is not null AND the model is not getting passed to the UI. Otherwise, the Original Value gets serialized as usual and gets saved to the DB.
    Areas Impacted: Clients using DataItem.OriginalValue for rules logic

  • DR 269282 Wizard - Fields being cleared and disabled, still showing validation messages after data entry. Problem: Under some situations, Wizards and Forms in Wizards would lose data entry when navigating between pages. Resolution: Fixed.
    Areas Impacted: Wizards, Forms in Wizards

2.25.97 August 5 UAT and August 10, 2022 Production Patch Release

Activities

  • DR 266921 Activity FLI_FIRM_ID data item does not have a value on some cases. Problem: For backward compatibility, the activity FLI_FIRM_ID data item, and also the activity FirmID and the user's FirmID, should always be set to the FirmRegistrationNum from the user's 1228, in addition to setting the FLI_CASE_ACCESS data item (and the user's and activity's CaseAccess properties), when the user's group does not have a Firm Org. This was not always happening. Resolution: Corrected all logic paths for updating of user information and creating of activities to properly set the FLI_FIRM_ID data item and FirmID property to the FirmRegistrationNum in the user's 1228 when the user's group does not have a Firm Org.
    Areas Impacted: Assignment of FLI_FIRMID data item during activity creation

Wizards and Forms

  • DR 266167 Wizard - Manage Optional forms dialog is not scrollable. Problem: Window would not scroll when 10 or more forms were available, hiding multiple options off-screen. Resolution: Changed from overflow: none, to overflow: auto, with some added CSS to hide scroll bar.
    Areas Impacted: Other Actions, Optional Forms, when 10 or more are available.

2.25.84 July 22 UAT and July 26, 2022 Production Patch Release

Fixed Defects

Wizards and Forms

  • DR 266424 Wizard - Date Slash Field Issue. Problem: Fields with a date mask containing slashes displayed "D_te Sl_sh" instead of a gray blank area like usual. The field behaved correctly once it was enabled. Resolution: This issue is fixed by iConnect 266448, which should correct all date fields and all other fields that have a mask.

    Areas Impacted: Wizard date fields containing slashes

  • DR 266448 Wizard - Error locating & rendering field mask during data entry. Problem: Disabled or read-only DatePicker fields with a mask were not implementing the mask. Resolution: Fixed the code to correctly lookup and enforce the field mask.

    Areas Impacted: Disabled/readonly DatePicker fields with a mask in the combined wizard+forms UI

  • DR 266455 Wizard - Distributor mask is overriding carrier mask with the same name. 1) Problem: With the changes to masking in the 2.25 release, when the carrier and distributor define a mask with the same name but different format, the system was prioritizing the distributor's definition of the mask. Resolution: Utilize the mask defined by the form/wizard of the owning organization. 2) Problem: Testing did not pass in QE. The correct org masks were being returned to the front-end with each page change, but the masks in the redux state were not being updated from returned model, so it was still using the wrong masks when changing from a distributor to carrier form. Resolution: Added the missing logic to update the masks in the redux state.

    Areas Impacted: Field masks in the combined wizard+forms UI, in a distributor environment where carrier & distributor define a mask with the same name and different value

  • DR 266499 Update field response contains masked values in DataItem.OriginalValue. Problem: In Wizards for fields with a Mask, the response from the server on UpdateField calls included the un-masked value in the Data.OriginalValue property. When these fields are masked the raw data should not be exposed in the response. This is a security risk. Resolution: We removed the serialization of the offending properties (OriginalValue). These properties are not needed in the browser and remove the security risk as well as slightly reduce the payload size.

    Areas Impacted: Clients using Masked inputs in Wizards would be affected

Fixed Defects

Activities

  • DR 260014 Copy activity logic and data item properties. Problem: The latest fix was found to cause a side effect of no longer locking data items for a next activity when Lock Transferred Data Items is enabled (TFS 33675). Resolution: Moved the logic into the correct location and do not reset data item properties if it is a next activity with Lock Transferred Data Items. Problem: The original fixes for this were incomplete. 1) The changes to reset Enabled / Visible / ReadOnly data item properties on a copy with changes were not having an effect because it was being done before the provider had been called to copy the original app data items. 2) The changes to reset Enabled / Visible properties when opening an activity for data entry corrected the issue for the combined wizard UI, but introduced the original problem in legacy forms UI (which worked correctly before this fix). Resolution: 1) Corrected the copy with changes logic to reset the data item properties after the provider copies the app data. 2) Modified the logic to reset properties when loading an activity to operate on the original data collection, rather than a clone of the data items, so they will not be incorrectly synced back to the rules collection in the legacy forms UI (the common tag sync logic is specific to data item values and doesn't handle other properties consistently).

    Areas Impacted: Read-only state of data items on copy to next activity. Data item enabled/visible properties initialization on copy with changes and prior to rules execution

  • DR 260682 CRM - Remove popup that generates when Clear is selected for Product Search input fields. Problem: This is occurring not because a provider-base action isn't assigned. The script code specifically checks for any button in the group being disabled, and if so, displays the Disabled message when clicking one of the buttons. Since the SEARCH button is disabled, it results in the Disabled message being displayed when clicking on CLEAR. Resolution: Removed the code to display the Disabled popup.

    Areas Impacted: CRM

  • DR 262149 Rule that updates data item on 100% complete is not always working. Problem: The FLI_PERCENT_COMPLETE data item was not refreshing correctly because the rules updates were not synced back correctly. Resolution:

    1. Moved the updating and processing of the FLI_PERCENT_COMPLETE data item to be after the initial rules run, and after the Model.PageMap and the Model.ShowValidationMap got refreshed.

    2. Corrected the refreshing of the Model.PageMap to check for both user/UI changes and rules changes. Also moved the refreshing to be before reading its values.

    3. Also handled when the FLI_PERCENT_COMPLETE change is the only change.

    Areas Impacted: Refreshing of the FLI_PERCENT_COMPLETE after user changes

  • DR 263500 Continue popup message showing for activity that has no Continue step. Problem: The data entry complete popup (for which the default text is "Data Entry has met the requirements. You may enter more data or CONTINUE to proceed to the next step.") is always displayed for an activity when it reaches 100%, and there is no way to suppress the message, but it is contradictory and confusing to display this message if there is no continue action possible for the activity (the reported example is Print Forms activity, which cannot be saved and does not have a continue action). Resolution: Modified the logic to not display the data entry complete popup if continue is not an available action (this includes activities that can't be saved, such as Print Forms, and also if Disable Save Applications is enabled in group options). NOTE: This fix is specific to activities in data entry status (for instance, excludes pending review, activity finished steps) and also excludes some activity types (fee inquiry and needs determination), which have their own popup messages, and these popups will not be suppressed even if continue is unavailable. Some of these popups have default text that mentions Continue. It would require more research to determine if and when it might be appropriate to suppress popup messages for other statuses and excluded activity types. It may also be beneficial to allow/support empty messages to be configured for specific popups in Activity profile to suppress those popups.

    Areas Impacted: Data entry complete popup at 100% complete

  • DR 263663 Error Message covering field. Problem: A legacy forms validation warning was blocking the input field. Resolution: Due to the design of the form designer, validation messages are positioned using the absolute position of the input on the page, plus an offset to make the message higher. This offset is designed for single-line messages. The warning message in this case was multi line message. I styled validation warning messages to force them to fit on a single line. This also matches the current styling for validation warnings, which are also forced to fit on one line.

    Areas Impacted: Applications using legacy forms that contain validation warning messages

  • DR 264953 No values in the residence dropdown while creating new application. Problem: The changes to limit jurisdictions for products at the group level was not correctly handling groups that did not have jurisdiction limitations, resulting in no jurisdictions being listed in some situations. Resolution: Corrected the code to only limit jurisdictions when limitations are defined at the group level.
    Areas Impacted: Resident jurisdiction list and product availability lists

DTCC

  • DR 265362 Modify DTCC DTI sequence numbers to avoid duplicates in test environments. Problem: DTCC has one test region. FireLight has 3 test environments (UAT, UATNext, & Staging). Clients needs to be able to submit apps from any or all of the 3 test environments to the DTCC test region. This can result in duplicate DTI rejects by DTCC if the same DTI or sequence number is used for activities from multiple test environments. Resolution: To reduce the likelihood of a duplicate, modify the base system to use different prefixes for DTI_Number & DTCC_Number counters returned from the appService.GetNextIncrementalNumber and BackOfficeProviderBase.GetSequenceNumber methods: in UATNext, the first digit of the number is replaced by the digit "5" and in UAT, the first digit of the returned number is replaced by the digit "9." NOTE: This could still result in duplicates if the current counter actually does start with 5 or 9, but it will reduce the possibility.

    Areas Impacted: DTCC

  • DR 265488 DTCC AppSub premium method value changed. Problem: There was a problem with the implementation of 263658, which had the result of altering the app sub values for some premium method / premium type / account type values for existing providers. Because the digital payment enumeration was not added to the end of the list, existing values following it were all shifted up to the wrong value, when converting from the enumeration to the app sub value. Resolution: Corrected this issue so that existing provider value mapping will not be affected. NOTE: It is necessary for the provider to be recompiled with the latest definitions after this is incorporated into the release.

    Areas Impacted: DTCC app sub generation

Email

  • DR 262507 Email Queue not Catching SmtpException Types. During times of email failure and queue build-up, system was not catching SmtpException (which is the type of exception occuring), causing scheduler to loop through thousands of failing emails. Resolution: Added SmtpException to break-out logic.

    Areas Impacted: Email Queue system (all emails)

  • DR 262763 Some email notifications are not recorded in the activity audits. Problem: Didn't have ESignLaterApproved/Declined audit. Resolution: Added audit. also added logic to trace possible un audited emails, to catch future needed audits.

    Areas Impacted: Sending Emails in email manager.

  • DR 263659 Email Group and Failed Email Notification fields no longer allowing multiple email addresses. Problem: Previously, email validation in FireLight did accept multiple email addresses separated by a comma or semicolon character (in addition to invalid email addresses like abc@xyz@123.com). That ability was removed with the fixes for iConnects 260693, 262060, and 261044. The system now a) requires a single email address when validating email fields, and b) validates the email address at the point of sending, throwing an error instead of sending a message to a To address that is multiple addresses. Resolution: Modified email validation to accept multiple email addresses separated by semicolon or comma characters for the following fields: Organization -> Failed Email Notifications, Review Queue -> Email Group, and Sales Agreement -> Deploy Email Group. Intentionally, the system will not allow multiple email addresses to be entered in most email fields (particularly not for fields that could be a From address, but also for cases where there is a To name also entered, and for legacy reasons this is not being enabled across the board).

    Areas Impacted: Entry of email addresses in the above fields

  • DR 264862 Problem: Submission reminder emails were being sent even if there was no agent email address to send them to. Resolution: Check for blank agent email address and do not send reminder in that case. Also removed duplicate logic to retrieve agent email address, skip e-delivery-specific logic for non-edelivery reminders, consistently use agent email and name in reminder messages, and store appId when sending reminder for a single activity.

    Areas Impacted: Submission reminder notifications

  • DR 254472 Rulesets do not appear to run until saved.
