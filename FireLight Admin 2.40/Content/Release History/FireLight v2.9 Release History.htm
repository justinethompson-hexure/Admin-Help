FireLight v2.9 Release History

NOTE: If a design document is associated with an enhancement, click the link in the iConnect number to open the design document.

2.9.0.xxx March 17, 2017 Production Release

Enhancements

Barcode Enhancement

  • CMR 144590 Barcode Enhancements - Make barcode available in datamatrix format, as well as able to include on PDF  – In FireLight Admin, added a Barcode Type field to the field properties for Barcode controls with options of QR Code or Datamatrix. Datamatrix type is always visible in the application during all phases of the application process (data entry, e-sign, submission) and it is visible online and in the pdf/printed version.

Fees and Expenses Repository

  • CMR 150568 DOL Fees and Expense repository  – In FireLight, added a repository to house the fees, expenses, and commissions data for clients. FireLight will receive the data for the repository from the DTCC and/or direct from the carrier. Distributors using data from the DTCC must be licensed to use the DTCC data. In addition, carriers can upload the files directly to FireLight. There is a licensing component for both the carriers and distributors to support this feature. Carrier clients interested in participating should contact their Strategic Account Manager for Work Order pricing. Distributors will need to work with their Strategic Account Manager for pricing and submit their fees and expense disclosures along with a work order to tag the disclosure document to their PM and Product Management as soon as possible . Carriers is required to tag their applications with data items for each Rider that will have a fee and set the export value to match the PPfA.

Groups

  • CMR 153453 Define a default external role code if none specified so that permissions can be restricted.  – Changed base to assign an external role code of DEFAULT if none is provided. Then, an organization could create a group with that TC code and configure its rights for users that login without an external role code. If no such group code is defined, it will operate the same as it does today with no group options checked, maintaining backward compatibility.

Needs Determination

  • CMR 147396 DOL New Transaction Type for Needs Determination  – Added a new Needs Determination transaction type for collecting the client data needed to make a Best Interest Recommendation. This new transaction sits in front of the application and supports launching from the Needs Determination transaction to the application. The transaction has its own audit report, output, and reporting. To support this feature, there is a licensing component for the distributors and for forms tagging. Distributors will need to work with the VP of Distribution sales, Steven Acosta, for pricing. Determine the types of Needs Determination workflows for your organization and submit a Work Order with your Strategic Account Manager to update your SSO, applicable database tables and rules for your organization and for any custom Upload button work, etc.
  • CMR 147397 DOL Filter capabilities from the Needs Determination to the Application – In FireLight Admin, added a Needs Filter tab on the Groups page in Profiles. Distribution enters product category filter criteria with the dataitems from the needs questionnaire that is used to filter the available products to select in the application. FireLight uses the filter criteria defined for a product to narrow down the available product options. For example: Accumulation, Income, Protection, Death Benefit, and Protection of Capital can be set as product categories on products available to BD1. When the Needs Determination or a pre-filled SSO identifies one of the defined categories, the products listed for selection will only show the products that meet that criteria. Distributor licensing is required to support this feature. Distributors need to work with the VP of Distribution sales, Steven Acosta, for pricing. Additionally, each client needs to submit a Work Order to define their specific filter criteria for the products on their shelf. This needs to be done in conjunction with your Needs Determination questionnaire or SSO.

Organizations

  • CMR 155988 SAML SSO - support multiple IDP Identifiers per organization.  – Added support for multiple IDP identifiers separated by commas (no spaces) in the SAML SSO, for example: IDP_Identifier1,IDP_Identifier2,IDP_Identifier3. Add the IDP Identifier(s) on the Organizations page in Profiles.

Products

  • CMR 153833 Add Group Annuity to Product Type  – In FireLight Admin, Profiles, added Group Annuity to the Product Type field on the Products page.
  • CMR 135771 Organization name for products  In FireLight Admin on the Products page, added a new Product Organization Name field to add the selling carrier organization name for the product. FireLight users will see the selling carrier organization name for each product when creating a new application. If the Product Organization Name field is left blank, it will default to the Organization Name on the Organizations page in Admin.

  • DR 152790 Unable to keep Assigned Package in desired order. This has been fixed. Distributor packages can now be placed before and after carrier packages.
  • DR 152846 Empty initial is allowed. This has been fixed. FireLight no longer accepts white space for an initial.
  • DR 153685 Continue button disabled after eSign Decline. This has been fixed.
  • DR 155044, DR 155297 Sending Passcode to cell phone. An error occurred when sending a Passcode to a client via SMS text message. This has been fixed.
  • DR 155108 Addendum/Soft Limit Functionality not working correctly. This has been fixed. Made changes to wrap text rather than move it.
  • DR 155287 CT Integration Mapping to form not working. This has been fixed.
  • DR 155358 Template forms when combined with Link allowing a user to download their completed apps. After submitting an application, Template forms were also being included in the forms to download list. This has been fixed.
  • DR 155868 Fees and Expense Repository not overwriting old PPfA. This has been fixed.
  • DR 156142 Repository: 2 RiderTypeCodes with the same number but different FeatureOptProduct.ProductCode options is not working. This has been fixed.
  • DR 156149 CT Integration - Signature detail not pulling back into the 1035 form thus signature tags not working. This has been fixed.
  • DR 156216 ATH application will not check out in ALM console. This has been fixed.
  • DR 156920 Surrender Charge pulling back zeros instead of fees. This has been fixed.

2.9.0.x December 30, 2016 Minor Production Release

Enhancements

Organizations - Purge Policy Settings

  • CMR 149903 Validate and limit purge policy settings – In FireLight Admin on the Organizations page in Profiles, the number of days allowable before purge of uncompleted applications must be between 30 and 180 days and the number of days allowable before purge of completed applications must be between 10 and 180 days.

Signature Controls

  • CMR 147003 Added the ability to upload invisible labels to create new signature controls. – In FireLight, users can now upload documents with invisible metadata to create new signature controls. The signature controls on the template form in FireLight Admin is replaced by the signature controls of the uploaded document. See CMR 147003 in iConnect for additional documentation.

Fixed Defects

  • DR 126897 Update user interface to indicate that Review Queue script is an organization setting. In FireLight Admin on the Review Queue page in Profiles, added "NOTE: The review queue script is global to all queues and is deployed using Organization Configuration." below the Edit Review Queue Script link.
  • DR 147711 Node Help: Get Set/getcustomlist – In FireLight Admin, the node help for <getcustomlist> had a bad example. This has been fixed.

  • DR 149563 Application not visible in manual review step (approve application) – In FireLight, when the reviewer in the manual review step (review step 3) selected to approve the application, the application was not visible. This has been fixed.

  • DR 150703 Uploaded signature definition not used when conditional package added. This has been fixed.
  • DR 151732 Global Atlantic Forms Generate Blank. This has been fixed.
  • DR 151857 Allow editing of Product Code from Admin UI. In FireLight Admin on the Product page in Profiles, added an editable Product Code field.
  • DR 151944 App - E-Signature - paging in review document is not functioning properly. This has been fixed.
  • DR 152050 Template forms in conditional packages cause all rules not to fire. This has been fixed.
  • DR 152067 App - Initials and paging issue in Chrome & Firefox browsers when Initials Signature are not on first page. This has been fixed.
  • DR 152138 Pages Keep Automatically Incrementing when Viewing in “Sign Later” as Client with Initials – This has been fixed.

2.9.0.x (October 28, 2016 Production Release)

Enhancements

Client Fill and Sign

  • CMR 116245 Client fill and signature in the same request – Added the ability for clients to fill in remaining information on an application and sign the application in one step. In FireLight Admin on the Organizations page in Profiles, added a new Allow Client Fill & Sign checkbox to turn on the feature. On the Designer page in Field Properties, added a new Office Use Only checkbox to be selected (checked) for any form fields that the agent is required to fill in. In FireLight, pages with required agent entry fields are indicated with a darker pink background and "Office Use Only" text in the Form Entry validation list for the application. The Other Actions menu contains a new Request Client to Fill & Sign option for the agent to initiate the client fill and sign request.

Console

  • CMR 144034 Add signing ceremony only form enhancement to console– In FireLight Console, the signing ceremony only form shows only during the signing ceremony for console now as well.

Cooperative Technologies Ceding Carrier Database Integration and Replacement Processing

  • CMR 133287 Integrate with Cooperative Technology Ceding Carrier Database – In FireLight, added the ability to integrate with Cooperative Technology Ceding Carrier Database for replacement processing. Workflow is based on replacement requirements passed in the file. For example, if the ceding carrier will not accept eSignature on the ACORD 951 form, then we do not allow eSignature on the order. Cooperative Technology is responsible for contracting directly with the clients. We are simply providing the integration to support our clients.
  • CMR 135497 Replacement processing enhancement – In FireLight, added a replacement processing enhancement via logic to denote replacement forms, creation of an easy print button for the replacement forms and a subsequent upload button for the easy upload of wet signed forms, along with integration with Cooperative Technologies Yellow Pages for the ceding carrier requirements. Clients should use the new pre-tagged and updated ACORD 951 eForm because it has added tags for the ceding carrier data and the new button. Contact the Forms team to get the form. See Replacement Instruction Page.

Deployments

  • CMR 86775 Admin: Provide support for administrative scheduled deployments – In FireLight Admin on the Deploy page, added the ability to schedule deployments via a new Schedule Deployment at text box and Schedule Deployment button. The former Deploy button was changed to the Deploy Now button.

Email

  • CMR 135145 Admin: Add the ability to update email template subject lines in the Admin tool – In FireLight Admin on the Profiles, Email page, added the ability to update email template subject lines. Subject lines for email templates can be modified through editing email templates, localization and branding in FireLight Admin. In Admin Help on the Email page, a new column includes tag names that can be used in the email templates.

eSignature

  • CMR 114027 Provide the ability to disable "use E-Signature" – Added the ability for clients to implement their individually defined eConsent form in their package of forms and then leverage this eConsent form as a "special" form and no longer present the base platform's "consent to eSignature" page. Added a new common tag, FLI_DECLINED_ESIGN that can be null/empty, true or false. If set to null or empty, the process will run as usual. If set to true, the e-signature process is skipped and the next defined step will take place. If set to false, the user will have to accept the e-signature process. To enable, add the common tag to a radio button (Name = FLI_DECLINED_ESIGN, Group = FLI_DECLINED_ESIGN, and Export = true).

Forms Designer

  • CMR 142186 Admin: Form Design - Protection property for Agent, add the missing feature to exclude from download – In FireLight Admin on the form Designer page in Field Properties, the Protection property now honors the Agent option in forms that are available for viewing and printing in FireLight. The original design of the client fill request was for an admin/home office function which is why the protection property was not honored. Over the past few releases we have added new features that can be used to share applications in a home office environment that work better and is more efficient. Because of that, client fill and sign should only be used for client requests and the form protection should be honored. This is a change to the use case around this feature. Clients should not use this feature for home office sharing but use the new sharing features that were released in 2.8.
  • CMR 133767 Rules: Enhance Rules Window search functionality – In FireLight Admin on the Designer page, Rules Window in the search field, added the ability to search/filter rules with a partial tag name and it will show get/set range rules if the tag is affected by the range.
  • CMR 133888 Admin: Conditionally set the contents of a dropdown – In FireLight Admin on the Designer page, Rules Window, added two new setcomboitems and getcomboitems nodes under Get Set. Use the new nodes to have a dropdown list (combo box) with content that changes based on the condition.
  • CMR 136217 Admin: Application copies – In FireLight Admin on the Designer page, Rules Window, added a new copyappstatus node under Get Set that identifies whether the current application is a copy and what the Product and State were on the original application. This helps to identify certain data items that need to be changed or reset/cleared to ensure they are elected for the new application.
  • CMR 136499 Admin: Rules for packages – In FireLight Admin on the Designer page, Rules Window, added a new setpackagestatebyname node under Get Set that can show/hide a whole package. This allows clients to use rules to determine which package to use within the selling agreement.

Integration

  • CMR 129292 Allow Rules to access Custom List data – In FireLight integration, added the ability for rules to access Custom List data. This allows users to store product variable spreadsheets rather than coding values directly into the rules.
  • CMR 135628 Add ability to receive client data in SSO without requiring inclusion of the product identifier in the ACORD 103 – In FireLight, added the ability to pre-populate client data using the ACORD 103, but not include the target (New Application, Jurisdiction and Product) and still allow the user to land on the FireLight home page.

Money Settlement Through DTCC

  • CMR 134970 Money Settlement through DTCC – Added the ability to use the DTCC's Insurance & Retirement Services to centralize cash flow through their National Securities Clearing Corporation (NSCC).

Organizations

  • CMR 135779 Admin: Default error placement location – In FireLight Admin on the Profiles, Organizations page, added a Default Validation Placement field (below the Edit Masks field) that is used to set the default location for validation errors to appear next to form fields in FireLight. The Organization level setting can be overridden by selecting a different Error Placement option in individual form field properties on the Designer page in Admin. At the individual form field level in the Error Placement field property, the Default option = the Organization level setting for the Default Validation Placement field. This change does not affect forms from previous versions before FireLight v2.9.
  • CMR 135803 Admin: Unique rule name required – In FireLight Admin on the Profiles, Organizations page, added a "Validate rule name for uniqueness" checkbox. Select the checkbox to require all new rule names added in the form Designer to be unique. Existing rule names will not be affected. Enable this feature to aid in testing and debugging forms. It is recommended you do not switch between enabling and disabling this feature.
  • CMR 139390 Admin: Add organization option for default state of Force Deploy option. – In FireLight Admin on the Organizations page, added an Enable Force Deploy checkbox. Select the Enable Force Deploy checkbox to cause the Force Deploy option on the Deploy page/Deployment tab to default to selected (checked).

Packages

  • CMR 134978 Admin: Present form during signing ceremony with no signature control – In FireLight Admin on the Profiles, Packages page, added a new "Signing Ceremony Only" document type in the Forms in Package form inclusion option selection list. This document type is for forms to be presented during the signing ceremony with no signature control (for example, necessary disclosures).
  • CMR 135783 Admin: Allow multiple forms in package to be moved up and down – In FireLight Admin on the Profiles, Packages page, added the ability to move multiple forms at a time within a forms package by selecting multiple forms and clicking the up or down arrow to the right of the Forms in Package list.

Post-Issue Services

  • CMR 127725 Build out support for service forms (Post Issue) – In FireLight Admin on the Groups page, added a new Enable Create Post Issue checkbox that allows users to create post issue service transactions in FireLight. In FireLight on the Home page, users click the New Application button and if more than one transaction type is enabled, the user clicks the applicable option to create a New Business Application, Post-Issue Service Forms, or Subsequent Payment. There is a licensing addendum to support Post Issue. Contact your Strategic Account Manager to begin the addendum process.
