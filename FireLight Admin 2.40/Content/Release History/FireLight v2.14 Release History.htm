FireLight v2.14 Release History

2.14.0.112 April 26, 2019 Production Release

Enhancements

Admin

  • CMR 191551 In Reports, added a new Wizard Field Report to the Admin Reports that lists field information for the selected wizard (similar to the Form Field Report). The CSV file is an extended version of the report with additional field properties. In the Admin Help, see Admin Reports"Admin Reports" to view a sample Wizard Fields Report. Contact your Strategic Account Manager to request this report for your organization.
  • CMR 195320 Added the ability to co-mingle wizards and forms within the same activity, which allows data entry in any order based on the order of the forms and wizards in FireLight® Admin on the Profiles > Packages page. Both wizards and forms are seen or hidden in the UI using the existing FireLight visibility logic. Forms can be required, conditional, print only, optional, signing ceremony, and signing ceremony only: conditional. Wizards can be required, conditional or optional in a Package. Forms with signing ceremony only are hidden in the UI, but included in the signing ceremony. Templates continue to use the existing FireLight logic. When wizards and forms are in the Package and Sales Agreement, the UI uses Previous and Next buttons that are in a fixed position near the bottom of the form and wizard screen unless the current activity is a consumer portal activity. Also added the ability to customize wizard pop-up text by client. In Admin Help, see "Add and Edit a Package" in Profiles > Packages for more information. In Profiles > Packages, enhanced the Signing Ceremony Only form inclusion type so it can be used to present data that was filled in via a wizard and allow signatures and submission to a carrier's back office by default. Also added a Signing Ceremony Only: Conditional version of the Signing Ceremony Only form, which requires a rule to be added to trigger the form. The Signing Ceremony Only and Signing Ceremony Only: Conditional forms can be excluded by setting the document type in the provider to SigningCeremonyOnly. See Signing Ceremony Only in form inclusion types in Profiles > Packages for more information.
  • CMR 195323 In Profiles > Organizations > Activity tab, added a Purge Uncompleted E-Delivery setting. In the text box, enter the number of days after an E-Delivery case was finished (this includes Expired, Canceled, Re-Issued and Completed cases) before purging the case. The default is 10 days. The default is much lower because E-Delivery cases use more space than other Activities and we need to limit the space used. The number of days allowable before purge of E-Delivery cases must be between 7 and 30 days.
  • CMR 195671 In Utilities, added three new utilities for converting dataitem tags from old tags to new tags: (1) Download Tag List utility for downloading a spreadsheet of existing dataitem tags within a package that can be manually updated to include old dataitem tags to new dataitem tags and list value columns; (2) Package Conversion utility for converting dataitem tags in forms, wizards and rule sets within a package to new forms, wizards and rule sets with new dataitem tags from the mapping spreadsheet, and setting an expiration date when the new forms, wizards, and rule sets will replace the old ones within a package; (3) Generate Mapping Rule Set for building a mapping rule set between one set of dataitem tags and another set of tags based on a mapping spreadsheet. These utilities are especially useful for converting tags to common tags. Contact your Strategic Account Manager for scheduling and development assistance with converting tags in the Provider. To enable the three new utilities, select the Enable Mapping Utilities checkbox on the Profiles > Groups page, Admin tab. In Admin Help > Utilities, see "Download Tag List," "Package Conversion," and "Generate Mapping Rule Set" topics for instructions.
  • CMR 195790 In Reports, added a Case Access Verification Report that lists applications that were viewed by someone other than the owning agent. The report provides an Exclude Role Code parameter to exclude designated group role codes from the report for support personnel that have a legitimate reason to view an agent's application. In Admin Help, see Detailed Activity Reports"Detailed Activity Reports" for a sample report and details.
  • CMR 197481 In the form designer, added "Client" to the Signer Type list in the Properties for the Signature Box.
  • Added new rule nodes to search for forms or wizards and set their data entry visibility: getformbyname, getformbynumber, getallforms, getallwizards, getformsbypackage, getwizardsbypackage, and setdataentryvisibility.

  • CMR 180273 Added three new Admin reports that show which forms, wizards, packages and rule sets are not found in any package/sales agreement. This allows Admin users to quickly determine which forms, wizards, rule sets, and packages can be deleted from the system.

    • Report that shows which forms and wizards are not found in any package.
    • Report that shows which rule sets are not found in any package.
    • Report that shows which packages are not found in any sales agreement.

Admin Help

  • In the "Design" section of the Admin Help, Common Tags topic, added an updated CommonTags.xlsx spreadsheet and an updated CommonTags_ListName.xlsx spreadsheet that include all common tags along with the new additions for the April 26, 2019 Production release. Removed the "FireLight Global Data Items" topic from the Admin Help.
  • In the "Getting Started" section, added a new Spotlight Feature Videos topic with URL links to videos of FireLight features from recorded Spotlight presentations.
  • Added teal highlighting to search results for locating search words and phrases more quickly in long topics. Click the Remove Highlights button on the upper right of a search result topic to remove the highlighting from the topic.

All Activities

  • CMR 192074 Added the ability for a user to share a single activity with one or more other users while an activity is in edit mode (unlocked) from the Other Actions menu. This applies to all activities except e-Delivery child cases. For example, an agent can share an activity with the sales desk or an assistant to complete the activity. In FireLight Admin on the Profiles > Groups > App tab, select the "Allow Share Activity" checkbox to enable this feature. When enabled, in FireLight Admin on the Profiles > Email page, the "Accept User Share - Single Activity," "User Share Request - Single Activity" and "Revoke User Share - Single Activity" email templates become available for sending. We renamed the existing User Share all activities email templates to "Accept User Share - All Activities," "User Share Request - All Activities," and "Revoke User Share - All Activities" to differentiate among the emails for sharing a single activity and for sharing all activities. On the Profiles > Groups page > App tab, we renamed the "Enable User Share" option to "Enable All Activities User Share." Added a User Share Report in Admin Reports. To streamline the process, we also removed the ability to Resend a user share request that has not been accepted. See the "Share" topic in the FireLight Platform User Guide under FireLight Basics > Other Actions for instructions.

Common Tags

  • CMR 199065 Spreadsheets with only new common tags and list values for the April 26, 2019 Production release including FireLight Base FLI common tags, Whole Life common tags added to Life, Schedule common tags added to Annuity and Life products, and other miscellaneous common tags and associated list names are available in iConnect.

E-Delivery

  • CMR 195226  Added the ability for the agent to sign documents within the e-Delivery workflow. The agent's ability to sign becomes available when the application package has required agent signature controls and after all other parties have signed the child applications. In FireLight Admin on the Profiles > Email page, added an E-Delivery Pending Agent Signature email template that is sent to the agent when the agent signature is required. See the FireLight E-Delivery Guide in PDF Guides for detailed information.
  • CMR 195322 Added the ability to have a consumer (client) directed workflow in e-Delivery. An e-Delivery case sent into FireLight can now go directly to the client without requiring any agent involvement. Added audits specific to client directed workflow and implemented base outbound messaging for client directed workflow.

FireLight Embedded API Updates

  • CMR 195324 Added several new enhancements to FireLight Embedded via the activity and client side APIs.

    • Activity API updates include: support the ability to pass in documents to FireLight when calling the create activity API and the ability to use the FireLight product selection dialog within FireLight from Embedded.
    • Client side API updates include: Navigation system can now be utilized when integrating with FireLight through embedded, support the ability to continue from one activity to another via client API call, and can show all validation messages via client API call.
    • CSS updates include CSS split into two levels for wizard: a bare minimum that renders the wizard (required CSS controls grid layout, responsive layout, etc.) and a default styling (implementer no longer has to create everything from scratch).
    • Added versioning to both APIs to make it easier to transition between new features within the API.

Illustrations

  • CMR 192396 Changed where the Save occurs on illustrations in FireLight because illustrations are often run as ‘what-if’ scenarios and do not need to be saved. After selecting a Jurisdiction and Product, removed the dialog for adding an illustration Name. FireLight no longer shows unsaved illustrations in the Recent list or All Activities page. If the illustration has not been saved and the user clicks Save on the navigation bar, a Save Illustration dialog opens where the user can enter an illustration Name and click Save to save the illustration. ‘Save As’ logic similar to ForeSight® (same as ‘Copy As Is’ while in activity edit mode) has been added to all activity types except e-Delivery.
  • CMR 195321 In the Wizard Designer, added the ability to add a schedule to an illustration wizard. Four pre-built rule templates provide common schedule configurations for one-input, two-input, three-input and four-input column schedules. See Schedule in Admin Help for guidelines, instructions and rule samples.

Navigation Bar

  • CMR 181446 In wizards and combined wizard and forms UI, updated the navigation bar so the header row (Alerts, % complete, steps, Other Actions, Home button, Save, Save As, etc.) is locked at the top of the screen when the user scrolls down.
  • CMR 195668 Updated the FireLight navigation bar. Added a link icon on the right if there are any linked cases with the current activity. Click the link icon and click a linked activity name in the dialog to open a linked activity (e.g., e-Delivery cases). In Wizards, near the top left, replaced the Open/Close button name with the name of the page the user is visiting. Open/Close button remains the same in forms only UI. The content displayed in the dropdown did not change. Any client specific branding pertaining to the Open/Close button remains unchanged.

Needs Determination

  • CMR 195673 Added the ability to rename the Needs Determination activity to the organization's desired name and skip the product and jurisdiction selection for the subsequent activity. Clients is able to change the name of the existing FireLight Needs Determination system wide, which includes the UI, the audit history, emails and reports. In April, the rename must be completed by the FireLight team. Contact your Strategic Account Manager to submit a work order for changing the Needs Determination activity name.

    Additionally, clients now have the option of selecting product and state in the presale activity that will pre-fill into the next activity such as the application. This adds the ability to skip the Product and Jurisdiction selections for e-Application when they have been selected in the presale activity. In order to skip the application product and state selections in the next activity, clients need to set two new FireLight global data items (FLI_NEXT_ACTIVITY_PRODUCT_CUSIP and FLI_NEXT_ACTIVITY_JURISDICTION). Clients can collect these values in input controls (dropdown lists or custom lists) in the presale activity, set the data items, and then use these values for the subsequent application. If the values are specified and are available, the user will automatically move into the next activity with these values. If the selections are not valid, or the user does not have access to create the product and jurisdiction, FireLight displays a message to the user and the user needs to go through the normal activity creation workflow. In the Admin Help, see FLI_NEXT_ACTIVITY_PRODUCT_CUSIP and FLI_NEXT_ACTIVITY_JURISDICTION in the "Common Tags" topic in Admin HelpCommon Tags spreadsheet for more information.

Security

  • CMR 196193 Need the ability to disable a direct login account after 120 days of inactivity. Added a check for last login time before validating a user. If it is more than 120 days, prevent login. Vendor Admin can reset the last login time to allow old user account to be re-enabled. NOTE: The same account types that are not required to change the password after 90 days are excluded from the logic to disable accounts after 120 days.

Signatures

  • CMR 188205 Updated the Send Email to Client for Signatures screen. Changed the word "Client" to "Signer" in the field names on the Send Email to Client for Signatures screen. Now the field names are: Signer Name, Signer Email, Your Name, Your Email, Signer Last 4 Digits of SSN/Government ID, Signer Birth Date, and Signer Mobile Number (if SMS Text Authentication is enabled).

  • CMR 196717 Changed the wording for the Require Signature Capture popup. This applies to anyone with a Group setting of "Require Signature Capture" or if you have the Cooperative Technologies integration that could force the live sign on the 1035 forms per the ceding carrier rules.

    Old Verbiage: You are required to sign your signature on the signature pad.

    Approved New Verbiage: A signature capture is required. Please sign using a mouse, your finger, or a stylus.

Admin

Firelight presented a pop-up if you would like to keep the changes to a form, but automatically went to the next form after a very moments and saved the changes. This was fixed as part of another change. The popup now behaves as expected, and will not change forms until an action is selected.
  • DR 189003 In Profiles > Email, added Agent Name as an available replacement tag for the Review Queue Message email template. The name used is the User Full Name in the application owning agent's Preferences page.
  • DR 189635 In Design > Wizard Designer, the Custom Action buttons did not have an option to specify which forms can be printed. This was fixed by adding the ability to specify an attachment ID in the Custom Action Parameter property in the Custom Action field properties. If an attachment ID is specified in the Custom Action Parameter, clicking the print button will only allow forms to be downloaded that have the specified attachment ID.
  • DR 190177 In Profiles > Groups, an exception occurred when updating the Groups profile. On the Needs Filter tab, when selecting a product for product access, the Filter drop-down data was not getting set up on the server. This was fixed by adding logic to fill out the filter product drop-down for the URL action that handles selection of product access.
  • DR 191527 Unable to export forms when logging in via SSO. You needed to specify a USER_ROLE of Carrier rather than Agent in the SAML for the form export feature to work properly. This was fixed by also authorizing the Agent role in admin SSO.
  • DR 191553 Email for auto-submit would be sent when the agent declined e-sign and the group options for Enable Auto Submit and Enable Auto Submit Email Notification were turned on, but the application was not submitted. This was fixed by verifying the Organization setting for Allow Submit on E-Sign Decline is turned on in addition to the Enable Auto Submit option in Groups. If Allow Submit on E-Sign Decline is turned off, the Auto Submit Notification Email is not sent, the application does not attempt submission, and the user is returned to instructions on e-sign decline.
  • DR 191759 In Design > Wizard Designer, the Custom Action buttons did not have an option to allow for non-PDF document type upload in the Custom Action button properties. Implemented the missing properties for Allow Non PDF File and Disable Change Document Type in Wizards.
  • DR 192044 In Profiles > Sales Agreements, when using the Zoom feature in the Firefox browser the Time Zone was missing from the page. Fixed a layout bug that applied only to Firefox on the Sales Agreement page.
  • DR 192562 Issue when using setcomboitem rule. If a drop-down list changed, and the new values did not contain the data item's current value, the data item's current value would not be displayed (which is why the 'required' rule appeared to not work) - additionally, a new additional blank space would be shown in the drop-down list. This was fixed. When the drop-down list changes via rules, the data item's current value will clear if it is not available in the new drop-down list.
  • DR 193392 Issue with secontrolstate and multiple dataitemids with radio buttons. Clear On Disabled was triggered when selecting an option lower in the order of the radio button. This meant there was a brief state where the rules ran, and everything was disabled. This was fixed by updating the Radio button to send over the checked item as the first dataitem so the update field will trigger the dataitem value change in the correct order.
  • DR 193562 In the Rule Set Window in the Wizard Designer, the box for filter rule sets was too tall. The default height for the Rule Set Window is 30%, which was overwritten in the client .css file by 85%. This was fixed by removing unused client .css from disk and adjusting the Rule Set Window height.
  • DR 194459 Address Autocomplete fields are not working with the Rules Engine. The fields should work with errormessage, setvisibility, and setcontrolstate rules to force the user to enter data and then use the setvisibility rule to hide it after data is entered into the underlying address field. The field should work with setcontrolstate to control when the field can be used (i.e., if no joint owner, it should be disabled). Hid autocomplete clear 'X' button when field should not be shown.
  • DR 194462 When deleting a row from one Custom List, a row was deleted from all Custom Lists instead of just the list being edited. This was fixed by passing the list ID and row number so only the row in the list being edited is deleted.
  • DR 194762 FLI_ Data Elements do not update to the new product when using the Replace Product (product replacement) in the sales agreement. This was fixed by updating the product related dataitems at the time that the app is opened to reflect the updated product.
  • DR 195343 Policy Block Settings Clarification. On the Profiles > Organizations tab, Policy Block Settings are not deployed via the Organization Configuration synchronization scope and the item is not marked with an asterisk like all other items which are not deployed. This was fixed by adding an asterisk to the Policy Block Settings* item on the Admin tab in Organizations.
  • DR 196383 When creating a Package and adding a rule set to the package, FireLight added all rule sets to the package instead of letting you select the rule sets when using the Search feature. All rule sets were added to the package after clicking the Search button. Search was not working if a package had been updated or created, but not saved. This was fixed by not refreshing the list on Search.
  • DR 197163 In the Wizard Designer > Text Properties dialog, the checkbox options at the bottom of the dialog had a sizing issue. The styling for the checkboxes was set to take up entire width. This was fixed by modifying styling to only take up checkbox and text width as checkable (selectable).

  • DR 192385 Wizard error messages were overlapping the radio buttons if the component hosting them was too small to contain the message. This was fixed by modifying the error message container so the error message is hidden and a tooltip is used when the container is too small to fit the message.

Error styling was only implemented for input fields (single-line text boxes) and not for textarea fields (multiple-line text boxes).   • DR 196086 Slow saving of wizards. This was fixed by removing double-sanitize to improve save speed because the second sanitize isn't necessary.
  • DR 197157 Labels on forms were not working in the forms within wizards UI. This was fixed by making modifications to forms within wizard UI so that labels more closely match that of native forms.
  • DR 197159 Wizard spacing issue: (1) Font sizes were incorrect when rending forms in the new UI. This was fixed by updating font size mapping logic in the new UI to match legacy forms rendering. (2) Prior to 2.14, we were allowing all text to show in field labels, which often resulted in label text overflowing other fields on the page. In 2.14, this was changed so label text is truncated and an ellipsis is shown if there is not enough room for the entire label, which meant there was not enough room for standard yes/no, true/false labels in 1-wide checkboxes within panels. This was fixed by restructuring styles and eliminating unnecessary horizontal spacing between components. Yes/no, true/false will now fit.   • DR 197549 Partial application was not saved or recovered in Internet Explorer browser. Session expiration times were not being updated unless at least half the session had passed. Because of this, if data was entered in the first half of a session, but not after the halfway point, the session would already be expired by the time the UI attempted to save data. This was fixed by adding a keep-alive check at specific intervals based on the configured authentication timeout.
  • DR 197571 On the product selection screen, the product type filter was not working correctly. This was fixed by initializing product type first.
  • DR 197598 Unable to edit an application name in 2.14. If an application name is too long, the edit icon gets hidden behind tool buttons so the user was unable to click the edit icon. This was fixed. If application name is too long, it will wrap to the next line.

  • DR 197606 v2.14 introduced the capability to edit and view forms and wizards together in the wizard user interface (iConnect 195320). The intended behavior is for forms-only applications to continue to use the legacy forms-only user interface, and for applications with wizards to also be able to show and edit forms. The system was checking for a non-null wizard definition to determine if an application contained any wizards and redirect to the wizard UI, but due to a data issue where the wizard definition was not null, but blank in some cases, for many organizations with the first UAT release on 3/22, even forms-only applications were opening in the new wizard user interface, which exposed various problems with forms in the new wizard interface, including alignment issues, circular radio buttons, controls displaying on the wrong page, issues with execution of rules, etc.

    The problem was reported on 3/25, the issue above was identified on the morning of 3/26, and the database was corrected in UAT Next at noon on 3/26, correcting the problem and causing forms-only applications to continue to use the legacy forms user interface. The code was also modified to correctly handle the scenario of a non-null, but blank wizard definition, in case that scenario should recur in the future.

    Some of the wizard UI issues that this exposed have also been resolved for the 3/29/2019 UAT release, and these are being tracked separately. Work will continue to achieve feature parity between the new wizard user interface and the legacy forms UI.

If the help text is added, the field width must be set to a minimum width of 2 to accommodate text and help text.

  • DR 182688 Revoke signature request could be triggered by clicking anywhere in the button and not on the Revoke Request link causing a request to be unintentionally revoked.   • DR 192472 Agent signature did not display on the completed application. The captured signature size was below the minimum image size. This was fixed by implementing server-side validation of minimum signature image size and non-blank signature.
  • DR 192932 Agent name did not map into the List of Required Signers page. It also did not map with the agent Preferences information. This was fixed.   • DR 193331 The Signatures automated service to initiate the signing ceremony was using the wrong email template. This was fixed. It now uses ESignature instead of EAppFill.
  • DR 194042 Require signature pad requirement was enforced even if form was excluded from the application. This was fixed.
  • DR 198589 Signing ceremony only forms were shown in edit application UI after entering signing when previous version did not show them. This was fixed by hiding the signing ceremony only forms in the edit application UI. Even if forms contain signatures, they can be viewed in the Print/Display PDF view.
