Create User, Transfer, Email Custom Action (Concierge Paper Workflow)

  • CMR 213304 (Distribution) Extend Custom Action to support Create User, Transfer Application, and provide Email access. Organizations are looking for a way to convert significant volumes of paper transactions still being submitted into digital transactions. Both carriers and distributors offer support services to key producers that submit paper applications.

    This workflow offers concierge service workflow that is extendable to any FireLight organization. When complete, organizations is able to manage nearly all functionality related to this workflow through the FireLight Admin tool; availability by organization/role, custom action button configuration, and email design/content through template. The only optional functionality requiring Hexure involvement would be branding. See the Create User, Transfer, Email Custom Action (Concierge Paper Workflow) topic in Admin Help for details.

    Back Office/Case Manager Workflow

    • Back office enters FireLight and keys in paper application.

      We are aware that alternative entry methods are being explored by some of our clients, i.e., OCR scanning, etc. This step need not be manual, but that will depend on the back-office process defined by each client. The significant aspect is that the application is initiated by someone other than the writing agent.

    • In Good Order applications are Submitted.
    • If the application is NIGO and the agent is not an existing FireLight user, the back-office initiates creation of the agent in FireLight, transfers ownership of application to that agent, and notifies and provides access to the application through an email to agent. All actions stated is triggered though a single click of a custom action button.

    • Once ownership of the activity is transferred to the new user, the back office will no longer have visibility of that application unless case access is granted to the transferee's role code through the back office group settings in the Admin.

    • If Create/Transfer/Email functionality is used to transfer an activity to a user that already exists based on the User ID entered, the case will still be transferred, but a new user will not be created.

    Agent/Adviser Experience

    • Agent Receives an email with verbiage defined by the organization.
    • Agent clicks on the link and is presented with a brand-able login page requesting information known by and unique to the agent.
    • After clicking on the Login link, the email, the agent is taken directly to Data Entry for the specific application that requires their attention. If desired, a branding code could be passed in the login to modify FireLight. Agent completes the requested changes and follows the workflow defined by the Organization for this particular scenario.

    Impacts

    • Admin: Addition of multi-parameter custom action capabilities in the field properties dialog to drive multiple custom actions from a single custom action button.
    • Admin: Add the ability to create a new user within the forms/wizard interface.
    • Admin: Addition of two new Email templates allowing you to manage the email content for both activity transfer scenarios; current Transfer Activity Request, and new Transfer Activity with Direct Link.

DTCC Subsystem

  • CMR 213369 (Carriers and Distributors that submit through the DTCC) This enhancement improves the way we're currently doing business with the DTCC. The DTCC subsystem needed to be revamped to support getting data to the DTCC Subsystem, moving that data to the DTCC, and then enhancing the notifications that are given from the DTCC to us and to our carriers, which includes receiving the final notification back to Hexure on success or failure of a DTCC attachment request and app/sub.

    For this release, we are focusing on the initial connection between Hexure and the DTCC. Currently, we are connected through an ANIRA VPN connection. We is moving to utilizing the internet through a SFTP connection. This will eliminate the private VPN requirements, and will establish an Internet connection that is maintained through a username/password login.

    Impacts:

    • Current DTCC Client Connection - This change should be transparent to clients; however the change will apply to all clients that have data submitted through the DTCC. Clients will not have to change anything for this new submission method, but clients will want to test this in UAT prior to production.

FireLight Real Time Activity Statuses

  • CMR 214065 Provide real time activity statuses for activities. Currently, in order to find a current status on activities in FireLight, activities must be checked individually. In situations where clients have a high volume of activities in flight, this could be a cumbersome process. If FireLight would post the current status to a designated client endpoint concurrent with each status change it would simplify and streamline the process of status updates and facilitate follow-up.

    The post to the client endpoint is executed by the client's provider code on each status change. The payload of the post will include:

    • Application ID
    • Status Code - High Level
    • Status Description
    • Status Code - Display Status
    • Date/Time of Status Change

Review Queue

  • CMR 213738 (Distribution) Review Queue access and permissions. This enhancement allows for a user to have more granularity in the selection of function for each review queue per role code. Within FireLight Admin currently, a user can select one permission for multiple queues for a given role code. With this enhancement, a user can now select multiple queues, but also have different permissions set up for each queue. For instance, if I had an Admin Role code, and within my queue 1, I could also act be a supervisory role and be able to unlock all of the activities within that queue. However, within my queue 2, I could only have viewing rights, so this would be set to a read only permission. This gives a user greater flexibility within the review queues under each role code.

    Impacts:

    • Pre-existing set permissions and queues for each role code - Each client will need to verify that previous set permissions and queues are set correctly from 2.17.
    • "Disable Email Action Emails" and "Enable Manual Queue Move" will stay global permissions, and cannot be changed per queue.
    • Review Queue and Review Queue Access tabs have been combined.

  • CMR 213806 (Distribution) Allow admin to return an application to a Review Queue. If a user needed to recall an activity that had recently been rejected, there was not a current solution that a user could use within FireLight to allow for this action to occur. This enhancement allows a reviewer the ability to recall a rejection request for an activity in the post-submission Review Queue. This is based on a role code setting within the Admin that will add a recall button to the Review Queue History tab if an activity has been rejected for any reason (i.e., Selecting Recall, or reaching the set deadline date). This enhancement allows the user the ability to control if an activity needs to be re-reviewed by an additional reviewer, and is able to stop potential mistakes made by selecting the reject button prematurely, or allowing the activity to reach the deadline date with any action taken.

    Impacts:

    • Reject Functionality - when an activity is rejected, a user now has the ability to re-release it back into the same queue it was released from.
    • Role Codes - This feature is only available through role codes.
    • Audits - This feature will keep a record of each rejection and when it is recalled.
    • Reviewer Documents - If an activity is rejected and recalled, the reviewer documents and any comments made by the previous reviewer are retained for the next reviewer to review/edit.
    • Review Queue History Tab - UI - An action button has been added to allow a recall on a rejected activity to occur.
    • Providers - a Provider hook was added alongside this functionality in case any provider needs to be done on an individual client basis. This shouldn't result in any impact at this time.
    • Sequential Review Queues - Reviewer documents is retained from a recall, and the comments retained in the original queue, but since each queue has a brand new document set, these comments will not be retained across sequential review queues. However, these comments can be copied through a client's provider at this time. If interested in this, please reach out to your Strategic Account Manager.

  • CMR 215553 (Distribution) Add Reviewer Document Link to the Review Queue. The user needs the ability to see reviewer uploaded documents within the review queue. In this enhancement, we added the ability for these documents to be seen within the History tab of the Review Queue. This is shown through a hyperlink, that when selected, shows the list of attached documents in a separate dialog. The user can also select all documents at once, or individually.

    Impacts:

    • Review Queue History - UI Changes - an additional hyperlink has been added that will display the attached documents upon selection.

  • CMR 216981 (Distribution) This enhancement allows for Reviewer and Supervisor comments that are included in the Reviewer Documents to be viewed in both the review Queue History and review Audit Report.     • Review Audit: New section within the Review Audit titled, Reviewer Audit Comments

  • CMR 218228 (Distribution) Improve Review Queues Search with Save and Results Persistence. Currently within FireLight, a user can select the advanced search in the Review Queue and enter the information to narrow their search, but there is not a feature to save this search to go back to at a later date. These issues were resolved: (1) modified the legacy forms model to not create the error placeholder field for locked activities, (2) modified the wizard UI to: a) handle the error placeholder field type by rendering as a label instead of showing an error message. (NOTE: the placeholder is defined with coordinates of 1, 1, 1, 1, and I'm not sure if the wizard UI will render such a label properly – changes may be required for rule errors to show properly) and b) continue to log error messages for an unsupported field type (all types should be supported at this point) to the server and client UI.
    Areas Impacted: Wizard UI
  • DR 230995, 231189 Wizard - issue with background color of labels. Labels were being used to hide text on a pdf with text labels put over top. The text from the pdf was still showing through. This was resolved. The labels on the form did not have a background color set and when the form is included in a wizard package, the labels were being set to transparent instead of white. This had been corrected with the resolution of another bug. This was changed back to a default of white for this release.
    Areas Impacted: Background of any form field converted to wizard.
