    New and Updated Insured Underwriting Common Tags and List Name:

    During a recent review of the insured underwriting common tags, we realized we had incorrectly mapped the ACORD mapping for the Insured_Underwriting_RiskClass common tag. Therefore we made the following changes:

    1. Insured_Underwriting_RiskClass –

      1. We removed the ACORD mapping since it does not apply to risk class.
      2. We changed the List Values in the List Name file to be negative values since there is no ACORD mapping. These now map to common FireLight risk classes typically used in life illustrations as well as in the new Quote activity added in 2.15.

    2. Insured_Underwriting_UnderwritingClass –

      1. Insured_Underwriting_UnderwritingClass is a new common tag.
      2. The ACORD mapping is now mapped to what was previously mapped in Insured_Underwriting_RiskClass in the prior 2.14 release, /TXLife/TXLifeRequest/OLifE/Holding/Policy/Life/Coverage/LifeParticipant/UnderwritingClass.

      3. There is also a new List Name, List_UnderwritingClass.

        1. The List Values are now set to what was previously shown in List_UnderwritingRiskClass in the prior 2.14 release.
        2. The ACORD List Name mapping is now mapped to what was previously mapped in List_UnderwritingRiskClass in the prior 2.14 release, ACORD OLI_LU_UNWRITECLASS.
    3. We also added Insured_Underwriting_Tobacco_Type and Insured_Underwriting_Tobacco_PremiumBasis underwriting common tags for life products.

    New Yes No List Name and List Values:

    It was recently pointed out that we have not standardized the List Values for Yes and No list values (a.k.a. export values).

    1. This applies to both dropdown list fields as well as checkbox and radio button fields.
    2. A new List Name was added, List_YesNo, with an ACORD mapping of OLI_LU_BOOLEAN. It is referenced in a variety of common tags. Refer to the CommonTags.xlsx to see which common tags are impacted.
    3. The List Value for Yes is 1. The List Value for No is 0.

    Updated List Values:

    We changed the List Values from negative to positive on the following List Names:

    1. List_Month (this list applies to the Annuity_Historical_Month common tag).
    2. List_NumberofYears (this list applies to the Annuity_Historical_NumberofYears and Life_Term_Duration common tags).
    3. List_NumberofMonths (this list applies to the Annuity_DollarCostAverage_Type common tag).

    This change was made because these values should be the actual numbers they represent, not negative values.

Configurable Pre-Sale Activity

  • CMR 196381 Added an Activity layer to the FireLight system that Admins can use to create and configure a new pre-sale activity and to select the next available activities without requiring code changes from the FireLight team. Use the new Activities page in Product Configuration to create and configure a new pre-sale activity. A unique ACORD Type is assigned by Hexure for each activity type. The ACORD Type is used to identify the pre-fill and for linking products and packages deployed from the carrier to the distributor. Multiple Activities can be created, but all must have a unique ACORD Type. An Organizations' existing activities appear on the Activities page.

    The distribution will define the activity settings in the Admin/Activities section. Carriers can use this for their organization too, but the Product Configuration/Activity does not deploy. Carriers can create a product and package with the ACORD Type Code that is deployed to the distributor via the selling agreement. Distributors can then link the product to the activity using the ACORD Type Code. This is why the ACORD Type Codes must be unique.

    Please note that Pre-Sale requires licensing. Contact your Strategic Account Manager for details.

    In the Admin Help and guide, see the Activities page in Product Configuration and in Admin Help see the Configurable Pre-Sale Guide in PDF Guides for guidelines and instructions.

    The following UI updates were also made for the Configurable Pre-Sale Activity feature:

    • Allow E-Approval: Removed the Allow E-Approval option from the Organization Settings page, Activity tab. Replaced it with the Allow Pre-Signature Review and Allow Post-Signature Review checkboxes for the PreSubmit review queue on the Activities page.
    • Products: Removed the Activity Type - Approved panel from the Products page. Replaced it with the ACORD Type and Activity Type lists on the Activities page.
    • Packages: Added the ACORD Type to the package. This is used to link Products and Packages to a distributor’s activity when a carrier deploys them.

Expiration Deadline Date (FINRA 7-Day Rule)

  • CMR 200031 To meet the FINRA 2330 rule that requires Variable Annuities be completed and processed before a 7-day timeframe, added the ability to track a review deadline in the Review Queue. In FireLight Admin: (1) On the Organization Configuration/Review Queue page: added a Deadline Length (days) field on a per-queue basis and added 3 options to select whether to count Weekends, FINRA Holidays, and Market Days as business days to be included in the deadline day count. Added a Deadline Date - Continue From Previous Queue checkbox. If your organization has the Enable Manual Queue Move option enabled in Groups on the Review Queue tab, you can select the Deadline Date - Continue From Previous Queue checkbox to continue the deadline date to the next queue when moving an order to another review queue. (2) Added an email template for notifying the Agent when the deadline expires.

    In Admin Help, see "Enable the Review Queue Expiration Deadline Date" in the Organization Configuration/Review Queue topic.

GIACT Service Integration

  • CMR 189466 Added the ability to integrate with GIACT services for bank account verification and authentication in FireLight. These services can be used for one-time and reoccurring ACH payments. gVerify is GIACT’s bank account verification service that verifies the status of bank accounts in real-time and reports any negative history associated with the account. Inquiries on currently assigned routing numbers will return the bank name assigned to the routing number. gAuthenticate is GIACT’s bank account authentication service that takes the gVerify one step further to authenticate the customer data against the account. These services work with both personal and business bank accounts. Implementation setup will require forms/wizard configurations along with rules to control the actions based on the GIACT responses. Contracting is required with GIACT and the configuration details of UserName and Password is needed in FireLight Third-Party Service Config Organization settings. No additional implementation fees from FireLight are required.

    In Admin Help, see GIACT Services Integration in Features.

Integration - Add PDF to an Inflight Activity Using the Provider

  • CMR 198621 Extended the Document Service to allow the Provider access to add or replace a PDF document. This new function lets the Provider send the binary PDF data as well as the DocumentTag associated with the PDF to add or replace an existing uploaded PDF. Polling for Web Service Response

  • CMR 200676 Added the ability to poll for web service response in FireLight. FireLight is now able to call out to a web service at any defined interval to attempt to receive a response, which allows the user to continue on within FireLight while the call out is running in the background. When a call has been placed that requires polling, there is a toast in data entry that is customizable in the provider communicating that they may continue with their application and the display status is changed to "Pending Response" (a sub-status of Data Entry). Once a response is sent back to FireLight, there is another toast in data entry that is customizable in the provider communicating next steps. A message is sent in the message center, which allows an e-mail to be sent if the agent has the "Send Message Center Emails" checkbox selected in Preferences, and the display status changes to "Response Received" (a sub-status of Data Entry). Some web service calls can take several days to process and order, so this feature provides the ability to poll the web service until a response can be retrieved. Contact your Strategic Account Manager to arrange for the integration provider work needed to implement this feature.

    In Admin Help, see Polling for Web Service Response in Features.

    The following changes were made to FireLight for this feature:

    • Admin: added a new rule node, <callasyncaction>.

    • App:

      • The display status of the application will get updated to reflect the current state of the external service call once the process begins (Pending Response) and again when it is complete (Response Received).
      • If the call is not complete, the Continue button is disabled.
      • If the user has Send Message Center Emails enabled in Preferences, the user will receive a message in the message center once the process has completed.

Security - Rework Authentication Scheme for APIs
