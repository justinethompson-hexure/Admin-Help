2.23.81 March 17 UAT / March 18, 2022 Production Patch Release

Activities

  • DR 254472 Rulesets do not appear to run until saved. Problem: We are becoming more suspicious that this problem is due to caching. We actually cache the rule script code as well as the deserialized rule definitions. The script code is cached internally, but the cached definitions are exposed to the rules engine, and if the cached definition was modified in some way, it could conceivably result in behavior like you are describing. Whenever a rule set is saved, it is discarded from the cache and reloaded from the database, which could explain why the problem clears up for a time after saving a rule set. Then at some later point, if the cached definitions change, the strange behavior could recur. Cached items are also dropped out of the cache after 2 hours, which could explain the problem disappearing for periods of time. We're not seeing where the rule definitions would be modified, but it seems worth it to at least rule that out as a possibility. If this doesn't resolve the issue, then we will know it wasn't the cause. Resolution: Modified the rule set caching to only cache the rule script code and not rule definitions. Note that this could have an impact on memory usage and processing time when loading rules for activity data entry.
    Areas Impacted: Loading and execution of rules

  • DR 254568 Illustration to Application, not able to edit App. Problem: Updating from an illustration would make the transferred data items readonly, regardless of the 'Lock Transferred Data Items' setting in the Illustration activity. Resolution: 'Lock Transferred Data Items' setting in the Illustration activity when updating existing app from an illustration. NOTE: organizations that want the data items to be readonly will need to check 'Lock Transferred Data Items' in their Illustration activity if they want to retain the old behavior.
    Areas Impacted: Read-only state of transferred data items in Update from Illustration

  • DRs 258960, 259439 Illustration - Value with % sign are divided by 100 and Annuities - Index allocations changing after generating illustrations. Problem: The Data Item conversion for 2.23 caused a change to the illustration app data mapping to modify the application's original data when converting the data to an appropriate format for the web request. This caused percentages to change in the app after running an illustration, as well as modifying some other fields of data. Resolution: We new create a new data items collection at the beginning of the mapping process that isolates the changes from the original data.
    Areas Impacted: FireLight Illustrations App Data Mapping

  • DR 258973 Policy number no longer seen on copied apps during signing ceremony and on generated PDF. Problem: Data items changed by rules were sometimes not sync'ed back to the data items collection. Resolution: Correctly sync'ed the rules changes to the data items collection in all scenarios after the rules run.
    Areas Impacted: Form only - Initializing and reinitializing the forms model

  • DR 259423 UI Not Refreshing after CustomAction. Problem: Async task data changes were not applied correctly and therefor the rules didn't run until the selected page is already validated and processed. Resolution: Committed the async task changes immediately and run rules before we validate the selected page.
    Areas Impacted: Processing pending async task data changes

Admin

  • DR 259604 Designer not allowing field names with | vertical bar character. Problem: The merge into the release branch mistakenly reverted the change for iConnect 256066, reintroducing that bug into the 2.23.79 release. Resolution: Put back the original code prior to the merge.
    Areas Impacted: Allowed field names in the form and wizard designer

Admin

  • DR 258951 Unable to edit rules from wizard designer. Problem: Changes for iConnect 250574 "Lock notification to prevent multi-user overwriting - Rule Set, Form and Wizard" introduced a bug that could result in an array index being exceeded because the code was using the active rule position (within the selected rule set) to index into the rule set collection. Resolution: Corrected the code to use the active rule set for lock information, instead of looking up a rule set in the collection based on the active rule position.

    Areas Impacted: Editing of rules for the selected field from the wizard designer

  • DR 259025 Checked Canadian provinces are unchecking themselves on rule sets after saving. Problem: Changes for iConnect 250574 "Lock notification to prevent multi-user overwriting - Rule Set, Form and Wizard," the result being that it isn't loading the jurisdiction approvals for higher jurisdiction IDs (> 63). If the rule set is then saved, without those jurisdictions being re-checked, those jurisdictions will no longer be approved for the rule set when utilized in app. Resolution: Fixed to load additional jurisdiction approvals that are stored for the rule set.

    Areas Impacted: Display of jurisdiction approvals in Admin -> Rule Sets

Embedded: Print Forms

  • DR 258915 Suitability paperwork is not displaying in Print Forms. Problem: When the activity runs rules for the first time when opening a packages gets included. This new package gets included in the users session but does not get saved correctly to the DB. When printing the DB is queried and the correct packages are not listed. Resolution: This was a timing issue. The model that manages the current "Save" state of the activity was not initialized and the IsDirty flag on the model was not getting set early enough.

    Areas Impacted: Embedded activities that add/remove packages on initial startup.

  • DR 259188 Loading of PDFs for print not working from BO doc service or REST API. Problem: Loading of documents for the BODocumentService.PrintPDFs method is no longer loading documents. This is a side effect of the fixes for iConnect 254670, which changed the call to get the application information in such a way that the app packages are no longer loaded, which is now required when loading of the documents for print in GetPrintPDFs -> LoadPrintOrder -> GetFormDefinitionListForPrint (due to changes for iConnect 257996). NOTE: This issue also impacts the REST API GetDocumentList and GetActivityPDF calls. Resolution: Corrected the code to load the app packages.

    Areas Impacted: BODocumentService.PrintPDFs, REST API GetDocumentList, GetActivityPDF

Provider

  • DR 258539 'Agent Data Collection Form' not populating with required data. Problem: The provider call from the rules engine was passing the data items collection without the recent rules changes. Resolution: Pass the updated data item collection.

    Areas Impacted: Rules engine calls to provider.

Review Queue

  • DRs 258908, 258921 Reviewer Documents - Expected values from application not mapping into Reviewer Documents. Problem 1: Changes for common tags had the effect of activity data items from the source activity no longer transferring to the first reviewer documents app. Resolution: Corrected the logic to copy the data items. Problem 2: The logic to load the original activity data into the reviewer doc app was still not working correctly if the reviewer document form contained fields with common tags. Resolution: Exclude common tag data items when determining whether to copy the source activity data items. Patched Production on 3/2/2022.

    Areas Impacted: The first reviewer docs app for an activity in the review queue (sequential queue copying of data was not affected), pre-filled data on the first reviewer docs app for an activity in the review queue

Wizards and Forms

  • DR 258926 Wizard - issues with field values and form inclusion during data entry - The Ownership Structure will not hold the value. Problem 1: The dynamic drop-down would generate the correct drop-down values but would lose the values if you turn the page. It would hold the values if it was not a newly created app. Resolution: The dataitem that holds the drop-down values was not being included in the list of dataitems for the page until after the rules had run and the model reinitialized when re-opening the case. Moved the initializing of the page dataitems to after the rules dataitems are created. Problem 2: The dataitems were updating based on the conditional packages and the rules do not run in a consistent order, so sometimes it would not include the dataitem with the drop-down values when creating the page view and would only get added when the case was re-opened. Resolution: Added an update to the page map of dataitems to update the page maps if there have been dataitems updated based on the adding/removing of conditional packages.
    Areas Impacted: Field updates in wizards, Conditional packages

  • DR 259190 Wizard - Dropdown (Combo Box) Options Blank when reloading activity. Problem: The issue is not due to it being a distributor, and is not due to setcomboitem rules. For FPS-2980 (TFA), there are no such rules for SA_EMPLOYER_STATE. Rather, the issue is that the distributor has wizards in the sales agreement, so the combined forms+wizard UI is used for data entry rather than the legacy forms UI (which is used in the carrier). Due to changes for iConnect 258527 (and possibly other scenarios), the "item source postfix" data item can be empty, and when the activity is reloaded, this results in the drop-down values being cleared (the same behavior that would [correctly] occur if the combo items rule did clear all of the items). The check for the postfix data item should be treating an empty value as not being initialized from rules (since, if all items were removed, the string would contain an empty json list, which is not null or empty with the common tags changes that have been implemented). Resolution: If the drop-down item source postfix data item is empty, use the item source of the underlying field rather than initializing with an empty drop-down list.
    Areas Impacted: Populating of combo drop-downs in the combined wizard+forms UI when reloading an activity.

  • DR 257624 Apps getting 'stuck' after pre-submit review. Problem: This problem applies when an application has Pre-Signature Review enabled in its activity, and FLI_REQUIREMANUALREVIEW is set to false. If the agent browses back to the application editing page while waiting on signatures to complete, then the process for automatically declining Pre-Signature Review will occur, which will put the app into a state that causes the continue button to get stuck. This seems to be the most common when an agent sends a email request for signing and then completes the agent signing section, then browses back to the application. Resolution: Added check on the application status for automatically declining Pre-Signature Review so the process occurs when locking, not again after locking. This automatic declining of Pre-Signature Review now will only occur if the application status is in a "ActiveApplication" (0) state or a "PendingReview" (7) state, and occurs for Wizards when locking, and Forms when continuing from the Edit Application page. There isn't a way to check if Pre-Signature Review has already been declined so checking to see if we are in the correct state to auto-decline Pre-Signature Review is difficult. This also means that there are a lot of different ways this fix can be tested.
    Areas Impacted: Applications that use Pre-Signature Review and have FLI_REQUIREMANUALREVIEW set to false

  • DR 258115 Sporadic occurrences of submitted applications not completing.

    Problem (TFS 26998): Occasionally, an application is submitted, is processed in the BeforeOSJ workflow, but then does not go through the Submit workflow, leaving the status at Submit Requested. Analysis of an occurrence found that the BeforeOSJ workflow completed, deleting of the workflow record timed out, which prevented the Submit workflow from being initiated, resulting in the above behavior. Resolution: Modify the workflow processing to ensure the Submit workflow is always initiated when the BeforeOSJ workflow processing finishes.

    Problem (TFS 29232): When inserting into the workflow via StartWorkflow, if a WorkflowInstance record already exists for that app, it just updates the workflow, but does not update the WorkflowType. So if, for instance, a SendAppStatus workflow was initiated but failed, it would remain in the workflow, and if another workflow (for instance SubmitAfterOSJ) was started, it would remain as a 'SendAppStatus' workflow and the SubmitAfterOSJ workflow would never occur. Some workflow types are more important than others. If a SubmitAfterOSJ workflow was started, but remained due to a failure, a SendAppStatus workflow should not replace it in the workflow (but the reverse should). Resolution: Modify UpdateWorkflow to update the WorkflowType for workflows other than status workflows.
    Areas Impacted: Submission of applications

  • DRs 258151, 258310 Weird Behavior with the Tables in Distributor Instances. Problem: This behavior was a side effect of two bugs.

    1. There is logic in the wizard UI to load custom lists for the activity, but it loads only the app organization's custom lists, so in a distributor organization, the carrier's custom lists were not getting loaded (originally reported in iConnect 208198, but never fixed). The system compares the timestamp of the custom list with the associated grid information to determine if the grid data needs to be updated due to a list change, but because this list of custom lists doesn't include the carrier's, it always treats the grid data as out-of-date and proceeds to #2.

    2. Elsewhere, there is logic in the wizard UI (implemented for iConnect 223527) to "Update the list definition for a grid without (read: attempting not to) replacing user entries." This logic a) doesn't rely on the custom lists loaded in #1, b) assumes that there is a unique key column for the custom list data (which there isn't for custom list System SubAccounts, associated with grid field Allocation_Data_Grid) and c) apparently does replace user entries in some cases. Because of issue #1, this replacement logic always executes when opening an activity in the distributor environment, which has the effect of assigning values from one row to other rows of the grid with the same ProductId, which results in the allocation values getting duplicated.

    Resolution:

    1. Modified the wizard logic in a distributor environment to also load the carrier's custom lists for the logic to determine if the grid data is out-of-date. So if the custom list hasn't changed, it will no longer proceed to #2.

    2. When the custom list has changed, modified the wizard logic to update the grid list definition only if the column that has been identified as the key column has unique values for each row of the list. Note that this could still result in the original bug reported in 223527 manifesting in some scenarios (if a custom list not having a unique key column is updated, in-flight apps' grid data associated with that custom list will not get updated). And apparently the fix for 223527 is incomplete in that it does sometimes replace user entries, and that issue still exists, but will not occur in this scenario. A more comprehensive review of the original bug and fix would be needed to completely avoid this possibility.

    3. After discussion with FireLight development, the client will add a unique identifier Key to their data grids to prevent the issue.

    Areas Impacted: Data grids in combined wizard + forms UI
