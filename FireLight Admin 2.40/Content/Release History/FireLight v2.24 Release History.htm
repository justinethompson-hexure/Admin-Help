FireLight v2.24 Release History

2.24.213 June 14 UAT / June 17, 2022 Production Patch Release

Fixed Defects

Activities

  • DR 262937 Radio buttons retaining previous values when updating application from illustration. Problem: Form fields have the “Group Name” field set to the same value so only one should be selected at a time. However, when coming in from a Quote, the updated value is being checked in addition to (instead of in place of) the original selection. Resolution: This was fixed.
    Areas Impacted: Radio buttons

Activity Share

  • DR 263530 Add Case Access by firm ID cases to Activity Share. Problem: The Group setting for Case Access by Firm ID is used to provide office staff and assistants in an organization access to activities based on the firm ID/rep code passed over during the SSO process. Currently, the office staff is required to search for the individual cases in "All Activities." Resolution: Adding the ability for the office staff to view activities shared based on firm ID/rep ID will provide an efficient process for locating the activity.
    Areas Impacted: Activity Share for Case Access by Firm ID

Client Fill & Sign

  • DR 263819 Client Fill & Sign Functionality not appearing as option when expected to appear. Problem: A bug fix in the May release (iConnect 237487 -- Office Use Only Fields That Are Made Visible Show Red Required Background Instead of Green - Wizards) that re-introduced the bug fixed by iConnect 230994 ("Wizard - Client Fill and Sign isn't presented until all fields with 'Office Use Only' are populated"). Resolution: Rolled back changes for iConnect 237487 and also made modifications to also exclude disabled fields from consideration for determining required office use only fields.
    Areas Impacted: Client Fill & Sign

Email

  • DR 263823 Review queue and failed email notifications to multiple addresses not being sent. Problem: This bug is for correcting the issue of emails no longer being sent when the To address is multiple email addresses. Resolution: Handle email values that actually contain multiple addresses, parsing into separate addresses and adding each recipient to the message, so that sending will succeed.
    Area Impacted: Sending of email for review queue and failed email notifications when configured with multiple email addresses

Purge Process

  • DR 264167 Purge process is permanently removing activities prior to the purge policy. Problem: The purge policy is supposed to: 1) purge activities at their purge policy, and 2) permanently delete unsubmitted activities at 2x their purge policy. A bug was introduced in the 2.22 release by the bug fix for TFS 26661 that is resulting in incomplete apps being permanently deleted (rather than purged) at 1x their purge policy. Resolution: Corrected the purge process to purge activities at their purge policy, and permanently remove unsubmitted apps at 2x the purge policy. Areas Impacted: Purge process

Wizards and Forms

  • DR 263040 Wizard - Data entry non-responsive due to incompatible mask between carrier and distributor. Problem: Masked controls apply the masking in the browser and if there is a change the server is called to save those changes. However, in 2.24 when a field is readonly the server call does not happen. Resolution: The code that was preventing the update call when the masked input was readonly was modified to allow the update when the app was in Data Entry and readonly. Note, that if the input control was disabled the update call does not happen. The distinction between disabled and read-only is read-only controls can still function and can have focus, where a disabled control cannot have focus and does not fire any events.
    Areas Impacted: This is a unique scenario between Distributor and Carrier forms where a rule will set a carrier phone number field based on a value from a distributor field. Also, the source field from the distributor view must be read-only.

  • DR 263435 Error occurs when clicking Continue into signatures after clicking Continue in popup. Problem: In the Wizard view the Continue button was not disabled after continue was initiated from the toast message or by clicking the Continue button. This allowed a user to click the Continue button again, which resulted in a null reference error. This did not prevent the user from continuing to signatures. It either required them to retry or it continued after the error was displayed. Resolution: When the Continue logic is initiated the Continue button is disables preventing subsequent clicks.
    Areas Impacted: This could affect all users using the Wizard Data Entry UI and continuing to signatures

Enhancement

Fixed Defects

Activities

  • DR 261899 Enrollment - Send Request email not being re-sent but reporting success. Problem: When the fill and sign request is created in the client Production environment, the stored message does not contain the To email address. This is because the code that initializes the message content copies the FromAddress, ToAddress, and ToName only if the FromAddress has a value -- since the From address will always ultimately be no-reply@firelighteapp.com, this is inconsequential to the immediate issue). When the user attempts to resend the request, the system attempts to send the message to an empty email address, which fails, but because it doesn't check the result of the send, the Send Request logic still reports success to the user. Resolution: Always store the To email address and name in the application message data when creating a fill and sign request, so that if that information is later used to resend the request, it will send to the correct email address instead of a blank address. NOTE: This will not correct the issue for outstanding requests, but it will correct the issue for all new requests that are sent (for new activities, or if the outstanding request is canceled).
    Areas Impacted: Resend of previous fill and sign requests.

  • DR 262361 Links in Pop-up messages not working as expected. Problem: ULC.css file was overwriting correct values in the organization table. Resolution: Client .css items customNeedsCompleteText and customNeedSubmitText were removed from the css file so the organization table overrides are correct.
    Areas Impacted:Client css customNeedsCompleteText and customNeedSubmitText popup messages

  • DR 262667 Multi-App Functionality should not allow child apps to be deleted after submission. Problem: Logic to delete child activities in multi-activity and enrollment logic was specifying ForceDelete, which permanently removes the database records. As a result, in some cases, submitted activities were getting permanently removed from the system, which should not be possible and can impact data integrity. Resolution: Removed the ForceDelete parameter from the DeleteApplication logic, and all calls with this parameter, to prevent permanent delete of submitted activities.
    Areas Impacted: deletion of activities in multi-app, enrollment, and the system in general

  • DR 262707 Pro advisory blocked from completing AOE in Avantax. Problem: Changes for common tags to sync data items between data entry and rules introduced an issue where a data item change from a rule could be cleared when syncing rules data when initializing grids during the initial rule run. Data items affected in this case were ContractType, OwnershipStructure, and possibly others. The sequence of changes was:

    "Enable Replacement 4 Fields" => setcontrolrangestate => "Replacement4Grid_AR" = "" => UpdateGridDataItem => CommitUpdates => ProcessUpdates => ContractType = null

    Resolution: Modified a previous fix for iConnect 258699 to only sync data item changes to rules during data entry and not during the initial rule run, preventing data items updated in an earlier rule from getting cleared. Also added null check.
    Areas Impacted: Data item values during initial rule run before data entry

  • DR 263001 Error when updating application from ForeSight illustration. Problem: This resulted from a changeset from version 2.23 not being merged properly into version 2.24. Resolution: Resolved by a manual patch in Production.
    Areas Impacted: Illustration activities

Branding

  • DR 262751 Branding issues impacting wizard. Problem: Some branding-specific items for the client were not working well with wizards, possibly related to some recent css changes with the 2.24 release. Resolution: Set up a client-specific wizard css config that sets the Continue button's left margin to a smaller size, and colors with the edit name button/icon.
    Areas Impacted: Continue button for the client, specifically location and color, not functionality

DTCC Processing

  • DR 262981 Error when processing DTCC app sub responses. Problem: Queries for app/sub response files were not accounting for the fact we are now re-using the Request table, and so they were pulling back attachment records, which was leading to exceptions in the code that processed the results. Resolution: Added in some logic to the necessary queries that ensures we only pull back app/sub records.
    Areas Impacted: app/sub response file handling

Provider

  • DR 262362 The client provider update from UATNext was deployed to UAT, Staging, and Production on May 17 and is ready for testing. It may also be necessary to deploy the form/product name changes to Staging & Production.

Wizards

  • DR 262670 Re-opened. Locked wizard UI behavior differs for presale continuing into application. Problem: Wrong error message text was being displayed for application submission of Needs Approval and other activity types. There were also visual issues with the Continue button. Text getting wrapped and icons displaying when they shouldn't. Resolution: This was fixed.
    Areas Impacted: Wizards

  • DR 262954 Issue with multiple source of funds pages in FireLight wizard. Problem: Metadata is being saved as the last form when there is copies of a form. Resolution: Take into account Form Index when storing Metadata on saved cases.
    Areas Impacted: Saving Wizard fields

  • DR 263167 Reported slow response times when in Data Entry. Problem: The logic to load the model when navigating pages was taking almost 2 seconds to load signature information. Resolution: This signature information is only needed after the activity has been locked. So, this logic has been skipped when in Data Entry.
    Areas Impacted: All clients using the Wizard view of Data Entry

  • DR 255104 FL Illustrations Base gender mapping incorrect for life products. Problem: The gender is not mapping correctly for illustration and sets to female each time. Resolution: The gender for annuitant and insured is mapped over for illustration using the description of Male/Female or the int values 1 or 2. The export value used in this case was M/F and was not recognized and would always default to female. Added the values for M/F to allow it to map correctly.
    Areas Impacted: Gender set for annuitant and insured for illustration

  • DR 257859 Application Submitted without Client Signature. Problem: The request to Decline Review was not validating to verify the activity was in the correct state and included and auto submit step that submitted the activity before all signatures were complete. Resolution: Added validation to the both the Accept and Decline requests to verify the activity can be reviewed. This will prevent an activity getting incorrect audits and getting submitted before it should.
    Areas Impacted: This would affect clients using Pre and Post Signature Review. NOTE: this problem only occurs if the user opened an activity in multiple windows (sessions), where one session unlocks the activity while the other session declines review.

  • DR 258395 NWL carrier is not showing unless "All" is selected from Product Type. Problem: The list of products (and based on that, the list of carriers) could be incomplete when the product type was Select Product Type. Resolution: When the product type selection is Select Product Type, do not populate the list of product types based on the default product type, which can result in a partial list of products, and instead display an empty list of products. Also change the default Product Type in Preferences to be All rather than Life, and add Select Product Type to the drop-down so if that is the current selection, it will show that (and to allow the default selection to be changed back to Select Product Type). Additionally, the reason that NWL carrier displays first in the list of carriers is because the organization name is prefixed with a space character—modified the carrier list for the new app workflow to trim spaces from the beginning of the organization name so that it is sorted alphabetically if prefixed with space characters.

Problem: The ready for agent sign email was not being sent. Resolution: The option for agent ready for sign was moved to the group options and was using the session state to check for the group options. These are not set for external signature requests. The agent ready for signing should use the activity's owning agent group options to determine whether to send the agent a ready for signing email after the last external signer has completed.
    Areas Impacted: Agent signatures when the group option for notify agent when ready for signing and agent sign last are set in the Admin group options

  • DR 260014 Copy activity logic and data item properties. Problem: On a copy with changes it appears the visibility and enable rules are retaining on tags even when there are no visibility rules that exist in the tags on a new product. Resolution: Modified the copy with changes logic to set all data items IsEnabled = true, IsVisible = true, IsReadonly = false. Modified the combined wizard+forms UI to also set Visible and Enabled to true prior to initializing the rules engine (only set IsReadonly to false on copy with changes, so that legacy behavior will not be affected, and due to the fact that data items can be locked on a next activity).
    Areas Impacted: Copy with changes, default visibility and enabled state of data items prior to running rules

  • DR 260929 Signature dates on Addendum documents in blue ink causing errors. Problem: Addendum documents render the signature date in blue ink, which is causing submissions to error out in the back office. Resolution: Signature date on the Addendum document is printed in BLACK color.

Admin

  • DR 257186 Performance: Admin - slow loading of Groups profile. Problem: As per finding the performance issue on the groups profile page, we found that while loading the groups page on the UI, there is a function i.e. GetTrainingModel() to fetch all training videos. Under this function, there is an API call to Vimeo, which gets all videos and saves them into cache. The first time it takes time to call the API and after that it always fetches from the cache. Resolution: After understanding the vimeo API documentation, we found some more filters to get accurate data with fewer recursive calls. Before my changes, when API calls were made, it was found that there were 352 videos and the API picked only 25 videos at a time. To retrieve all of the data, the API was called 15 times. We found some more filters, i.e. per_page, query_fields, and query, to get only required videos and it calls the API once only. By using this concept, it improves the performance.
    Areas Impacted: Groups profile

  • DR 257989 Deleted Items consistently showing in deployment scopes. Resolution: Modified the logic to find eligible items for deployment -- for deleted items, do not consider changes to constituent items, and only evaluate the timestamp of the top-level item itself (i.e., product).
    Areas Impacted: Inclusion of deleted items for deployment in All, Distributor All, Package and Distributor Package configuration scopes

  • DR 259166 Admin form designer toolbox - Unable to move fields. Problem: Decimals were being passed into the MoveSelectedControls for the Twidth and Thight parameters.     Areas Impacted: Forms Toolbox, directly related to the "Move Controls" button

  • DR 259402 Rule set audits for rename and scope change show same old and new value. Problem: The audits always show the same old and new value because the value is updated before the audit is written. Resolution: Corrected the audits to show the old and new value.
    Areas Impacted: History of changes rule to name and scope

    Areas Impacted: Admin-> Product Config -> Sales Agreement

Email

  • DRs 260578, 260023 Emails originating from scheduler do not include branded company logo. Problem: The reminder emails, which always originate from the scheduler, have never supported branding with the company logo, because the scheduler does not have access to the logos, which are only defined in EGApp. This is a long-standing limitation of emails from the scheduler. With the 2.23 release, apps processed in a workflow (for instance, submit, but there are several other workflows as well) which fail on the first attempt are now retried in the scheduler (this is intentional, in preparation for the cloud. On the retry, if any emails are sent during the workflow processing from the scheduler (for instance, final emails on activity completion), and if the org is using their company logo in an HTML email template, the email will not include the company logo. Since this could impact emails that previously would correctly include the branded logo, the existing limitation should be removed, so that emails originating from the scheduler will properly include the company logo. This will also address the reported issue of reminder emails not including the company logo. Resolution: Modified the scheduler to locate and use the branding images in EGApp.
    Areas Impacted: Branded emails with company logo originating from scheduler (reminders, final emails, etc.)

  • DR 260693 Email request submitted for invalid email address and stuck in email queue. Problem: The changes for iConnect 202342 "Unable to enter special character in email address" had the side effect of allowing invalid email address such as: abc@xyz@gmail.com or abc@email1.com;abc@email2.com. Resolution: Fixed the email validation to more closely track the RFC requirements of an email address, based on https://www.regular-expressions.info/email.html, https://stackoverflow.com/a/34096956/156738, https://stackoverflow.com/a/201336/156738.
    Areas Impacted: Validation of email addresses in FireLight

  • DR 261044 Emails being sent to no-reply@firelighteapp.com. Problem: Emails were being incorrectly sent to the "no-reply@firelighteapp.com." This email address is never valid. Also, SendGrid failed response were not including the message body. Resolution: Added an exception if an email was being sent to noreply@firelighteapp.com" accidentally to "disable" this possibility. Also, added the message to the failed SendGrid response email.
    Areas Impacted: Email system

  • DR 262060 From name in email message can be incorrectly shown as no-reply@firelighteapp.com. Problem: Two different scenarios have been identified where the From name in an email out of FireLight can incorrectly display as the organization email address or no-reply@firelighteapp.com. 1) When possible, the From name & email address are based on the owning agent's login information, but there are several factors that can result in that not being used: a) The group option Enable Use Agent Email must be checked. b) The user preference option Use Agent Email for All Communications must be checked. c) The agent must have a name passed in their SSO or configured in preferences. If any of those are not the case, the organization email address is used as the final fallback for the From address & name (and the default organization email address is no-reply@firelighteapp.com), and also for the AGENT_NAME email template variable. Instead of using the organization email address for the From name, it should instead use the organization name for this fallback. 2) Even with the previous corrected, there is another scenario where the From name of the email message will not be specified and is shown as fl-noreply@firelighteapp.com, if the organization does not have Use DMARC configured in org settings. In addition, the system does not set the To name of any email messages, only the To address. This is less noticeable, because many email clients will show the receiving user's name even though it isn't in the mail headers, but this should also be fixed. Resolution: 1) Default the From name in emails to the organization name instead of org email as a final fallback when not using the agent name due to group, preference, and user settings. 2) Ensure that the From name in the email From information is always using the From Name associated with the email message. 3) Trim whitespace from the beginning and end of the From address since it can prevent sending of messages. 4) Check for null or whitespace instead of null or empty when determining the email address and name values to use for an email message.
    Areas Impacted: All emails

  • DR 259932 Rolled back per DR 261999. Labels not covering fields on PDF. Problem: Z-ordering is used to define which controls display on top of other controls at the same location, which is used before activity lock when rendering fields within FireLight. The PDF file format doesn't have the same concept (it might support layers, but the PDF functionality in FireLight doesn't utilize that). When generating the PDF, FireLight first stamps labels, then other form fields, and the ones stamped last "win" and is the top-most controls and therefore will display over labels. Resolution: I discovered that if the labels are stamped after the form fields, the labels is top-most and cover other fields at the same position. It isn't actually honoring the Z-order defined in FireLight, but it will have the expected effect of layering labels on top of other form fields.
    Areas Impacted: All PDF generation in FireLight except fillable PDFs

    Areas Impacted: Print, Submission, and anywhere else FireLight prints PDFs with data in them—this is specific to forms with labels not marked as 'Exclude From PDF'.
