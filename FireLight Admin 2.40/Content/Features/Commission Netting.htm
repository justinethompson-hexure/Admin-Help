Commission Netting
Description, Setup, User Experience, and Additional Information

Description

Guidelines:

  • There is only one distributor per DPfA. The distributor is identified as the TXLife/TXLifeRequest/OLife/Party aggregate that has a child Producer aggregate.

  • The distributor will match a valid organization in FireLight (the Distributor's OrgCode in the DPfA should match an Organization's External Map Code).

  • While there can be multiple instances of a carrier/distributor combination, they will differ on the start date. If a match on carrier, distributor, and start date is found, the user is prompted to confirm they wish to overwrite the existing file to avoid an accidental overwrite.

  • Any DPfA file contains data of a 1204 message type and conforms to ACORD Life and Annuity Transaction Specifications with version 4.0.1.

Calculating the Net Commission:

There are two ways to calculate the commissions for an application:

  • When a carrier knows that their rates do not vary, the first rate found is used to multiply the amount by to get the commission for the whole application.

  • When the rates do vary based on the Comm Event, the total commission amount is the sum of the amounts associated with the Comm Event multiplied by the rate associated with the comm event. At the current time the following events are supported: Cash with App, Internal Exchange, External Exchange.

The value for the commission is returned as 0 if a rate cannot be found for any reason. Depending on the result, the provider will then decide if the App/Sub file should be sent or not.

Audit Administrative DPfA Activity:

Administrative auditing consists of keeping an account of all imports, updates, and/or deletions of a client’s Distribution Profiles (DPfAs), which include information for Commission Netting. Each audit will account for the user responsible, the date of the action, the action itself as well as a message with more information.

Purge Old DPfAs from the System:

There can be multiple DPfAs for a particular Carrier and Distributor due to the carrier being able to upload a file before it is scheduled to go live. The process is created following the standard data purge policy in FireLight for DPfA entries that are no longer valid as they have been replaced with a new version (via the Start Date) after the new version goes live.
