ACORD Tx103 Requirements and Usage

Overview

The ACORD Tx103 request allows application data to be sent over in the single sign-on request and pre-filled in the corresponding fields on the application. The data in the 103 is either mapped directly over by the provider into the edit application process or saved in the case of the delayed 103. Application data is only available for reuse in the delayed 103 process.

Requirements

The following table describes the elements that are required by FireLight® to process the 103 transaction properly. The product code should be omitted in the case of a delayed 103 transaction.

Required Elements and Descriptions
Element Description
<TransRefGUID> Transaction Reference GUID. Unique identifier used to uniquely identify this transaction with FireLight.
<ProductCode> Identifies the product being sold in FireLight. This string must match up to one, and only one, product within FireLight. The value of this field is matched against the CUSPID property within the Product tab of the Admin tool.
<CarrierCode> Identifies the organization selling the product identified in <ProductCode>. This value must match to one, and only one, organization defined in FireLight. The value is matched against the External Map Code for the organization. Insurance Technologies will provide you with the code for your implementation.
<Jurisdiction> Identifies the selling jurisdiction for the application. The “tc” value must match one of the ACORD OLI_LU_STATE lookup values.
<Party> At least one Party object with a relationship type of “37” (OLI_REL_PRIMAGENT) must be present within the transaction. The generated application will be tied to the agent identified within this instance.
<Person>/<LastName> The name of the created application will be the primary agent’s last name suffixed with “Application”.
<CarrierAppointment> The contents of this construct are used by FireLight to associate the application to a user created in FireLight to represent the primary agent.
<CompanyProducerID> A unique identifier of the agent within the organization. The newly created application will be associated with this identifier (aka user). Note: the identifier must only be unique within the organization identified in the <CarrierCode> element within the <CarrierAppointment> construct.
<CarrierCode> Identifies the organization associated with the agent identified in <CompanyProducerID >. Currently in FireLight this should be the same value as the <CarrierCode> within the <Policy> construct.
<Relation> This construct is used to associate the primary agent’s Party instance to the primary Holding instance.
OriginatingObjectID Must match the primary Holding instance’s ID.
ReleatedObjectID Must match the primary agent’s, Party instance’s ID.
<OriginatingObjectType> The “tc” value must be 4.
<RelatedObjectType> The “tc” value must be 6.
<RelationRoleCode> The “tc” value must be 37.

Usage

In addition to application data, the Tx103 request is used to post over the transaction type. The following transaction types are used in the 103 request to initiate a new application: New Business, Subsequent Payment*, Post Issue*, Needs Determination, and Fee Inquiry. Post Issue requires an additional transaction subtype to be sent over in the request to indicate the type of post issue maintenance. Needs Determination requires a transaction type of 116 and a subtype of 11600. Fee Inquiry requires a transaction type of 1241. The transaction type and transaction subtype will be stored in the global dataitems FLI_TRANSTYPE and FLI_TRANSSUBTYPE for use in the rules within the Admin form designer.

To create a new business application, the 103 request should include the following:

<TransType tc="103">OLI_TRANS_NBSUB</TransType>

To create a subsequent payment application, the 103 request should include the following:

<TransType tc="508">OLI_TRANS_PAYMNT</TransType>

To create a post issue application with the maintenance type of Admin changes (changes to name, address, email, phone, owner, or beneficiary), the 103 request should include the following:

<TransType tc="113">OLI_TRANS_POLICYADMIN</TransType>

<TransSubType tc="113"> OLI_TRANS_POLICYADMIN </TransSubType>

To create a post issue application with the maintenance type of withdrawal request, the 103 request should include the following:

<TransType tc="113">OLI_TRANS_POLICYADMIN</TransType>

<TransSubType tc="105"> OLI_TRANS_WITHDR </TransSubType>

To create a post issue application with the maintenance type of program additions/changes (automatic withdrawal services, rebalancing, DCA), the 103 request should include the following:

<TransType tc="113">OLI_TRANS_POLICYADMIN</TransType>

<TransSubType tc="107">OLI_TRANS_UPDARR</TransSubType>

To create a fee inquiry transaction, the 103 request should include the following:

<TransType tc="1241"> OLI_TRANS_TRNFEEEXPENSE </TransType>

To create a needs determination transaction, the 103 request should include the following:

<TransType tc="116">OLI_TRANS_NEEDSDET</TransType>

<TransSubType tc="11600">OLI_TRANSSUB_NEEDSDET</TransSubType>

* There is an additional license fee for subsequent payment, post issue and fee inquiry (fees, expenses and commissions repository) support. Existing clients will need an addendum to their contract to support this new feature.
