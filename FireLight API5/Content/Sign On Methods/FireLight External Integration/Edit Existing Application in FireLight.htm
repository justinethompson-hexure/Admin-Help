Edit Existing Application in FireLight

At times, you may want to provide the user the ability to launch directly into the Edit Application view within FireLight® without sending over a 103 with update information. The following diagram illustrates the launch sequence for this scenario.

The following is an explanation of each step:

  1. The user is authenticated and authorized by the external system.

  2. The user clicks on a link to launch into the FireLight dashboard.

  3. The external system sends back to the user a specially crafted response to force the browser to POST an ACORD Tx1228 transaction within a hidden form field to the FireLight passive endpoint.

  4. FireLight authenticates with the STS using a predefined shared user login and password set up specifically for the implementation. If the credentials are valid, FireLight will impersonate the user identified within the 1228 for the remainder of the session. If the credentials are not valid, the user will be redirected to a generic error page within FireLight.

  5. FireLight will send back a redirect to the FireLight dashboard.

A query string named “appid” with a value retained from the original <TXLifeResponse>/<HoldingKey> is required to be sent within the action on the form post. See “Sample HTML Form” in Post ACORD TX 1228 Transaction.
